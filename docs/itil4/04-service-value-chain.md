## 1. 서비스 가치 사슬 개요

**서비스 가치 사슬 (Service Value Chain)**은 SVS의 중심 요소로, 수요를 가치로 변환하는 **6가지 상호 연결된 활동**으로 구성된다.

| 항목 | 설명 |
|------|------|
| 정의 | 제품과 서비스를 생성, 제공, 지원하기 위해 조직이 수행하는 상호 연결된 활동의 운영 모델 |
| 특징 | 비선형적 (Non-linear), 필요에 따라 활동 순서가 변함 |
| 구성 | 6가지 핵심 활동 |
| ITIL v3와 차이 | 선형적 생명주기 → 유연한 가치 사슬 |

### 6가지 활동 요약

| 활동 | 영문 | 핵심 역할 |
|------|------|----------|
| 계획 | Plan | 공유된 비전과 방향 정의 |
| 개선 | Improve | 모든 활동의 지속적 향상 |
| 참여 | Engage | 이해관계자와의 상호작용 |
| 설계 및 전환 | Design & Transition | 사양에 맞는 서비스 설계와 전환 |
| 획득/구축 | Obtain/Build | 서비스 구성요소 획득 및 구축 |
| 제공 및 지원 | Deliver & Support | 합의된 사양에 따른 서비스 제공과 지원 |

---

## 2. 계획 (Plan)

조직의 모든 수준에서 비전, 현재 상태, 개선 방향을 공유하는 활동이다.

| 항목 | 설명 |
|------|------|
| 목적 | 4가지 차원과 모든 제품/서비스에 걸쳐 공유된 이해 보장 |
| 범위 | 전략적 계획, 전술적 계획, 운영 계획 |
| 핵심 활동 | 포트폴리오 결정, 아키텍처 관리, 정책 수립 |

### 계획 활동의 주요 관심사

| 관심사 | 설명 |
|--------|------|
| 전략 방향 | 조직의 IT 전략이 비즈니스 전략과 정렬되어 있는지 확인 |
| 포트폴리오 | 어떤 서비스를 유지/개선/폐기할 것인지 결정 |
| 자원 배분 | 인력, 예산, 기술 자원의 최적 배분 |
| 개선 우선순위 | 어떤 개선 기회를 먼저 추진할 것인지 결정 |

---

## 3. 개선 (Improve)

모든 가치 사슬 활동과 4가지 차원에 걸쳐 지속적 향상을 보장하는 활동이다.

| 항목 | 설명 |
|------|------|
| 목적 | 모든 수준에서 제품, 서비스, 프랙티스의 지속적 개선 보장 |
| 범위 | 서비스 가치 사슬의 모든 활동에 적용 |
| 핵심 활동 | 개선 기회 식별, 측정, 분석, 실행 |
| 관련 프랙티스 | 지속적 개선 (Continual Improvement) |

### 개선의 적용 수준

| 수준 | 대상 | 예시 |
|------|------|------|
| 전략적 | 비즈니스 및 조직 전략 | IT 전략의 비즈니스 정렬도 향상 |
| 전술적 | 프로세스 및 프랙티스 | 인시던트 관리 프로세스 효율화 |
| 운영적 | 일상적 활동 | 서비스 데스크 응대 스크립트 개선 |

---

## 4. 참여 (Engage)

모든 이해관계자의 필요를 이해하고 투명하며 지속적인 관계를 유지하는 활동이다.

| 항목 | 설명 |
|------|------|
| 목적 | 이해관계자의 요구사항을 이해하고 좋은 관계 유지 |
| 범위 | 내부/외부 모든 이해관계자 |
| 핵심 활동 | 요구사항 수집, 기대치 관리, 커뮤니케이션 |

### 참여 대상별 활동

| 이해관계자 | 참여 활동 |
|-----------|----------|
| 고객 (Customer) | 비즈니스 요구사항 수집, SLA 협의 |
| 사용자 (User) | 서비스 요청 처리, 피드백 수집 |
| 공급자 (Supplier) | 계약 관리, 성과 평가 |
| 내부 팀 | 요구사항 전달, 우선순위 조율 |
| 경영진 | 전략적 방향 정렬, 보고 |

---

## 5. 설계 및 전환 (Design and Transition)

제품과 서비스가 이해관계자의 기대와 품질 기준을 충족하도록 설계하고 전환하는 활동이다.

| 항목 | 설명 |
|------|------|
| 목적 | 제품과 서비스가 합의된 사양과 이해관계자 기대를 지속적으로 충족하도록 보장 |
| 범위 | 신규 서비스 설계, 기존 서비스 변경, 서비스 전환 |
| 핵심 활동 | 아키텍처 설계, 테스트, 릴리스 관리 |

### 설계 및 전환의 핵심 고려사항

| 고려사항 | 설명 |
|---------|------|
| 사양 충족 | 비즈니스 요구사항이 기술적 사양으로 변환되었는지 확인 |
| 비용 효율성 | 설계와 전환 비용이 예산 내에서 관리되는지 확인 |
| 리스크 관리 | 새로운 서비스나 변경으로 인한 리스크 평가 및 대응 |
| 규정 준수 | 관련 법규 및 규정 요구사항 충족 확인 |
| 전환 계획 | 기존 서비스에서 새 서비스로의 안전한 전환 보장 |

---

## 6. 획득/구축 (Obtain/Build)

서비스 구성요소를 필요한 사양에 맞게 사용할 수 있도록 보장하는 활동이다.

| 항목 | 설명 |
|------|------|
| 목적 | 서비스 구성요소가 합의된 사양에 따라 사용 가능하도록 보장 |
| 범위 | 인프라, 소프트웨어, 서비스 구성요소의 조달 및 구축 |
| 핵심 활동 | 구매/조달, 개발, 구성, 통합 |

### 획득 vs 구축

| 구분 | 획득 (Obtain) | 구축 (Build) |
|------|-------------|-------------|
| 정의 | 외부에서 자원/구성요소 조달 | 내부에서 자원/구성요소 개발 |
| 예시 | SaaS 구독, 하드웨어 구매, 아웃소싱 | 자체 소프트웨어 개발, 커스터마이징 |
| 고려사항 | 공급자 관리, 계약, 비용 | 역량, 일정, 품질 관리 |

---

## 7. 제공 및 지원 (Deliver and Support)

합의된 사양과 이해관계자 기대에 따라 서비스가 제공되고 지원되도록 보장하는 활동이다.

| 항목 | 설명 |
|------|------|
| 목적 | 서비스가 합의된 사양에 따라 제공되고 지원되도록 보장 |
| 범위 | 일상적 서비스 운영, 인시던트 처리, 사용자 지원 |
| 핵심 활동 | 서비스 제공, 인시던트 해결, 서비스 요청 처리 |

### 제공 및 지원의 핵심 요소

| 요소 | 설명 |
|------|------|
| SLA 준수 | 합의된 서비스 수준 유지 |
| 인시던트 관리 | 서비스 중단 시 신속한 복구 |
| 서비스 요청 처리 | 사용자 요청의 효율적 이행 |
| 문제 해결 | 반복 장애의 근본 원인 분석 및 해결 |
| 커뮤니케이션 | 사용자에게 서비스 상태 정보 제공 |

---

## 8. 각 활동의 입력과 출력

### 입력/출력 요약표

| 활동 | 주요 입력 | 주요 출력 |
|------|----------|----------|
| 계획 (Plan) | 거버넌스 정책, 포트폴리오 정보, 개선 기회, 외부 요인 분석 | 전략/전술/운영 계획, 포트폴리오 결정, 아키텍처, 정책 |
| 개선 (Improve) | 성과 정보, 이해관계자 피드백, 지식 자산, 개선 기회 | 개선 이니셔티브, 상태 보고서, 개선된 프랙티스 |
| 참여 (Engage) | 고객/사용자 요구사항, 서비스 성과 보고서, 마케팅 기회 | 통합된 요구사항, 서비스 요청, 변경 요청, 피드백 |
| 설계 및 전환 (D&T) | 요구사항, 아키텍처, 서비스 구성요소, 변경 요청 | 신규/변경된 서비스, 서비스 설계 패키지, 전환 계획 |
| 획득/구축 (O/B) | 아키텍처, 설계 사양, 요구사항, 변경 요청 | 서비스 구성요소, 인프라, 소프트웨어, 구성 항목 |
| 제공 및 지원 (D&S) | 서비스 구성요소, SLA, 운영 절차, 사용자 요청 | 제공된 서비스, 인시던트 해결, 서비스 성과 정보 |

### 활동 간 상호작용

| 출발 활동 | 도착 활동 | 전달 내용 |
|----------|----------|----------|
| 계획 | 모든 활동 | 전략적 방향, 정책, 우선순위 |
| 참여 | 설계 및 전환 | 통합된 요구사항, 변경 요청 |
| 참여 | 제공 및 지원 | 서비스 요청, 장애 보고 |
| 설계 및 전환 | 획득/구축 | 설계 사양, 구축 요구사항 |
| 획득/구축 | 설계 및 전환 | 서비스 구성요소, 테스트 결과 |
| 획득/구축 | 제공 및 지원 | 완성된 서비스 구성요소 |
| 제공 및 지원 | 개선 | 성과 데이터, 인시던트 패턴 |
| 개선 | 모든 활동 | 개선 이니셔티브, 권고사항 |

## 9. 가치 흐름 (Value Stream)

**가치 흐름 (Value Stream)**은 수요를 가치로 변환하기 위해 서비스 가치 사슬의 활동을 조합한 특정 경로이다.

| 항목 | 설명 |
|------|------|
| 정의 | 조직이 수요에 대응하여 가치를 창출하기 위해 수행하는 특정 활동 조합 |
| 특징 | 동일한 가치 사슬 활동을 다른 순서/조합으로 활용 가능 |
| 설계 원칙 | Lean 가치 흐름 매핑 (Value Stream Mapping) 기법 활용 |

### 가치 흐름 예시

| 가치 흐름 | 관련 활동 (순서) | 설명 |
|----------|----------------|------|
| 신규 서비스 개발 | 참여 → 계획 → 설계 및 전환 → 획득/구축 → 제공 및 지원 | 새로운 서비스를 설계하고 배포 |
| 인시던트 해결 | 참여 → 제공 및 지원 → 개선 | 장애 접수부터 해결까지 |
| 서비스 요청 이행 | 참여 → 제공 및 지원 | 사용자 요청의 접수와 이행 |
| 서비스 개선 | 개선 → 계획 → 설계 및 전환 → 획득/구축 | 기존 서비스의 품질 향상 |

### 가치 흐름 설계 시 고려사항

| 고려사항 | 설명 |
|---------|------|
| 낭비 제거 | 가치를 더하지 않는 단계를 식별하고 제거 |
| 병목 해소 | 흐름을 지연시키는 병목 지점 해결 |
| 자동화 기회 | 반복적 활동의 자동화 가능성 평가 |
| 측정 | 리드 타임, 처리량 등 핵심 지표 모니터링 |
