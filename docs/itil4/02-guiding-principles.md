## 1. 지도 원칙이란?

**지도 원칙 (Guiding Principles)**은 조직이 어떤 상황에서든 적용할 수 있는 보편적인 권고 사항이다. ITIL 4에서는 7가지 지도 원칙을 정의하며, 이는 SVS(서비스 가치 시스템)의 모든 구성요소에 적용된다.

| 특징 | 설명 |
|------|------|
| 보편성 | 모든 이니셔티브와 이해관계자 관계에 적용 |
| 독립성 | 특정 기술이나 방법론에 종속되지 않음 |
| 상호작용 | 7가지 원칙은 서로 연관되며 함께 작용 |
| Agile 연관 | Agile Manifesto의 원칙과 높은 유사성 |

---

## 2. 가치에 집중 (Focus on Value)

> 모든 활동은 직접적 또는 간접적으로 이해관계자를 위한 가치와 연결되어야 한다.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "이 활동이 고객/이해관계자에게 어떤 가치를 제공하는가?" |
| 가치의 관점 | 서비스 소비자가 정의하는 것이지, 제공자가 결정하는 것이 아님 |
| 고객 경험 (CX) | 고객 경험 전체를 고려하여 가치 평가 |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 서비스 소비자 파악 | 누가 서비스를 사용하는지 명확히 정의 |
| 가치 동인 이해 | 고객이 중요하게 여기는 것이 무엇인지 파악 |
| 고객 경험 매핑 | 서비스 접점(touchpoint)마다 고객 경험 분석 |
| 피드백 수집 | 정기적인 만족도 조사 및 피드백 루프 운영 |

### 실무 예시

- IT 헬프데스크: 해결 시간(Resolution Time)보다 고객 만족도(CSAT) 중심으로 KPI 설정
- 클라우드 마이그레이션: 기술적 이점뿐 아니라 비즈니스 민첩성 향상 관점에서 가치 평가

## 3. 현재 상태에서 시작 (Start Where You Are)

> 처음부터 다시 시작하지 말고, 기존에 사용 가능한 것을 먼저 고려하라.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "현재 무엇이 이미 존재하고, 활용할 수 있는가?" |
| 위험 | 현재 상태를 무시하면 불필요한 중복 작업과 자원 낭비 발생 |
| 관찰 | 직접 관찰하여 현재 상태를 객관적으로 측정 |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 현재 상태 평가 | 기존 서비스, 프로세스, 도구를 객관적으로 측정 |
| 데이터 기반 판단 | 가정이 아닌 측정 데이터를 근거로 결정 |
| 재사용 우선 | 새로 만들기 전에 기존 자산 활용 가능성 검토 |
| 점진적 개선 | 대대적 교체보다 기존 시스템의 점진적 개선 추구 |

### 실무 예시

- ITSM 도구 도입: 기존 도구의 활용도를 먼저 평가한 후 교체 여부 결정
- 프로세스 개선: 기존 프로세스 분석 후 약점만 보완하는 방식 채택

## 4. 피드백과 함께 반복적으로 진행 (Progress Iteratively with Feedback)

> 하나의 큰 작업을 한 번에 완료하려 하지 말고, 작은 단위로 나누어 반복적으로 진행하라.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "더 작은 단위로 나눠서 빠르게 피드백을 받을 수 있는가?" |
| 반복 (Iteration) | 관리 가능한 작은 단위로 분할하여 실행 |
| 피드백 루프 | 각 반복마다 피드백을 수집하여 방향 조정 |
| MVP | 최소 기능 제품 (Minimum Viable Product) 개념 활용 |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 타임박싱 | 작업을 시간 단위로 제한 (Sprint, Iteration) |
| 피드백 채널 | 이해관계자와 정기적 피드백 교환 체계 수립 |
| 실험과 학습 | 작은 실험을 통해 검증 후 확대 |
| 리스크 관리 | 반복마다 리스크를 재평가하고 대응 |

### 실무 예시

- 서비스 개선: 전체 ITSM 프로세스를 한 번에 바꾸지 않고, 인시던트 관리부터 개선
- 소프트웨어 배포: Big-bang 배포 대신 Canary/Blue-Green 배포 활용

## 5. 협업하고 가시성 촉진 (Collaborate and Promote Visibility)

> 올바른 사람을 올바른 역할에 참여시키고, 정보를 투명하게 공유하라.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "관련 이해관계자가 모두 참여하고, 정보가 투명한가?" |
| 사일로 해소 | 부서 간 벽을 허물고 협업 촉진 |
| 가시성 | 작업 현황, 의사결정 과정을 투명하게 공유 |
| 신뢰 구축 | 투명성을 통해 이해관계자 간 신뢰 형성 |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 크로스 펑셔널 팀 | 다양한 부서의 전문가로 팀 구성 |
| 대시보드 공유 | 서비스 현황을 실시간으로 공유하는 대시보드 운영 |
| 정기 회의 | 이해관계자와의 정기적 상태 공유 회의 |
| 문서화 | 의사결정 과정과 근거를 문서로 기록 |

### 실무 예시

- 장애 대응: 주요 장애 발생 시 실시간 상태 페이지(Status Page) 운영
- 프로젝트 관리: Kanban 보드를 활용한 작업 현황 가시화

## 6. 전체적으로 사고하고 작업 (Think and Work Holistically)

> 어떤 서비스나 구성요소도 독립적으로 존재하지 않으며, 전체 시스템의 관점에서 사고하라.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "이 변경이 전체 시스템에 어떤 영향을 미치는가?" |
| 시스템 사고 | 개별 요소가 아닌 전체 시스템의 상호작용 이해 |
| 통합적 관점 | 4가지 차원(사람, 기술, 파트너, 프로세스)을 모두 고려 |
| 엔드투엔드 | 서비스의 시작부터 끝까지 전체 흐름 파악 |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 서비스 맵핑 | 서비스 간 의존 관계를 시각적으로 매핑 |
| 영향도 분석 | 변경 전 영향 범위를 사전 분석 |
| 엔드투엔드 모니터링 | 사용자 관점에서의 전체 서비스 흐름 모니터링 |
| 패턴 인식 | 개별 인시던트에서 반복 패턴 식별 |

### 실무 예시

- 장애 분석: 단일 서버 문제가 아닌 전체 서비스 체인에서의 영향 분석
- 변경 관리: 하나의 API 변경이 연관된 모든 시스템에 미치는 영향 평가

## 7. 단순하고 실용적으로 유지 (Keep it Simple and Practical)

> 가치를 제공하지 않는 불필요한 단계나 프로세스는 제거하라.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "이 단계가 정말 가치를 더하는가? 더 간단한 방법은 없는가?" |
| 단순성 | 최소한의 단계로 결과를 달성 |
| 실용성 | 이론보다 실제 적용 가능한 방법 우선 |
| 과잉 설계 방지 | 필요 이상의 복잡한 프로세스 지양 |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 가치 스트림 분석 | 각 단계가 가치를 더하는지 검증 |
| 예외 최소화 | 예외 처리를 줄이고 표준화 추구 |
| 결과 중심 | 수단이 아닌 결과에 집중 |
| 점진적 복잡화 | 필요할 때만 복잡도 추가 |

### 실무 예시

- 승인 프로세스: 10단계 승인을 3단계로 축소하여 처리 속도 향상
- 문서화: 100페이지 문서 대신 핵심만 담은 1-2페이지 운영 가이드 작성

---

## 8. 최적화하고 자동화 (Optimize and Automate)

> 먼저 최적화한 후 자동화하라. 비효율적인 프로세스를 자동화하면 비효율만 빨라진다.

| 항목 | 설명 |
|------|------|
| 핵심 질문 | "프로세스가 최적화되어 있는가? 자동화할 부분은 어디인가?" |
| 최적화 우선 | 자동화 전에 반드시 프로세스 최적화 선행 |
| 자동화 | 반복적이고 표준화된 작업을 기술로 자동화 |
| 순서 | 최적화 → 자동화 (역순 금지) |

### 실천 방법

| 실천 항목 | 구체적 행동 |
|----------|-----------|
| 프로세스 분석 | 병목 지점과 낭비 요소 식별 |
| 표준화 | 자동화 전에 프로세스를 표준화 |
| 자동화 대상 선정 | 반복적, 예측 가능한 작업부터 자동화 |
| 지속적 개선 | 자동화 후에도 지속적으로 모니터링 및 개선 |

### 실무 예시

- 서비스 요청: 비밀번호 초기화를 셀프서비스 포털로 자동화
- 배포 관리: CI/CD 파이프라인 구축으로 배포 자동화
- 모니터링: 임계치 기반 자동 알림 및 자동 복구 (Auto-healing) 구현

## 9. 7대 원칙 요약 비교

| 번호 | 원칙 (한국어) | 원칙 (영문) | 핵심 메시지 |
|------|------------|-----------|-----------|
| 1 | 가치에 집중 | Focus on Value | 모든 활동은 가치와 연결되어야 한다 |
| 2 | 현재 상태에서 시작 | Start Where You Are | 기존 자산을 활용하고 객관적으로 평가하라 |
| 3 | 피드백과 함께 반복 진행 | Progress Iteratively with Feedback | 작게 나누어 빠르게 피드백 받으며 진행하라 |
| 4 | 협업하고 가시성 촉진 | Collaborate and Promote Visibility | 투명하게 공유하고 함께 일하라 |
| 5 | 전체적으로 사고하고 작업 | Think and Work Holistically | 부분이 아닌 전체 시스템 관점으로 접근하라 |
| 6 | 단순하고 실용적으로 유지 | Keep it Simple and Practical | 불필요한 것을 제거하고 실용적으로 유지하라 |
| 7 | 최적화하고 자동화 | Optimize and Automate | 최적화 먼저, 자동화는 그 다음이다 |
