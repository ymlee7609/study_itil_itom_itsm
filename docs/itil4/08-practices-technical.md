## 1. 개요

**기술 관리 프랙티스 (Technical Management Practices)**는 기술 관리 분야에서 발전해온 프랙티스이다. 총 **3개**의 프랙티스로 구성되며, 서비스의 기술적 구현과 운영에 초점을 맞춘다.

| 항목 | 설명 |
|------|------|
| 적용 범위 | IT 기술 관리 영역 |
| 개수 | 3개 |
| 기원 | 기술 관리 (Technology Management) 분야 |
| 특징 | 서비스의 기술적 기반을 다루는 프랙티스 |

### 3개 프랙티스 요약

| 번호 | 프랙티스 (한국어) | 영문 | 핵심 역할 |
|------|----------------|------|----------|
| 1 | 배포 관리 | Deployment Management | 서비스 구성요소를 운영 환경에 배포 |
| 2 | 인프라 및 플랫폼 관리 | Infrastructure and Platform Management | 기술 인프라 감독 및 관리 |
| 3 | 소프트웨어 개발 및 관리 | Software Development and Management | 소프트웨어 수명주기 관리 |

## 2. 배포 관리 (Deployment Management)

### 정의 및 목적

| 항목 | 설명 |
|------|------|
| 목적 | 신규/변경된 하드웨어, 소프트웨어, 문서, 프로세스 등을 운영 환경으로 이동 |
| 범위 | 개발/스테이징에서 운영 환경으로의 배포 전 과정 |
| 핵심 원칙 | 서비스 중단 최소화, 반복 가능한 배포 프로세스 |

### 릴리스 관리 vs 배포 관리

| 구분 | 릴리스 관리 (Release Management) | 배포 관리 (Deployment Management) |
|------|-------------------------------|--------------------------------|
| 초점 | "무엇을" 배포할 것인가 | "어떻게" 배포할 것인가 |
| 활동 | 릴리스 계획, 승인, 구성 결정 | 실제 설치, 구성, 활성화 |
| 질문 | 어떤 버전을 언제 배포하는가? | 어떤 방법과 환경에 배포하는가? |
| 비유 | 이사할 짐 목록 작성 | 실제 이사 작업 수행 |

### 배포 방식

| 배포 방식 | 영문 | 설명 | 장점 | 단점 |
|----------|------|------|------|------|
| 빅뱅 | Big Bang | 모든 환경에 한 번에 배포 | 빠른 전환 | 높은 리스크 |
| 단계적 | Phased | 사용자/환경 그룹별 순차 배포 | 리스크 분산 | 긴 배포 기간 |
| 지속적 제공 | Continuous Delivery | 자동화된 파이프라인으로 지속 배포 | 빠른 피드백 | 자동화 구축 비용 |
| 풀 방식 | Pull | 사용자가 원할 때 새 버전 설치 | 사용자 주도 | 버전 관리 어려움 |

### 현대적 배포 전략

| 전략 | 영문 | 설명 |
|------|------|------|
| 블루/그린 배포 | Blue/Green Deployment | 두 환경을 교대로 사용, 즉시 롤백 가능 |
| 카나리 배포 | Canary Deployment | 소수 사용자에게 먼저 배포 후 점진적 확대 |
| 롤링 업데이트 | Rolling Update | 인스턴스를 하나씩 순차적으로 교체 |
| A/B 테스트 | A/B Testing | 두 버전을 동시에 운영하여 성과 비교 |
| 피처 플래그 | Feature Flag | 코드는 배포하되 기능 활성화를 분리 제어 |

### 배포 관리의 핵심 활동

| 활동 | 설명 |
|------|------|
| 배포 계획 | 배포 시기, 방식, 순서 결정 |
| 환경 준비 | 대상 환경의 사전 점검 및 준비 |
| 배포 실행 | 실제 배포 수행 (자동화/수동) |
| 검증 | 배포 후 정상 작동 확인 |
| 롤백 | 문제 발생 시 이전 상태로 복원 |
| 배포 후 검토 | 배포 프로세스 개선을 위한 사후 검토 |

## 3. 인프라 및 플랫폼 관리 (Infrastructure and Platform Management)

### 정의 및 목적

| 항목 | 설명 |
|------|------|
| 목적 | 조직이 사용하는 기술 인프라를 감독하고 관리하여 서비스 지원 |
| 범위 | 물리적/가상 인프라, 클라우드 플랫폼, 네트워크, 스토리지 등 |
| 핵심 원칙 | 안정성, 확장성, 비용 효율성, 보안 |

### 관리 대상

| 대상 | 예시 |
|------|------|
| 서버 | 물리 서버, 가상 머신 (VM), 컨테이너 |
| 네트워크 | LAN, WAN, SDN, 로드 밸런서, 방화벽 |
| 스토리지 | SAN, NAS, 오브젝트 스토리지, 블록 스토리지 |
| 클라우드 플랫폼 | IaaS (AWS EC2, Azure VM), PaaS (AWS Lambda, Azure Functions) |
| 데이터베이스 | RDBMS, NoSQL, 캐시 |
| 미들웨어 | 메시지 큐, API 게이트웨이, 웹 서버 |

### 클라우드 서비스 모델

| 모델 | 영문 | 설명 | 관리 범위 |
|------|------|------|----------|
| IaaS | Infrastructure as a Service | 인프라를 서비스로 제공 | OS 이상 고객 관리 |
| PaaS | Platform as a Service | 개발 플랫폼을 서비스로 제공 | 애플리케이션만 고객 관리 |
| SaaS | Software as a Service | 소프트웨어를 서비스로 제공 | 설정/데이터만 고객 관리 |
| FaaS | Function as a Service | 함수 단위 실행 환경 제공 | 함수 코드만 고객 관리 |

### 핵심 활동

| 활동 | 설명 |
|------|------|
| 용량 계획 | 현재/미래 수요에 맞는 인프라 용량 계획 |
| 프로비저닝 | 인프라 자원의 배포 및 구성 (IaC 활용) |
| 모니터링 | 인프라 성능, 가용성, 보안 모니터링 |
| 유지보수 | 패치 적용, 업그레이드, 하드웨어 교체 |
| 보안 관리 | 접근 제어, 취약점 관리, 네트워크 보안 |
| 비용 최적화 | 미사용 자원 정리, Reserved/Spot 인스턴스 활용 |

### Infrastructure as Code (IaC)

| 항목 | 설명 |
|------|------|
| 정의 | 인프라를 코드로 정의하고 버전 관리하여 자동 프로비저닝 |
| 장점 | 반복 가능성, 일관성, 버전 관리, 자동화 |
| 도구 | Terraform, Ansible, CloudFormation, Pulumi |
| 원칙 | 선언적 정의, 멱등성 (Idempotency), 불변 인프라 (Immutable Infrastructure) |

## 4. 소프트웨어 개발 및 관리 (Software Development and Management)

### 정의 및 목적

| 항목 | 설명 |
|------|------|
| 목적 | 애플리케이션이 비즈니스 요구에 부합하도록 개발, 배포, 유지보수 |
| 범위 | 소프트웨어 수명주기 전체 (설계, 개발, 테스트, 유지보수) |
| 핵심 원칙 | 품질, 보안, 유지보수성, 확장성 |

### 개발 방법론 비교

| 방법론 | 영문 | 특징 | 적합한 상황 |
|--------|------|------|-----------|
| 폭포수 | Waterfall | 순차적 단계, 명확한 요구사항 | 규제 산업, 고정 요구사항 |
| 애자일 | Agile | 반복적, 적응적, 고객 협업 | 요구사항 변동이 큰 프로젝트 |
| 스크럼 | Scrum | 스프린트 기반, 역할 정의 | 팀 기반 소프트웨어 개발 |
| 칸반 | Kanban | 흐름 기반, WIP 제한 | 지속적 서비스 운영 |
| DevOps | DevOps | 개발+운영 통합, 자동화 | 빈번한 배포가 필요한 환경 |
| SAFe | Scaled Agile Framework | 대규모 조직의 애자일 적용 | 대기업, 다수 팀 협업 |

### 소프트웨어 수명주기 단계

| 단계 | 영문 | 주요 활동 |
|------|------|----------|
| 요구사항 분석 | Requirements Analysis | 비즈니스/기술 요구사항 수집 및 분석 |
| 설계 | Design | 아키텍처 설계, 상세 설계 |
| 개발 | Development | 코딩, 단위 테스트, 코드 리뷰 |
| 테스트 | Testing | 통합 테스트, 시스템 테스트, UAT |
| 배포 | Deployment | 운영 환경 배포 |
| 운영/유지보수 | Operations/Maintenance | 버그 수정, 기능 개선, 성능 최적화 |
| 폐기 | Retirement | 서비스 종료, 데이터 마이그레이션 |

### 핵심 활동

| 활동 | 설명 |
|------|------|
| 코드 품질 관리 | 코드 리뷰, 정적 분석, 코딩 표준 준수 |
| 버전 관리 | Git 기반 소스 코드 관리, 브랜치 전략 |
| 빌드 자동화 | CI 파이프라인을 통한 자동 빌드 및 테스트 |
| 보안 내재화 | DevSecOps, SAST/DAST, 의존성 스캔 |
| 기술 부채 관리 | 기술 부채(Technical Debt) 식별 및 해소 계획 |
| 문서화 | API 문서, 운영 가이드, 아키텍처 문서 |

## 5. DevOps/Cloud와의 연관성

ITIL 4는 DevOps, Cloud, Agile 등 현대적 접근법과의 통합을 명시적으로 지원한다.

### ITIL 4와 DevOps의 연관

| ITIL 4 프랙티스 | DevOps 연관 영역 | 통합 포인트 |
|---------------|----------------|-----------|
| 배포 관리 | CI/CD 파이프라인 | 자동화된 배포, 지속적 전달 |
| 인프라 관리 | Infrastructure as Code | Terraform, Ansible 기반 자동 프로비저닝 |
| 소프트웨어 개발 | Agile/DevOps 문화 | 개발-운영 협업, 자동화 |
| 변경 지원 | 지속적 배포 (CD) | 자동화된 변경 승인, 표준 변경 확대 |
| 모니터링/이벤트 관리 | Observability | 로깅, 메트릭, 트레이싱 통합 |
| 인시던트 관리 | SRE (Site Reliability Engineering) | 에러 버짓, 자동 복구, Chaos Engineering |

### ITIL 4와 Cloud의 연관

| 클라우드 개념 | 관련 ITIL 프랙티스 | 설명 |
|-------------|-------------------|------|
| IaaS/PaaS/SaaS | 인프라 및 플랫폼 관리 | 클라우드 서비스 모델 선택 및 관리 |
| Auto-scaling | 용량 및 성능 관리 | 자동 확장을 통한 탄력적 용량 관리 |
| Multi-cloud | 공급자 관리 | 다수 클라우드 제공자 관리 |
| Serverless | 소프트웨어 개발 및 관리 | 서버리스 아키텍처의 개발 및 관리 |
| Cloud Security | 정보보안 관리 | 클라우드 환경의 보안 통제 |
| FinOps | 서비스 재무 관리 | 클라우드 비용 최적화 |

### ITIL + DevOps + Agile 통합 프레임워크

| 프레임워크 | 제공 가치 | 적용 영역 |
|-----------|----------|----------|
| ITIL 4 | 서비스 관리 체계와 거버넌스 | 전략, 거버넌스, 프랙티스 |
| DevOps | 개발-운영 협업과 자동화 | 빌드, 배포, 운영 |
| Agile | 유연한 개발 방법론 | 요구사항 관리, 개발 프로세스 |
| Lean | 낭비 제거와 효율성 | 가치 흐름 최적화 |
| SRE | 신뢰성 엔지니어링 | 운영 안정성, 자동화 |
