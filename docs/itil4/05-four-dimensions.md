## 1. 4가지 차원 개요

ITIL 4는 효과적이고 효율적인 서비스 관리를 위해 **4가지 차원 (Four Dimensions)**을 균형 있게 고려해야 한다고 강조한다. 어느 한 차원만 집중하면 서비스가 기대한 가치를 제공하지 못할 수 있다.

| 항목 | 설명 |
|------|------|
| 목적 | 서비스 관리에 대한 전체적(Holistic) 접근 보장 |
| 적용 범위 | SVS 전체, 특히 서비스 가치 사슬의 모든 활동에 적용 |
| 관계 | 4가지 차원은 상호 의존적이며, 균형이 필요 |
| ITIL v3 대비 | v3의 "4P" (People, Process, Products, Partners) 개념의 발전 형태 |

### 4가지 차원 요약

| 번호 | 차원 (한국어) | 영문 | 핵심 질문 |
|------|------------|------|----------|
| 1 | 조직과 사람 | Organizations and People | 누가 일하고, 어떤 문화와 역할이 필요한가? |
| 2 | 정보와 기술 | Information and Technology | 어떤 정보와 기술이 필요한가? |
| 3 | 파트너와 공급자 | Partners and Suppliers | 누구와 협력하고, 어떤 관계를 맺을 것인가? |
| 4 | 가치 흐름과 프로세스 | Value Streams and Processes | 어떤 활동과 흐름으로 가치를 전달하는가? |

## 2. 조직과 사람 (Organizations and People)

서비스 관리의 성공은 사람과 조직 문화에 크게 의존한다.

### 핵심 고려사항

| 고려사항 | 설명 |
|---------|------|
| 조직 구조 | 서비스 전략을 지원하는 적절한 조직 구조 설계 |
| 역할과 책임 | 명확한 역할(Role), 책임(Responsibility), 권한(Authority) 정의 |
| 공식 구조 | 보고 체계, 부서 구분, 의사결정 구조 |
| 문화 | 공유된 가치관, 태도, 행동 양식 |
| 역량 | 필요한 기술(Skill)과 역량(Competency) 확보 |
| 리더십 | 비전 제시, 변화 주도, 팀 동기부여 |

### 조직 문화의 핵심 요소

| 요소 | 설명 | 예시 |
|------|------|------|
| 신뢰 | 팀 간 상호 신뢰 | 실패를 비난하지 않는 문화 (Blameless) |
| 투명성 | 정보의 자유로운 공유 | 장애 사후 분석 보고서 (Post-Mortem) 공유 |
| 학습 | 지속적 학습과 성장 | 정기적 교육, 기술 공유 세션 |
| 협업 | 부서 간 경계를 넘는 협력 | 크로스 펑셔널 팀 (Cross-functional Team) |

### RACI 모델

역할과 책임을 명확히 하기 위해 RACI 모델을 활용한다.

| 약어 | 영문 | 한국어 | 설명 |
|------|------|--------|------|
| R | Responsible | 실행 책임자 | 작업을 실제로 수행하는 사람 |
| A | Accountable | 최종 책임자 | 최종 승인/결정 권한을 가진 사람 (1명) |
| C | Consulted | 자문 | 의견을 구하는 사람 (양방향 소통) |
| I | Informed | 통보 | 진행 상황을 통보받는 사람 (단방향) |

## 3. 정보와 기술 (Information and Technology)

정보와 기술은 서비스 관리의 핵심 인프라이다.

### 정보 관리 고려사항

| 고려사항 | 설명 |
|---------|------|
| 정보 관리 | 서비스 관리에 필요한 정보의 식별, 수집, 저장, 활용 |
| 데이터 품질 | 정확성, 완전성, 적시성, 일관성 보장 |
| 데이터 보안 | 기밀성(Confidentiality), 무결성(Integrity), 가용성(Availability) - CIA 삼각형 |
| 규정 준수 | GDPR, 개인정보보호법 등 관련 규정 준수 |

### 기술 고려사항

| 고려사항 | 설명 |
|---------|------|
| 기술 선택 | 비즈니스 요구에 맞는 적절한 기술 선택 |
| 자동화 | 반복 작업의 자동화 기회 식별 |
| 클라우드 | IaaS, PaaS, SaaS 등 클라우드 활용 전략 |
| AI/ML | 인공지능/머신러닝 활용한 서비스 개선 |
| 통합 | 시스템 간 통합 및 데이터 흐름 관리 |

### 기술 유형별 서비스 관리 도구

| 기술 영역 | 도구/기술 예시 | 용도 |
|----------|-------------|------|
| ITSM 도구 | ServiceNow, Jira Service Management | 인시던트, 변경, 서비스 요청 관리 |
| 모니터링 | Prometheus, Grafana, Datadog | 인프라/서비스 모니터링 |
| 자동화 | Ansible, Terraform, Jenkins | 인프라 프로비저닝, CI/CD |
| 지식 관리 | Confluence, SharePoint | 지식 저장 및 공유 |
| 커뮤니케이션 | Slack, Teams | 팀 간 실시간 소통 |

## 4. 파트너와 공급자 (Partners and Suppliers)

모든 서비스를 자체적으로 제공하는 조직은 없으며, 외부 파트너와 공급자와의 관계 관리가 중요하다.

### 파트너 vs 공급자

| 구분 | 파트너 (Partner) | 공급자 (Supplier) |
|------|-----------------|------------------|
| 관계 수준 | 전략적 협력 관계 | 계약 기반 거래 관계 |
| 목표 공유 | 공동 목표 추구 | 서비스/제품 제공에 집중 |
| 리스크 공유 | 리스크와 보상을 공유 | 계약 조건에 따른 리스크 이전 |
| 기간 | 장기적 관계 | 계약 기간에 한정 |
| 예시 | 클라우드 플랫폼 파트너, 전략적 아웃소싱 | 하드웨어 공급, 네트워크 회선 제공 |

### 서비스 통합 및 관리 (SIAM)

| 모델 | 영문 | 설명 |
|------|------|------|
| 인소싱 | Insourcing | 조직 내부에서 직접 수행 |
| 아웃소싱 | Outsourcing | 외부 조직에 위탁 |
| 단일 공급자 | Single Supplier | 하나의 공급자에게 집중 |
| 멀티 공급자 | Multi-supplier | 여러 공급자의 서비스를 통합 관리 |
| 파트너십 | Partnership | 전략적 파트너와 공동 가치 창출 |

### 공급자 관계 유형

| 유형 | 특징 | 관리 수준 |
|------|------|----------|
| 상품형 (Commodity) | 표준화된 서비스/제품, 대체 가능 | 낮음 (계약 관리 중심) |
| 운영형 (Operational) | 일상적 운영 지원 서비스 | 중간 (SLA 기반 관리) |
| 전술적 (Tactical) | 특정 프로젝트/이니셔티브 지원 | 높음 (성과 기반 관리) |
| 전략적 (Strategic) | 비즈니스 전략에 핵심적인 파트너 | 최상 (경영진 수준 관리) |

## 5. 가치 흐름과 프로세스 (Value Streams and Processes)

조직이 가치를 창출하는 방식을 정의하는 차원이다.

### 가치 흐름 (Value Stream)

| 항목 | 설명 |
|------|------|
| 정의 | 조직이 수요에 대응하여 소비자를 위한 가치를 창출하는 일련의 단계 |
| 특징 | 서비스 가치 사슬의 활동을 특정 순서로 조합 |
| 설계 원칙 | Lean 사상의 가치 흐름 매핑 (Value Stream Mapping) 활용 |
| 목표 | 낭비 제거, 효율성 향상, 가치 최대화 |

### 프로세스 (Process)

| 항목 | 설명 |
|------|------|
| 정의 | 입력을 출력으로 변환하는 상호 관련된 활동의 집합 |
| 특징 | 측정 가능, 반복 가능, 특정 결과 산출 |
| 구성 요소 | 입력, 활동, 출력, 트리거, 통제 |

### 가치 흐름 vs 프로세스

| 구분 | 가치 흐름 (Value Stream) | 프로세스 (Process) |
|------|------------------------|-------------------|
| 관점 | 고객/소비자 관점 | 내부 운영 관점 |
| 범위 | 엔드투엔드 (End-to-end) | 특정 활동 범위 |
| 목표 | 가치 전달 | 효율적 실행 |
| 최적화 | 전체 흐름 최적화 | 개별 활동 최적화 |

## 6. PESTLE 외부 요인

4가지 차원은 **PESTLE 요인**이라는 외부 환경의 영향을 받는다. 조직이 통제할 수 없는 외부 변수이다.

| 요인 | 영문 | 설명 | 예시 |
|------|------|------|------|
| 정치적 | Political | 정부 정책, 규제 변화 | 데이터 지역화 요구, 정부 보안 정책 |
| 경제적 | Economic | 경제 상황, 시장 조건 | 경기 침체로 인한 IT 예산 축소 |
| 사회적 | Social | 사회적 트렌드, 인구 변화 | 재택근무 확산, 디지털 네이티브 세대 |
| 기술적 | Technological | 기술 발전, 신기술 등장 | AI/ML 활용, 클라우드 전환 트렌드 |
| 법적 | Legal | 법률, 규제, 컴플라이언스 | GDPR, 개인정보보호법, 산업별 규제 |
| 환경적 | Environmental | 환경 변화, 지속 가능성 | 그린 IT, 탄소 중립, 재난/재해 |

## 7. 4가지 차원 종합 비교

| 차원 | 핵심 질문 | 주요 관심사 | 관련 프랙티스 |
|------|----------|-----------|-------------|
| 조직과 사람 | 누가, 어떤 역할로? | 구조, 문화, 역량, 리더십 | Workforce and Talent Mgmt, Organizational Change Mgmt |
| 정보와 기술 | 어떤 정보/기술로? | 데이터 관리, 보안, 도구, 자동화 | Information Security Mgmt, Knowledge Mgmt |
| 파트너와 공급자 | 누구와 협력? | 소싱 전략, 계약, 관계 관리 | Supplier Mgmt, Relationship Mgmt |
| 가치 흐름과 프로세스 | 어떻게 전달? | 가치 흐름 설계, 프로세스 최적화 | All Service Management Practices |

### 4가지 차원의 균형

| 불균형 상태 | 발생 문제 |
|-----------|----------|
| 기술만 강조 | 사람과 프로세스 없이 도구만 도입하여 활용도 저하 |
| 프로세스만 강조 | 관료주의적 프로세스로 인한 민첩성 저하 |
| 사람만 강조 | 표준화/자동화 부족으로 인한 비효율 |
| 공급자만 의존 | 핵심 역량 내재화 실패, 공급자 종속(Vendor Lock-in) |
