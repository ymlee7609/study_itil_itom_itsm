## 1. 개요

이 문서는 ITIL 4에서 가장 빈번하게 활용되는 4가지 핵심 프랙티스의 상세 워크플로우를 다룬다. Foundation 시험에서도 가장 비중이 높은 영역이다.

| 프랙티스 | 영문 | 핵심 질문 |
|---------|------|----------|
| 인시던트 관리 | Incident Management | 서비스 중단을 어떻게 빠르게 복구하는가? |
| 문제 관리 | Problem Management | 반복 장애의 근본 원인을 어떻게 제거하는가? |
| 변경 지원 | Change Enablement | 변경을 어떻게 안전하게 적용하는가? |
| 서비스 요청 관리 | Service Request Management | 사용자 요청을 어떻게 효율적으로 처리하는가? |

## 2. 인시던트 관리 (Incident Management)

### 정의

| 항목 | 설명 |
|------|------|
| 인시던트 (Incident) | 서비스의 비계획적 중단 또는 서비스 품질의 저하 |
| 목적 | 인시던트의 부정적 영향을 최소화하고, 정상 서비스를 가능한 빨리 복구 |
| 핵심 원칙 | "원인 분석보다 서비스 복구가 우선" |

### 인시던트 분류 (Categorization)

| 분류 기준 | 예시 |
|----------|------|
| 서비스 유형 | 이메일, ERP, 네트워크, 데이터베이스 |
| 기술 영역 | 하드웨어, 소프트웨어, 네트워크, 보안 |
| 비즈니스 기능 | 재무, 인사, 영업, 제조 |

### 우선순위 (Priority) 매트릭스

우선순위는 **영향도 (Impact)**와 **긴급도 (Urgency)**의 조합으로 결정된다.

| | 높은 긴급도 | 중간 긴급도 | 낮은 긴급도 |
|---|----------|----------|----------|
| **높은 영향도** | P1 (Critical) | P2 (High) | P3 (Medium) |
| **중간 영향도** | P2 (High) | P3 (Medium) | P4 (Low) |
| **낮은 영향도** | P3 (Medium) | P4 (Low) | P4 (Low) |

### 우선순위별 대응 기준

| 우선순위 | 영문 | 목표 응답 시간 | 목표 해결 시간 | 대응 체계 |
|---------|------|-------------|-------------|----------|
| P1 | Critical | 15분 이내 | 4시간 이내 | Major Incident 프로세스 발동 |
| P2 | High | 30분 이내 | 8시간 이내 | 전문 엔지니어 즉시 배정 |
| P3 | Medium | 4시간 이내 | 24시간 이내 | 일반 처리 절차 |
| P4 | Low | 8시간 이내 | 72시간 이내 | 일반 처리 절차 |

### 에스컬레이션 (Escalation)

| 유형 | 영문 | 설명 | 예시 |
|------|------|------|------|
| 기능적 에스컬레이션 | Functional Escalation | 기술적으로 더 전문적인 팀으로 이관 | 1선 → 2선 → 3선 |
| 계층적 에스컬레이션 | Hierarchical Escalation | 관리 계층을 통해 상위 레벨에 보고 | 팀장 → 부서장 → CIO |

### 에스컬레이션 단계

| 단계 | 영문 | 역할 | 전형적 활동 |
|------|------|------|-----------|
| 1선 지원 | First Line / L1 | 서비스 데스크 | 접수, 분류, 기본 해결, 에스컬레이션 |
| 2선 지원 | Second Line / L2 | 기술 지원 팀 | 상세 진단, 기술적 해결 |
| 3선 지원 | Third Line / L3 | 전문가/벤더 | 심층 분석, 코드 수정, 벤더 지원 |

### Major Incident (중대 인시던트)

| 항목 | 설명 |
|------|------|
| 정의 | 비즈니스에 심각한 영향을 미치는 최고 수준의 인시던트 |
| 별도 프로세스 | 일반 인시던트와 구분된 별도 프로세스로 관리 |
| Major Incident Manager | 전담 매니저가 복구 활동 조율 |
| 커뮤니케이션 | 경영진, 고객에게 정기적 상태 업데이트 제공 |
| 사후 검토 (PIR) | Post-Incident Review: 사후 원인 분석 및 재발 방지 대책 수립 |

### 인시던트 관리 워크플로우

| 단계 | 활동 | 담당 |
|------|------|------|
| 1. 감지 및 기록 | 인시던트 접수, 기본 정보 기록 | 서비스 데스크 / 모니터링 시스템 |
| 2. 분류 | 서비스, 카테고리, 하위 카테고리 분류 | 서비스 데스크 |
| 3. 우선순위 결정 | 영향도 x 긴급도 = 우선순위 | 서비스 데스크 |
| 4. 초기 진단 | 알려진 해결책 검색, 기본 조치 | 1선 지원 |
| 5. 에스컬레이션 | 필요시 상위 지원 그룹으로 이관 | 1선 → 2선/3선 |
| 6. 조사 및 진단 | 상세 원인 분석, 해결 방안 도출 | 2선/3선 지원 |
| 7. 해결 및 복구 | 해결 조치 적용, 서비스 정상 복구 확인 | 담당 지원 팀 |
| 8. 종료 | 사용자 확인, 인시던트 기록 완료 | 서비스 데스크 |

## 3. 문제 관리 (Problem Management)

### 정의

| 항목 | 설명 |
|------|------|
| 문제 (Problem) | 하나 이상의 인시던트의 원인 또는 잠재적 원인 |
| 알려진 오류 (Known Error) | 근본 원인이 분석되었고, 아직 해결되지 않은 문제 |
| 임시 해결책 (Workaround) | 인시던트나 문제의 영향을 줄이거나 제거하는 해결책 (근본 해결은 아님) |
| 근본 원인 (Root Cause) | 인시던트 또는 문제를 발생시킨 근본적인 원인 |

### 문제 관리의 3가지 단계

| 단계 | 영문 | 활동 | 산출물 |
|------|------|------|--------|
| 문제 식별 | Problem Identification | 인시던트 트렌드 분석, 반복 장애 식별 | 문제 레코드 |
| 문제 통제 | Problem Control | 근본 원인 분석(RCA), 임시 해결책 도출 | 알려진 오류 (Known Error) |
| 오류 통제 | Error Control | 영구 해결책 도출, 변경 요청(RFC) 제출 | 변경 요청 (RFC) |

### 사후적 vs 사전적 문제 관리

| 구분 | 사후적 (Reactive) | 사전적 (Proactive) |
|------|------------------|-------------------|
| 트리거 | 인시던트 발생 후 | 인시던트 발생 전 |
| 활동 | 반복 인시던트의 패턴 분석 | 잠재적 문제 사전 식별 |
| 데이터 소스 | 인시던트 기록, 에스컬레이션 데이터 | 모니터링 데이터, 트렌드 분석, 위험 평가 |
| 목표 | 기존 문제 해결 | 인시던트 예방 |
| 예시 | 동일 서버 장애가 3회 발생하여 근본 원인 분석 | 디스크 사용량 트렌드 분석으로 용량 부족 사전 예방 |

### 근본 원인 분석 (RCA) 기법

| 기법 | 영문 | 설명 |
|------|------|------|
| 5 Why | 5 Whys | "왜?"를 5번 반복하여 근본 원인 추적 |
| 이시카와 다이어그램 | Ishikawa / Fishbone Diagram | 원인을 6가지 범주(6M)로 분류하여 분석 |
| 파레토 분석 | Pareto Analysis | 80/20 법칙: 상위 20% 원인이 80%의 문제 유발 |
| 장애 트리 분석 | Fault Tree Analysis (FTA) | 논리 다이어그램으로 장애 경로 분석 |
| 타임라인 분석 | Timeline Analysis | 시간 순서대로 이벤트를 나열하여 인과관계 파악 |

### 인시던트 vs 문제 비교

| 구분 | 인시던트 (Incident) | 문제 (Problem) |
|------|-------------------|---------------|
| 초점 | 서비스 복구 (빠른 해결) | 근본 원인 제거 (재발 방지) |
| 시급성 | 즉각적 대응 | 장기적 분석 |
| 해결 방식 | 임시 조치도 허용 | 근본 해결 추구 |
| 관계 | 증상 (Symptom) | 원인 (Cause) |
| KPI | MTTR (해결 시간) | 반복 인시던트 감소율 |

## 4. 변경 지원 (Change Enablement)

### 정의

| 항목 | 설명 |
|------|------|
| 변경 (Change) | 서비스나 서비스 구성요소에 영향을 줄 수 있는 추가, 수정, 제거 |
| 목적 | 리스크를 적절히 평가/관리하면서 성공적인 IT 변경을 극대화 |
| 핵심 원칙 | 변경을 차단하는 것이 아니라 "가능하게 하는 것 (Enablement)" |

### 변경 유형 상세

| 유형 | 영문 | 특징 | 승인 방식 | 예시 |
|------|------|------|----------|------|
| 표준 변경 | Standard Change | 저위험, 사전 승인, 잘 이해된 변경 | 사전 승인 (Pre-authorized) | 비밀번호 초기화, 패치 적용 |
| 일반 변경 | Normal Change | 일정에 따라 평가/승인, 리스크 수준에 따라 승인 권한 다름 | 변경 권한자 (Change Authority) | 서버 교체, 신규 애플리케이션 배포 |
| 긴급 변경 | Emergency Change | 즉시 실행 필요, 사후 검토 | 긴급 CAB (ECAB) | 보안 취약점 긴급 패치 |

### 변경 권한자 (Change Authority)

| 변경 수준 | 변경 권한자 | 예시 |
|----------|-----------|------|
| 전사적 영향 | 경영진 / IT 이사 | 핵심 ERP 시스템 교체 |
| 서비스 영향 | CAB (Change Advisory Board) | 서버 마이그레이션 |
| 표준 변경 | 사전 승인 (자동화 가능) | 비밀번호 초기화, 패치 |
| 일상 변경 | 팀 리더 / 기술 책임자 | 설정 변경, 모니터링 임계치 조정 |

### CAB (Change Advisory Board)

| 항목 | 설명 |
|------|------|
| 정의 | 변경을 평가하고 변경 권한자에게 조언하는 자문 기구 |
| 구성원 | IT 관리자, 기술 전문가, 비즈니스 대표, 보안 담당 등 |
| 활동 | 변경 요청 검토, 리스크 평가, 일정 조율, 승인 권고 |
| ECAB | Emergency CAB - 긴급 변경 시 신속히 소집되는 축소된 CAB |

### Change Schedule (변경 일정)

| 항목 | 설명 |
|------|------|
| 정의 | 승인된 변경의 실행 일정과 상태를 보여주는 달력 |
| 용도 | 변경 충돌 방지, 유지보수 윈도우 관리, 변경 동결 기간 관리 |
| 포함 정보 | 변경 ID, 실행 일시, 영향 범위, 담당자, 롤백 계획 |

### 변경 관리 워크플로우

| 단계 | 활동 | 산출물 |
|------|------|--------|
| 1. 변경 요청 | RFC (Request for Change) 제출 | 변경 요청서 |
| 2. 변경 평가 | 리스크, 영향, 비용, 이점 분석 | 변경 평가 보고서 |
| 3. 변경 승인 | 변경 권한자의 승인/거부/보류 결정 | 승인 기록 |
| 4. 변경 계획 | 상세 실행 계획, 롤백 계획 수립 | 변경 실행 계획서 |
| 5. 변경 실행 | 승인된 계획에 따라 변경 실행 | 변경 실행 기록 |
| 6. 변경 검토 | PIR (Post-Implementation Review) | 검토 보고서 |
| 7. 변경 종료 | 변경 기록 완료 및 종료 | 종료된 변경 기록 |

## 5. 서비스 요청 관리 (Service Request Management)

### 정의

| 항목 | 설명 |
|------|------|
| 서비스 요청 (Service Request) | 서비스 제공의 일부로 포함된 사전 정의된 사용자 요청 |
| 목적 | 합의된 품질과 사용자 기대를 충족하며 모든 서비스 요청을 처리 |
| 핵심 원칙 | 표준화, 자동화, 셀프서비스 극대화 |

### 서비스 요청 유형

| 유형 | 영문 | 설명 | 예시 |
|------|------|------|------|
| 정보 요청 | Information Request | 정보 제공 요청 | FAQ 안내, 서비스 정보 문의 |
| 접근 권한 요청 | Access Request | 시스템/서비스 접근 권한 요청 | VPN 계정 생성, 폴더 접근 권한 |
| 서비스 행위 요청 | Service Action Request | 표준 서비스 행위 요청 | 소프트웨어 설치, 장비 이동 |
| 피드백/불만 | Feedback/Complaint | 서비스에 대한 피드백이나 불만 제기 | 서비스 품질 불만, 개선 제안 |

### 인시던트 vs 서비스 요청

| 구분 | 인시던트 (Incident) | 서비스 요청 (Service Request) |
|------|-------------------|---------------------------|
| 성격 | 비정상적 상황 (장애) | 정상적 요청 (사전 정의됨) |
| 긴급성 | 보통 긴급 | 보통 비긴급 |
| 처리 방식 | 진단 → 해결 | 이행 절차 실행 |
| 예시 | 이메일 서비스 장애 | 새 이메일 계정 생성 요청 |
| 자동화 | 일부 가능 | 대부분 자동화 가능 |

### 셀프서비스 (Self-Service)

| 항목 | 설명 |
|------|------|
| 정의 | 사용자가 직접 서비스 요청을 제출하고 처리할 수 있는 시스템 |
| 도구 | 셀프서비스 포털, 챗봇, 자동화 워크플로우 |
| 장점 | 서비스 데스크 부하 감소, 사용자 만족도 향상, 24/7 이용 가능 |
| 자동화 대상 | 비밀번호 초기화, 소프트웨어 설치, 접근 권한 요청 |

### 서비스 요청 관리 워크플로우

| 단계 | 활동 | 담당 |
|------|------|------|
| 1. 요청 접수 | 서비스 포털/데스크를 통해 요청 접수 | 사용자 / 서비스 데스크 |
| 2. 요청 검증 | 요청의 적합성과 권한 확인 | 서비스 데스크 |
| 3. 요청 이행 | 사전 정의된 절차에 따라 요청 처리 | 이행 담당자 / 자동화 |
| 4. 완료 확인 | 요청 결과를 사용자에게 전달 및 확인 | 서비스 데스크 |
| 5. 만족도 조사 | 처리 결과에 대한 만족도 조사 | 자동화 시스템 |
| 6. 종료 | 요청 기록 완료 | 서비스 데스크 |

---

## 6. 4대 프랙티스 간 관계

| 관계 | 설명 |
|------|------|
| 인시던트 → 문제 | 반복되는 인시던트가 문제 레코드 생성을 트리거 |
| 문제 → 변경 | 문제의 근본 해결을 위해 변경 요청(RFC) 제출 |
| 변경 → 인시던트 | 잘못된 변경이 새로운 인시던트를 유발할 수 있음 |
| 서비스 요청 → 변경 | 서비스 요청 이행 시 표준 변경 실행 가능 |
| 문제 → 인시던트 | 알려진 오류의 임시 해결책을 인시던트 해결에 활용 |
| 모니터링 → 인시던트 | 예외 이벤트가 인시던트를 자동 생성 |

### 통합 시나리오 예시

| 단계 | 프랙티스 | 활동 |
|------|---------|------|
| 1 | 모니터링/이벤트 관리 | 데이터베이스 성능 저하 이벤트 감지 |
| 2 | 인시던트 관리 | 인시던트 기록, P2 우선순위 배정, 서비스 복구 (임시 조치) |
| 3 | 문제 관리 | 동일 유형 인시던트 3회 발생 → 문제 레코드 생성, RCA 수행 |
| 4 | 문제 관리 | 근본 원인: 인덱스 누락 → Known Error 등록 |
| 5 | 변경 지원 | 인덱스 추가를 위한 RFC 제출 → CAB 승인 |
| 6 | 배포 관리 | 변경 사항(인덱스 추가) 운영 환경에 배포 |
| 7 | 문제 관리 | 문제 레코드 종료, Known Error 해결 표시 |
