## 1. 핵심 비교표

| 관점 | ITOM | ITSM |
|------|------|------|
| **정의** | IT 인프라 및 운영 관리 | IT 서비스 전달 및 관리 |
| **초점** | 기술 (Technology) | 서비스 (Service) |
| **대상** | IT 직원 / 운영팀 | 최종 사용자 / 고객 |
| **핵심 활동** | 모니터링, 용량 계획, 성능 최적화 | 서비스 요청, 인시던트, 문제, 변경 관리 |
| **주요 목표** | 효율적이고 안정적인 IT 운영 | 고품질 IT 서비스 제공 |
| **위치** | "비하인드 더 씬" (Behind the Scenes) | "프론트 오브 하우스" (Front of House) |
| **관계** | ITSM의 기반 (Foundation) | ITOM 위에서 서비스 전달 |
| **비유** | 자동차 엔진, 변속기, 전기 시스템 | 운전 경험, 고객 서비스, 딜러십 |
| **ITIL 4 연계** | 기술 관리 실천 (Technical Management Practices) | 서비스 관리 실천 (Service Management Practices) |
| **자동화 초점** | 인프라 자동화, 프로비저닝 | 워크플로우 자동화, 셀프서비스 |

## 2. 상세 비교

### 2.1 프로세스 비교

| 프로세스 영역 | ITOM 프로세스 | ITSM 프로세스 |
|-------------|-------------|-------------|
| 이벤트/인시던트 | 이벤트 관리 (Event Management) | 인시던트 관리 (Incident Management) |
| 구성/자산 | 디스커버리 (Discovery), 서비스 매핑 | CMDB 관리, IT 자산 관리 |
| 변경 | 자동 배포, 패치 관리 | 변경 관리 (Change Management) |
| 용량/성능 | 용량 계획, 성능 모니터링 | SLA 관리, 서비스 수준 관리 |
| 보안 | 보안 모니터링, 취약점 스캔 | 보안 인시던트 관리 |
| 클라우드 | 클라우드 프로비저닝, 비용 관리 | 클라우드 서비스 카탈로그 |

### 2.2 도구 비교

| 영역 | ITOM 도구 | ITSM 도구 |
|------|----------|----------|
| 모니터링 | Datadog, Zabbix, Nagios, Prometheus | - |
| 로그 관리 | Splunk, ELK Stack | - |
| 서비스 데스크 | - | ServiceNow, Jira SM, Freshservice |
| 자동화 | Ansible, Terraform, Puppet | 워크플로우 엔진 (ServiceNow Flow) |
| 통합 플랫폼 | ServiceNow ITOM | ServiceNow ITSM |
| 클라우드 | AWS CloudWatch, Azure Monitor | 클라우드 서비스 카탈로그 |

### 2.3 KPI 비교

| 측정 영역 | ITOM KPI | ITSM KPI |
|-----------|---------|---------|
| 가용성 | 시스템 가동률 (Uptime %) | 서비스 가용성 (Service Availability %) |
| 응답 시간 | 서버 응답 시간 (ms) | 인시던트 응답 시간 (분/시간) |
| 복구 | 평균 복구 시간 (MTTR) | 인시던트 해결 시간 |
| 변경 | 배포 성공률 | 변경 성공률 |
| 효율성 | 자동화율 (%) | 첫 번째 연락 해결률 (FCR) |
| 비용 | 인프라 비용 / 사용량 | 티켓당 비용 (Cost per Ticket) |

## 3. ITOM과 ITSM의 관계

### "기반 관계 (Foundation Relationship)"

ITOM은 성공적인 ITSM의 **필수 기반**이다. ITOM 없이는 ITSM이 효과적으로 작동할 수 없다.

| 관계 유형 | 설명 | 예시 |
|-----------|------|------|
| 데이터 공급 | ITOM이 ITSM에 인프라 데이터 제공 | 디스커버리 결과 → CMDB 자동 갱신 |
| 이벤트-인시던트 연결 | ITOM 이벤트가 ITSM 인시던트 자동 생성 | 서버 다운 감지 → 인시던트 티켓 생성 |
| 영향 분석 | ITOM 서비스 맵이 ITSM 영향 범위 제공 | 변경 요청 시 영향받는 서비스 표시 |
| 자동 복구 | ITOM 자동화가 ITSM 인시던트 자동 해결 | 디스크 용량 초과 → 자동 정리 → 인시던트 해결 |
| 성능 데이터 | ITOM 성능 지표가 ITSM SLA 측정 지원 | 응답 시간 데이터 → SLA 달성 여부 확인 |

### 정보 흐름도

```
[ITOM 영역]                        [ITSM 영역]
디스커버리 ──────────────────────→ CMDB
이벤트 관리 ─────────────────────→ 인시던트 관리
서비스 매핑 ─────────────────────→ 변경 관리 (영향 분석)
성능 모니터링 ───────────────────→ SLA 관리
클라우드 관리 ───────────────────→ 서비스 카탈로그
자동화/오케스트레이션 ──────────→ 서비스 요청 이행
```

## 4. 통합 시나리오

### 시나리오 1: 이벤트에서 인시던트까지

| 단계 | 역할 | 활동 | 시스템 |
|------|------|------|--------|
| 1 | ITOM | 서버 CPU 사용률 95% 초과 이벤트 감지 | 모니터링 시스템 |
| 2 | ITOM | 이벤트 상관분석, 근본 원인 식별 | 이벤트 관리 |
| 3 | ITOM → ITSM | 자동으로 인시던트 티켓 생성 | ITSM 연동 |
| 4 | ITSM | 우선순위 지정, 담당자 배정 | 인시던트 관리 |
| 5 | ITOM | 자동 복구 시도 (서비스 재시작 등) | 자동화 엔진 |
| 6 | ITSM | 인시던트 해결 확인 및 종료 | 인시던트 관리 |

### 시나리오 2: 디스커버리에서 CMDB까지

| 단계 | 역할 | 활동 |
|------|------|------|
| 1 | ITOM | 네트워크 자동 스캔, 신규 디바이스 발견 |
| 2 | ITOM | 디바이스 속성 (OS, IP, 소프트웨어 등) 수집 |
| 3 | ITOM → ITSM | CI (Configuration Item) 자동 생성/갱신 |
| 4 | ITSM | CMDB 정확도 향상, 관계 매핑 |
| 5 | ITSM | 변경 관리, 영향 분석에 최신 데이터 활용 |

### 시나리오 3: 클라우드 프로비저닝

| 단계 | 역할 | 활동 |
|------|------|------|
| 1 | ITSM | 사용자가 서비스 카탈로그에서 VM 요청 |
| 2 | ITSM | 승인 워크플로우 처리 |
| 3 | ITSM → ITOM | 승인 완료, 프로비저닝 요청 전달 |
| 4 | ITOM | 클라우드 자동 프로비저닝 실행 |
| 5 | ITOM | 디스커버리로 새 VM 등록, 모니터링 설정 |
| 6 | ITSM | 서비스 요청 완료 처리, 사용자 통지 |

## 5. ITOM + ITSM + ITBM 통합 프레임워크

**ITBM (IT Business Management)**은 IT 투자와 비즈니스 전략을 연계하는 영역이다. 세 영역의 통합은 IT의 비즈니스 가치를 극대화한다.

| 영역 | 역할 | 핵심 질문 |
|------|------|-----------|
| ITBM | IT 전략 및 투자 관리 | "어떤 IT에 투자해야 하는가?" |
| ITSM | IT 서비스 전달 및 관리 | "IT 서비스를 어떻게 제공할 것인가?" |
| ITOM | IT 인프라 운영 관리 | "IT 인프라를 어떻게 운영할 것인가?" |

### 통합 데이터 흐름

| 방향 | 데이터/정보 | 목적 |
|------|-----------|------|
| ITOM → ITSM | 이벤트, 성능, 디스커버리 데이터 | 인시던트 생성, CMDB 갱신 |
| ITSM → ITOM | 변경 요청, 서비스 요청 | 인프라 변경, 프로비저닝 |
| ITOM → ITBM | 인프라 비용, 사용량 데이터 | 투자 분석, 비용 최적화 |
| ITSM → ITBM | 서비스 비용, SLA 데이터 | 서비스 포트폴리오, 재무 관리 |
| ITBM → ITSM | 투자 우선순위, 예산 | 서비스 개선, 신규 서비스 기획 |
| ITBM → ITOM | 인프라 투자 계획 | 용량 계획, 기술 로드맵 |

### ServiceNow 플랫폼에서의 통합

| 모듈 | 영역 | 주요 기능 |
|------|------|-----------|
| ITOM Visibility | ITOM | 디스커버리, 서비스 매핑, CMDB |
| ITOM Health | ITOM | 이벤트 관리, AIOps, 헬스 로그 분석 |
| ITOM Optimization | ITOM | 클라우드 관리, 비용 최적화 |
| ITSM | ITSM | 인시던트, 문제, 변경, 서비스 요청 |
| ITSM Pro/Enterprise | ITSM | AI 지원, 가상 에이전트, 예측 인텔리전스 |
| SPM (Strategic Portfolio Management) | ITBM | 포트폴리오 관리, 재무 관리 |

## 6. 역할 및 조직 구조

### ITOM 역할

| 역할 | 책임 | 필요 역량 |
|------|------|-----------|
| IT 운영 관리자 (IT Operations Manager) | 전반적인 IT 운영 관리 | 인프라 지식, 리더십, 프로세스 관리 |
| 시스템 관리자 (System Administrator) | 서버, OS 관리 | Linux/Windows, 스크립팅, 자동화 |
| 네트워크 관리자 (Network Administrator) | 네트워크 인프라 관리 | 네트워크 프로토콜, 보안, 모니터링 |
| 클라우드 엔지니어 (Cloud Engineer) | 클라우드 환경 관리 | AWS/Azure/GCP, IaC, 컨테이너 |
| SRE (Site Reliability Engineer) | 서비스 신뢰성 확보 | 코딩, 모니터링, 자동화, SLI/SLO |

### ITSM 역할

| 역할 | 책임 | 필요 역량 |
|------|------|-----------|
| 서비스 데스크 매니저 (Service Desk Manager) | 서비스 데스크 운영 | 고객 서비스, 팀 관리, ITIL |
| 인시던트 매니저 (Incident Manager) | 인시던트 프로세스 관리 | 위기 관리, 커뮤니케이션 |
| 문제 관리자 (Problem Manager) | 근본 원인 분석, 문제 해결 | 분석적 사고, 기술 지식 |
| 변경 관리자 (Change Manager) | 변경 프로세스 관리 | 리스크 관리, 프로세스 설계 |
| 서비스 수준 관리자 (SLM) | SLA 관리, 서비스 품질 | 비즈니스 이해, 협상, 분석 |
