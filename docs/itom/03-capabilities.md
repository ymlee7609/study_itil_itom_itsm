## 역량 개요

| 역량 | 핵심 목적 | 입력 | 출력 |
|------|-----------|------|------|
| 이벤트 관리 | 이벤트를 실행 가능한 인사이트로 변환 | 모니터링 알림, 로그 | 상관분석 결과, 인시던트 |
| 모니터링 | 실시간 성능/가용성 측정 | 메트릭, 로그, 트레이스 | 대시보드, 알림, 이벤트 |
| 디스커버리 | IT 자산 자동 식별 | 네트워크 스캔 | CI 데이터, CMDB 갱신 |
| 서비스 매핑 | 구성요소-서비스 관계 시각화 | 디스커버리 데이터 | 서비스 맵, 의존관계 |
| 클라우드 관리 | 클라우드 리소스 최적화 | 클라우드 API 데이터 | 비용 보고, 자동 프로비저닝 |
| 구성 관리 | 시스템 설정 표준화 | 정책, 베이스라인 | 컴플라이언스 보고 |
| 자동화/오케스트레이션 | 루틴 작업 자동 실행 | 트리거, 워크플로우 | 작업 완료, 상태 보고 |
| 분석/보고 | 데이터 기반 의사결정 | 메트릭, 이벤트, 로그 | 트렌드 보고, 예측 |

## 1. 이벤트 관리 (Event Management)

이벤트 데이터를 **실행 가능한 인사이트 (Actionable Insights)**로 변환하는 프로세스이다.

| 항목 | 설명 |
|------|------|
| 목적 | 대량의 이벤트에서 의미 있는 정보를 추출하고 적절한 대응 수행 |
| 핵심 과제 | 이벤트 노이즈 감소 (Noise Reduction), 근본 원인 도출 |
| ITIL 4 연계 | [모니터링 및 이벤트 관리 실천](../itil4/01-overview.md) |

### 이벤트 처리 파이프라인

| 단계 | 활동 | 설명 |
|------|------|------|
| 1. 수집 (Collection) | 이벤트 수신 | 다양한 소스에서 이벤트 데이터 수집 |
| 2. 필터링 (Filtering) | 노이즈 제거 | 중복, 불필요한 이벤트 제거 |
| 3. 정규화 (Normalization) | 표준 형식 변환 | 다양한 형식의 이벤트를 통일된 형식으로 변환 |
| 4. 상관분석 (Correlation) | 관련 이벤트 그룹화 | 시간/공간/인과 관계로 이벤트 연결 |
| 5. 분류 (Classification) | 심각도 판단 | 정보/경고/예외/위기로 분류 |
| 6. 조치 (Action) | 대응 실행 | 자동 복구, 인시던트 생성, 알림 |

### 이벤트 분류 체계

| 유형 | 심각도 | 대응 | 예시 |
|------|--------|------|------|
| 정보 (Informational) | 낮음 | 로깅만 수행 | 백업 완료, 서비스 시작 |
| 경고 (Warning) | 중간 | 모니터링 강화 | 디스크 사용률 80% |
| 예외 (Exception) | 높음 | 즉시 대응 필요 | 서비스 장애, 보안 침입 |
| 위기 (Critical) | 최고 | 에스컬레이션 | 핵심 서비스 중단 |

## 2. 모니터링 (Monitoring)

서버, 네트워크, 애플리케이션의 **실시간 성능 및 가용성**을 측정하는 활동이다.

### 모니터링 유형

| 유형 | 대상 | 주요 메트릭 | 도구 예시 |
|------|------|-----------|----------|
| 인프라 모니터링 (Infrastructure Monitoring) | 서버, VM, 컨테이너 | CPU, 메모리, 디스크, 네트워크 | Zabbix, Datadog, Prometheus |
| APM (Application Performance Monitoring) | 애플리케이션 | 응답 시간, 처리량, 에러율 | Datadog APM, New Relic, Dynatrace |
| 네트워크 모니터링 (Network Monitoring) | 스위치, 라우터, 방화벽 | 대역폭, 지연, 패킷 손실 | SolarWinds, PRTG, Nagios |
| 로그 모니터링 (Log Monitoring) | 시스템/앱 로그 | 에러율, 패턴, 이상 | Splunk, ELK Stack, Fluentd |
| 합성 모니터링 (Synthetic Monitoring) | 사용자 경험 시뮬레이션 | 가용성, 트랜잭션 시간 | Datadog Synthetics, Catchpoint |
| RUM (Real User Monitoring) | 실제 사용자 경험 | 페이지 로드 시간, 에러 | Datadog RUM, New Relic Browser |

### 관측성 3대 핵심 요소 (Observability Pillars)

| 요소 | 설명 | 용도 |
|------|------|------|
| 메트릭 (Metrics) | 수치 데이터 (시계열) | 트렌드 분석, 알림 |
| 로그 (Logs) | 이벤트 기록 (텍스트) | 디버깅, 감사 추적 |
| 트레이스 (Traces) | 요청 흐름 추적 | 병목 식별, 성능 분석 |

## 3. 디스커버리 (Discovery)

네트워크를 자동으로 스캔하여 **디바이스, 소프트웨어, 클라우드 리소스를 식별**하는 프로세스이다.

| 항목 | 설명 |
|------|------|
| 목적 | IT 환경의 모든 구성요소를 자동으로 파악하여 CMDB 정확도 향상 |
| 범위 | 물리 서버, VM, 컨테이너, 네트워크 장비, 클라우드 리소스, 소프트웨어 |
| 결과물 | CI (Configuration Item) 데이터, 관계 정보 |

### 디스커버리 방식 비교

| 항목 | 에이전트 기반 (Agent-based) | 에이전트리스 (Agentless) |
|------|--------------------------|------------------------|
| 방식 | 대상 시스템에 에이전트 설치 | SSH, WMI, SNMP 등 원격 접속 |
| 데이터 깊이 | 깊음 (실시간, 상세) | 보통 (스케줄 기반, 기본 정보) |
| 배포 난이도 | 높음 (설치 필요) | 낮음 (설치 불필요) |
| 네트워크 부하 | 낮음 (로컬 수집) | 보통 (원격 쿼리) |
| 보안 영향 | 에이전트 취약점 리스크 | 인증 정보 관리 필요 |
| 적합 환경 | 상세 모니터링 필요 환경 | 보안 제약 높은 환경 |

### 디스커버리 대상

| 계층 | 수집 정보 | 활용 |
|------|----------|------|
| 하드웨어 (Hardware) | 모델, 시리얼, CPU, 메모리, 스토리지 | 자산 관리, 용량 계획 |
| OS/플랫폼 (Platform) | OS 버전, 패치 수준, 서비스 | 패치 관리, 컴플라이언스 |
| 소프트웨어 (Software) | 설치된 SW, 버전, 라이선스 | 라이선스 관리, 취약점 스캔 |
| 네트워크 (Network) | IP, MAC, 포트, 연결 관계 | 네트워크 토폴로지, 장애 분석 |
| 클라우드 (Cloud) | 인스턴스, 서비스, 태그 | 비용 관리, 거버넌스 |

## 4. 서비스 매핑 (Service Mapping)

IT 구성요소와 **비즈니스 서비스 간의 관계를 시각화**하는 프로세스이다.

| 항목 | 설명 |
|------|------|
| 목적 | IT 인프라가 비즈니스 서비스에 미치는 영향을 이해 |
| 핵심 가치 | 변경 영향 분석, 장애 영향 범위 파악, 우선순위 결정 |
| 구성 요소 | 서비스 맵, 의존관계 맵, 비즈니스 영향 분석 |

### 서비스 맵 구성 요소

| 구성 요소 | 설명 | 예시 |
|-----------|------|------|
| 비즈니스 서비스 (Business Service) | 비즈니스 기능 단위 | 온라인 결제, 이메일 |
| 애플리케이션 서비스 (Application Service) | 앱 컴포넌트 | 웹서버, API, 데이터베이스 |
| 인프라 구성요소 (Infrastructure CI) | 물리/가상 자원 | 서버, 네트워크, 스토리지 |
| 의존관계 (Dependency) | 구성요소 간 관계 | 웹서버 → API → DB |

### 활용 시나리오

| 시나리오 | ITOM 활용 | ITSM 활용 |
|----------|----------|----------|
| 장애 발생 | 영향받는 비즈니스 서비스 즉시 파악 | 인시던트 우선순위 자동 설정 |
| 변경 계획 | 변경 영향 범위 사전 분석 | [변경 관리](../itsm/) 리스크 평가 |
| 용량 계획 | 서비스별 리소스 사용 현황 | SLA 기반 용량 결정 |
| 비용 분석 | 서비스별 인프라 비용 산출 | 서비스 비용 모델링 |

## 5. 클라우드 관리 (Cloud Management)

**멀티클라우드/하이브리드 환경**에서의 리소스 관리 및 최적화이다.

| 영역 | 설명 | 주요 활동 |
|------|------|-----------|
| 소비 관리 (Consumption Management) | 클라우드 리소스 사용량 추적 | 비용 분석, 예산 설정, 이상 비용 탐지 |
| 자동 프로비저닝 (Auto Provisioning) | IaC 기반 자동 배포 | Terraform, CloudFormation, Ansible |
| 컴플라이언스 (Compliance) | 클라우드 보안/규정 준수 | 정책 검증, 감사 로그, 설정 검사 |
| 비용 인사이트 (Cost Insights) | 비용 최적화 권장사항 | 미사용 리소스, 예약 인스턴스, 라이트사이징 |

### 멀티클라우드/하이브리드 관리

| 관리 과제 | 단일 클라우드 | 멀티클라우드/하이브리드 |
|-----------|-------------|---------------------|
| 가시성 | 단일 콘솔 | 통합 대시보드 필요 |
| 비용 관리 | 단일 과금 체계 | 복수 과금 통합, 비교 분석 |
| 보안 | 단일 보안 모델 | 통합 보안 정책, 멀티 IAM |
| 거버넌스 | 단일 정책 | 크로스 클라우드 정책 동기화 |
| 네트워크 | 단일 VPC | 클라우드 간 연결, 지연 관리 |

## 6. 구성 관리 (Configuration Management)

시스템 **업데이트, 패칭, 컴플라이언스를 자동화**하는 프로세스이다.

| 항목 | 설명 |
|------|------|
| 목적 | 시스템 설정의 일관성 유지 및 보안 취약점 해소 |
| 핵심 원칙 | Infrastructure as Code (IaC), 불변 인프라 (Immutable Infrastructure) |
| 도구 | Ansible, Puppet, Chef, SaltStack |

### 구성 관리 활동

| 활동 | 설명 | 자동화 수준 |
|------|------|-----------|
| 패치 관리 (Patch Management) | OS/SW 보안 업데이트 적용 | 높음 |
| 베이스라인 관리 (Baseline Management) | 표준 설정 정의 및 적용 | 높음 |
| 드리프트 탐지 (Drift Detection) | 표준 대비 설정 변경 감지 | 자동 |
| 컴플라이언스 검사 (Compliance Check) | 보안 정책 준수 여부 확인 | 자동 |
| 롤백 (Rollback) | 문제 발생 시 이전 설정 복원 | 반자동 |

## 7. 자동화 및 오케스트레이션 (Automation & Orchestration)

**루틴 작업을 자동화**하고 복잡한 워크플로우를 조율하는 역량이다.

| 구분 | 자동화 (Automation) | 오케스트레이션 (Orchestration) |
|------|-------------------|----------------------------|
| 범위 | 단일 작업 | 다중 작업의 조율 |
| 복잡도 | 낮음~중간 | 높음 |
| 예시 | 서버 재시작, 로그 정리 | 전체 배포 파이프라인, DR 절체 |
| 도구 | cron, systemd, Ansible | ServiceNow Flow, Rundeck, AWX |

### 자동화 대상 영역

| 영역 | 자동화 예시 | 효과 |
|------|-----------|------|
| 프로비저닝 | VM 생성, 네트워크 설정 | 배포 시간 단축 |
| 패치 관리 | 자동 패치 적용, 테스트 | 보안 취약점 해소 |
| 장애 복구 | 자동 재시작, 페일오버 | MTTR 단축 |
| 용량 관리 | 오토스케일링, 리소스 조정 | 비용 최적화 |
| 보안 대응 | 자동 격리, 방화벽 규칙 | 대응 시간 단축 |

## 8. 분석 및 보고 (Analytics & Reporting)

**예측 분석, 머신러닝, 이상 탐지**를 활용한 데이터 기반 운영 의사결정이다.

| 분석 유형 | 설명 | 활용 |
|-----------|------|------|
| 서술적 분석 (Descriptive) | "무엇이 발생했는가?" | 대시보드, 보고서 |
| 진단적 분석 (Diagnostic) | "왜 발생했는가?" | 근본 원인 분석, 상관분석 |
| 예측적 분석 (Predictive) | "무엇이 발생할 것인가?" | 용량 예측, 장애 예측 |
| 처방적 분석 (Prescriptive) | "어떻게 해야 하는가?" | 최적화 권장, 자동 조치 |

### 주요 보고서

| 보고서 | 내용 | 대상 |
|--------|------|------|
| 가용성 보고서 (Availability Report) | 서비스/인프라 가동률 | IT 관리자, 경영진 |
| 성능 보고서 (Performance Report) | 응답 시간, 처리량, 리소스 사용 | 운영팀 |
| 용량 보고서 (Capacity Report) | 현재 사용량, 트렌드, 예측 | 용량 관리자 |
| 비용 보고서 (Cost Report) | 인프라 비용, 최적화 기회 | 재무팀, 경영진 |
| 컴플라이언스 보고서 | 정책 준수율, 미준수 항목 | 보안팀, 감사팀 |
