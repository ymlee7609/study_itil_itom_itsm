# Policy vs Standard vs Procedure vs Guideline

거버넌스·컴플라이언스 문서의 **4단 계층**. 한국어로 "규정"으로 뭉쳐 쓰기 쉽지만, **구속력·상세도·승인 주체**가 다르다.

## 1. 공식 정의

| 구분 | Policy (정책) | Standard (표준) | Procedure (절차) | Guideline (지침) |
|------|--------------|-----------------|------------------|-----------------|
| **정의** | 조직이 따를 **원칙·방향** | Policy를 구현하는 **측정 가능한 요구사항** | Standard를 달성하는 **단계별 실행 방법** | **권고 사항** (구속력 없음) |
| **구속력** | **필수** (MUST) | **필수** (MUST) | **필수** (MUST) | 권고 (SHOULD/MAY) |
| **상세도** | 최소 (원칙 수준) | 중간 (요구사항) | 최대 (Step-by-Step) | 가변 |
| **승인** | 경영진·이사회 | CTO·CSO 급 | 부서장 | 담당자 |
| **변경 빈도** | 낮음 (연간) | 중간 (분기) | 높음 (월간) | 높음 |
| **예** | "모든 데이터는 보호되어야 한다" | "중요 데이터는 AES-256 암호화" | "데이터 암호화 설정 10단계" | "가능하면 관리형 KMS 사용 권장" |

## 2. 문서 계층

```
Top-down (구속력 ↑, 상세도 ↓)
         │
         ▼
   ┌─────────────┐
   │   Policy    │ "왜·무엇을" (원칙)
   │   (정책)    │  경영진 승인
   └──────┬──────┘
          │ 구현
          ▼
   ┌─────────────┐
   │  Standard   │ "무엇을 얼마나" (요구사항)
   │  (표준)     │  CTO·CSO 승인
   └──────┬──────┘
          │ 실행
          ▼
   ┌─────────────┐
   │  Procedure  │ "어떻게 단계별로" (절차)
   │  (절차)     │  부서장 승인
   └──────┬──────┘
          │ 보완
          ▼
   ┌─────────────┐
   │  Guideline  │ "가능하면 이렇게" (권고)
   │  (지침)     │  팀 참고
   └─────────────┘
```

## 3. 실무 예시

### 예시 1: 정보보안 문서 체계

| 문서 | 내용 |
|------|------|
| **Policy** | "회사의 모든 정보 자산은 기밀성·무결성·가용성을 보장해야 한다." |
| **Standard** | "중요 등급(Confidential) 데이터는 저장 시 AES-256, 전송 시 TLS 1.3 이상 암호화해야 한다." |
| **Procedure** | "데이터베이스 암호화 설정 절차: ①SSL 인증서 발급 → ②RDS 파라미터 수정 → ③KMS 키 생성 → ... (10단계)" |
| **Guideline** | "가능하다면 자체 키 관리보다 AWS KMS 등 관리형 서비스 사용을 권장한다." |

### 예시 2: 변경 관리 문서 체계

| 문서 | 내용 |
|------|------|
| Policy | "모든 운영 환경 변경은 승인과 기록을 거쳐야 한다." |
| Standard | "일반 변경은 48시간 사전 공지, 긴급 변경은 eCAB 사후 24시간 승인 필수." |
| Procedure | "Emergency Change 처리 절차: ①티켓 생성 → ②현장 실행 → ③eCAB 호출 → ... (7단계)" |
| Guideline | "가능하면 Maintenance Window 내에 변경 수행 권장." |

### 예시 3: 혼동 사례

| 잘못된 표현 | 문제 | 올바른 표현 |
|------------|------|-----------|
| "암호화 정책 AES-256" | Policy인지 Standard인지 불명 | "암호화 **표준(Standard)**: AES-256" |
| "비밀번호 지침 12자 이상 **필수**" | Guideline인데 구속력 표현 | "비밀번호 **표준(Standard)**: 12자 이상" |
| "Procedure로 원칙 기술" | 상세도 부족 | "Policy로 원칙을, Procedure로 실행을 분리 기술" |

## 4. 왜 이 구분이 중요한가

### (1) 감사·컴플라이언스 대응

- ISO 27001, ISMS-P, SOX 등은 **Policy → Standard → Procedure 계층 증명** 요구
- 계층이 무너지면 감사 미흡 지적
- 증거 자료 체계도 계층별로 분리

### (2) 변경 난이도·승인 복잡도

| 문서 | 변경 승인 부담 |
|------|---------------|
| Policy | 이사회/경영진 (고비용) |
| Standard | 기술 리더십 (중비용) |
| Procedure | 부서장 (저비용) |
| Guideline | 담당자 (최저) |

→ Policy에 Procedure 내용이 섞이면 **사소한 절차 변경에도 이사회 결의 필요**해짐.

### (3) 위반 시 대응 차이

| 위반 | 결과 |
|------|------|
| Policy 위반 | 중대 징계·법적 책임 (원칙 무시) |
| Standard 위반 | 감사 지적, 개선 요구 |
| Procedure 위반 | 재교육, 프로세스 개선 |
| Guideline 미준수 | **제재 없음** (권고였으므로) |

### (4) 신입 교육·온보딩

- 전체 문서를 **Policy → Procedure 순서로 전달**해야 맥락 이해
- Procedure부터 주면 "왜 이렇게 하는지 모르고 기계적으로 수행"

### (5) 변경 영향 범위

- Procedure 변경: 해당 팀만
- Standard 변경: 여러 팀·시스템 영향
- Policy 변경: **조직 전체** 영향 (모든 하위 문서 재검토 필요)

### (6) RFC (Request for Comments) 연결

- Guideline은 RFC의 "SHOULD/MAY"
- Standard는 RFC의 "MUST"
- Policy는 RFC로 표현되지 않는 조직 **자체의 의지**

## 5. 한눈 정리

```
            구속력 ↑                       상세도 ↑
    │                                           │
    │                                           │
Policy ───► Standard ───► Procedure ───► Work Instruction
 (원칙)      (요구사항)    (단계별 실행)   (작업 지시서)
 MUST        MUST          MUST           MUST
  
                     ───────────────
Guideline (별도 트랙, 권고 사항)
SHOULD / MAY — 구속력 없음
```

---

## 한 줄 요약

**Policy는 원칙(왜·무엇), Standard는 측정 가능한 요구사항(얼마나), Procedure는 단계별 실행(어떻게), Guideline은 권고(SHOULD).** 앞 셋은 구속력 있고, Guideline만 선택적. 계층을 섞으면 감사·변경 관리가 무너진다.
