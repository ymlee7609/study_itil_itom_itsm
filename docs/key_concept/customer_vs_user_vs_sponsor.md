# Customer vs User vs Sponsor

세 단어 모두 "고객"으로 번역돼 뭉뚱그려 쓰지만, **결정권자·계약자·실사용자**는 각각 다른 사람이다. 요구사항 수집·SLA 협상·커뮤니케이션 실패의 주 원인.

## 1. 공식 정의

| 구분 | Customer (고객) | User (사용자) | Sponsor (스폰서) |
|------|----------------|--------------|-----------------|
| **정의** | 서비스의 **요구사항을 정의하고 결과에 책임**지는 사람 | 서비스를 **실제로 사용**하는 사람 | 서비스 사용 **예산을 승인**하는 사람 |
| **관심사** | 비즈니스 outcome, SLA 협상 | 일상 업무 편의, UX | ROI, 예산 집행 타당성 |
| **권한** | 요구사항 확정, SLA 서명 | 사용 중 피드백 | **구매 결정, 예산 배정** |
| **ITIL 4 재정의** | 서비스 소비자의 3가지 역할 중 하나 | 서비스 소비자의 3가지 역할 중 하나 | 서비스 소비자의 3가지 역할 중 하나 |

ITIL 4에서는 **서비스 소비자(Service Consumer)** = Customer + User + Sponsor 3가지 역할로 정의.

---

## 2. 역할 관계

```
        서비스 소비자 (Service Consumer) 조직
 ┌──────────────────────────────────────────────┐
 │                                              │
 │   Sponsor (예산 승인)                         │
 │     │                                        │
 │     ▼                                        │
 │   Customer (요구사항 정의, SLA 서명)          │
 │     │                                        │
 │     ▼                                        │
 │   User (실제 사용, 피드백)                    │
 │                                              │
 └──────────────────────────────────────────────┘
                    ▲
                    │ 서비스 제공
                    │
            서비스 제공자 (Provider)
```

- **한 사람이 여러 역할**을 겸할 수 있음 (중소기업 CEO = 3역 동시)
- 대기업은 **3역이 완전 분리**됨 → 커뮤니케이션 복잡도 증가

## 3. 실무 예시

### 예시 1: 기업용 CRM 도입

| 역할 | 해당 인물 | 관심사 |
|------|-----------|--------|
| **Sponsor** | CFO / CEO | "예산 5억 투입 대비 ROI 어떻게 나오나?" |
| **Customer** | 영업본부장 | "영업 전환율 15% 개선, 월 리포트 자동화 필요" |
| **User** | 영업사원 100명 | "화면 빨라야 하고, 모바일에서도 잘 보여야 함" |

→ Sponsor만 만나면 ROI 수치에 매몰. User만 만나면 "예쁜 UI" 편향. Customer가 균형 유지.

### 예시 2: 사내 이메일 서비스

| 역할 | 인물 | 요구 |
|------|------|------|
| Sponsor | IT 임원 | "라이선스 비용 절감" |
| Customer | 전사 IT서비스팀장 | "가용성 99.9%, 스팸 필터 강화" |
| User | 전 직원 | "빠른 로그인, 검색 잘 되기" |

### 예시 3: 경계가 흐려지는 경우

| 상황 | 설명 |
|------|------|
| **B2C 서비스 (예: Netflix)** | Customer = User = Sponsor (한 사람이 3역) |
| **B2B2C (예: 통신 회선 납품)** | 통신사 Customer, 기업 IT팀 User, 기업 CFO Sponsor, 일반 직원 End-User |
| **SaaS 무료 티어** | User만 존재, Customer/Sponsor 미정의 상태 (유료 전환 시 정립) |

## 4. 왜 이 구분이 중요한가

### (1) 요구사항 수집 실패 방지

| 실패 사례 | 원인 |
|----------|------|
| Sponsor 의견만 반영 | 현장 User 불만 폭발 → 사용 저조 |
| User 의견만 반영 | 비즈니스 가치 불명확 → 예산 삭감 |
| Customer 의견만 반영 | Sponsor 승인 불투명, User 체감 부족 |

→ **3역 모두 인터뷰** 필수.

### (2) SLA 협상 주체 명확화

- SLA 서명: **Customer** (Sponsor 아님)
- XLA 측정 대상: **User** (Customer 만족도와 다를 수 있음)
- 예산 재협상: **Sponsor**

### (3) 에스컬레이션 경로 설계

| 이슈 유형 | 에스컬레이션 |
|----------|-------------|
| 일상 기능 불편 | User → Service Desk |
| SLA 위반 | Customer → Service Provider Manager |
| 예산·계약 이슈 | Sponsor → Account Manager |

### (4) 체인지 매니지먼트

- 신규 기능 도입 시:
  - Sponsor 설득: **비즈니스 가치, ROI**
  - Customer 설득: **운영 영향, SLA 유지**
  - User 설득: **학습 부담, 업무 변화**
- 같은 변경이라도 **대상별 메시지 다름**

### (5) 피드백 루프 구분

- User 피드백: 일상 UX, 기능 요청 (높은 빈도)
- Customer 피드백: 분기 리뷰, SLA 미팅 (중간 빈도)
- Sponsor 피드백: 연간 ROI 리뷰, 예산 회의 (낮은 빈도)

---

## 5. 한눈 정리

```
Sponsor  ── "돈을 내는 사람"      ── 예산·ROI
   │
   ▼
Customer ── "요구를 정하는 사람"   ── SLA·비즈니스 outcome
   │
   ▼
User     ── "실제로 쓰는 사람"     ── 일상 UX·체감 품질
```

---

## 한 줄 요약

**Sponsor는 예산 승인, Customer는 요구사항 정의와 SLA 체결, User는 실제 사용.** B2B에서는 3역이 분리되며 인터뷰·협상·에스컬레이션 모두 대상별로 달라진다. B2C에서는 한 사람이 3역을 겸한다.
