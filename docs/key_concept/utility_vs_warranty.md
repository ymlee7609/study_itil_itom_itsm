# Utility vs Warranty

ITIL 4 서비스 정의의 **양 축**. "유용성"과 "보증성"으로 번역되며, **둘 다 충족되어야 가치가 발생**한다. 한쪽만 있으면 서비스는 실패한다.

## 1. 공식 정의

| 구분 | Utility (유용성) | Warranty (보증성) |
|------|-----------------|-------------------|
| **정의** | 서비스가 **특정 요구를 충족**시키는 기능성 | 서비스가 **합의된 수준으로 작동**한다는 보증 |
| **슬로건** | "Fit for **Purpose**" | "Fit for **Use**" |
| **관심사** | "무엇을 할 수 있나" (What it does) | "얼마나 잘 작동하나" (How well it works) |
| **측정 대상** | 기능·성능 충족 | 가용성·용량·연속성·보안 |
| **사용자 체감** | "이 기능 좋네" | "항상 되네, 안전하네" |
| **부재 시 증상** | 기능 부족 ("안 되는 게 많네") | 가동 문제 ("되다 안 되다 함") |

---

## 2. 관계식

```
   Utility   +   Warranty   =   Value
 (유용성)       (보증성)       (가치)
 AND 조건 — 둘 중 하나라도 0이면 Value = 0
```

ITIL 4의 **가치 공식**:

> Value = f(**Utility**, **Warranty**, Cost, Risk)

- Utility ✓ + Warranty ✗ → 기능은 있지만 불안정 → 가치 없음
- Utility ✗ + Warranty ✓ → 안정적이지만 쓸모없음 → 가치 없음
- Utility ✓ + Warranty ✓ → 가치 창출

## 3. 실무 예시

### 예시 1: 이커머스 결제 시스템

| 측면 | Utility | Warranty |
|------|---------|----------|
| 내용 | 카드·계좌이체·간편결제 지원, 할부 처리 | 99.99% 가용성, 3초 이내 응답, PCI-DSS 준수 |
| 부재 시 | 카드 결제만 되면 고객 이탈 | 결제가 간헐적으로 실패 → 매출 손실 |

### 예시 2: 사내 Wi-Fi

| 측면 | Utility | Warranty |
|------|---------|----------|
| 내용 | 전 건물 커버, 게스트 네트워크 분리, VPN 지원 | 24/7 가동, 대역폭 100Mbps 보장, WPA3 암호화 |
| 부재 시 | 회의실에서 안 터짐 → 불만 | 오후만 되면 느려짐 → 생산성 저하 |

### 예시 3: 사내 메신저 (Slack/Teams)

| 측면 | Utility | Warranty |
|------|---------|----------|
| 내용 | DM, 채널, 화상회의, 파일 공유, 봇/앱 연동 | 다운타임 < 5분/월, 메시지 유실 없음, 이중화 |
| 한쪽만 있을 때 | 기능 많은데 자주 죽음 → 이탈 | 잘 안 죽는데 화상회의 없음 → 부족 |

## 4. 왜 이 구분이 중요한가

### (1) 요구사항 분해에 필수

신규 서비스 기획 시 Utility와 Warranty를 **별도 목록으로 분리**해야 누락 없음.

| 요구사항 유형 | 예 |
|--------------|-----|
| Functional (Utility) | "~ 기능 지원", "~ 처리 가능" |
| Non-functional (Warranty) | "가용성 99.9%", "응답 2초 이내", "동시 사용자 1만 명" |

### (2) SLA/XLA 설계의 토대

- **SLA는 주로 Warranty 측면** (가동률, 응답시간, 복구시간 = 모두 Warranty 지표)
- Utility는 계약 체결 전 **Scope 정의**로 보장 (무엇을 제공할 것인가)

### (3) "워터멜론 서비스" 방지

- Warranty는 녹색(SLA 달성), Utility는 빨강(기능 부족) → 사용자 불만
- 반대로 Utility 녹색, Warranty 빨강 → 역시 불만
- **양쪽 지표를 동시에 대시보드화**해야 실상 파악 가능

### (4) 아웃소싱/벤더 관리의 양날

- 벤더 계약 시 Utility 명세(기능 목록) + Warranty SLA(가동률) **둘 다 명시**
- 한쪽만 쓰면 "기능은 있지만 자주 죽는" 또는 "안 죽지만 쓸모없는" 서비스 납품됨

### (5) ITIL 5 관점

ITIL 5의 **경험 관리(XLA)**는 Warranty 개념을 **사용자 체감 경험**으로 확장:
- 전통 Warranty: "서버 가용률 99.9%"
- XLA Warranty: "사용자가 업무 중단을 느끼지 않은 시간 비율"

---

## 5. 한눈 정리

```
               Value (가치)
                    │
         ┌──────────┴──────────┐
         │                     │
     Utility               Warranty
   (Fit for Purpose)     (Fit for Use)
         │                     │
    "뭘 할 수 있나"        "얼마나 잘 되나"
    기능성/성능            가용성/연속성
    제안 단계 심사         운영 단계 측정
```

---

## 한 줄 요약

**Utility는 "무엇을 할 수 있나"(Fit for Purpose), Warranty는 "얼마나 잘 작동하나"(Fit for Use).** 곱셈 관계라서 한쪽이 0이면 가치도 0. SLA는 Warranty, Scope는 Utility를 다룬다.
