# SLA vs OLA vs UC vs XLA

서비스 계약·합의의 **4개 층위**. 실무에서는 "SLA"라는 한 단어로 다 뭉뚱그려 쓰지만, 각각 당사자·목적·측정 대상이 다르다. 특히 **XLA**는 ITIL 5에서 공식 강조되는 신개념이다.

## 1. 공식 정의

| 구분 | 풀네임 | 당사자 | 목적 |
|------|--------|--------|------|
| **SLA** | Service Level Agreement | **서비스 제공자 ↔ 고객** (대외) | 고객에게 약속한 서비스 수준 |
| **OLA** | Operational Level Agreement | **제공자 내부 팀 간** | SLA 이행을 위한 내부 팀 협약 |
| **UC** | Underpinning Contract | **제공자 ↔ 외부 공급사** | SLA 이행을 뒷받침하는 외주 계약 |
| **XLA** | Experience Level Agreement | **제공자 ↔ 고객(사용자)** | **사용자 체감 경험** 수준 약속 |

---

## 2. 계층 구조

```
        ┌─────────────────────────────────────────┐
        │        고객 (Customer/User)              │
        └────────────────▲────────────────────────┘
                         │
                    SLA / XLA (대외)
                         │
        ┌────────────────┴────────────────────────┐
        │      서비스 제공자 조직                    │
        │                                          │
        │  ┌─────────┐  OLA  ┌─────────┐          │
        │  │ 개발팀   │◄─────►│ 운영팀   │          │
        │  └─────────┘       └─────────┘          │
        │       ▲                 ▲                │
        │       │ UC              │ UC             │
        └───────┼─────────────────┼────────────────┘
                ▼                 ▼
         ┌──────────┐      ┌──────────┐
         │ AWS/벤더 │      │ 네트워크  │
         └──────────┘      │ 회선사    │
                           └──────────┘
```

**핵심 관계**: 고객 SLA/XLA는 내부 OLA와 외부 UC의 **합성 결과**다. OLA·UC가 SLA보다 약하면 SLA는 지킬 수 없다.

## 3. 실무 예시

### 예시 1: 이커머스 가용성 99.9% SLA

| 계약 | 당사자 | 내용 |
|------|--------|------|
| **SLA** | 이커머스 → 고객사 | 전체 서비스 가용성 **99.9%** (월 43분 다운 허용) |
| **OLA #1** | 운영팀 ↔ 개발팀 | 장애 에스컬레이션 15분 이내, 긴급 패치 2시간 내 제공 |
| **OLA #2** | 운영팀 ↔ DBA팀 | DB 복구 RTO 30분 |
| **UC #1** | 이커머스 ↔ AWS | EC2 SLA 99.99%, RDS 99.95% |
| **UC #2** | 이커머스 ↔ CDN사 | CDN 가용성 99.95%, 응답 < 50ms |

→ **계산 검증**: SLA 99.9%를 달성하려면 UC/OLA 합성 가용성이 이를 상회해야 함. 한 개라도 느슨하면 SLA 파기.

### 예시 2: SLA vs XLA 비교

동일 메일 시스템:

| 측정 | SLA (Warranty 기반) | XLA (Experience 기반) |
|------|--------------------|-----------------------|
| 지표 | "메일 서버 가용률 99.9%" | "직원이 메일 업무를 중단 없이 수행한 시간 비율" |
| 결과 | 99.95% ✅ | 87% ❌ (로그인 지연·검색 느림·첨부 실패 등 체감) |

→ SLA는 녹색이지만 XLA는 빨강. **"워터멜론 효과"**의 대표 사례.

## 4. 왜 이 구분이 중요한가

### (1) SLA만 쓰면 실제 서비스 품질을 놓친다

"SLA 100% 달성했는데 왜 고객은 불만일까?" → SLA가 시스템 가동만 보고 **사용자 경험**을 반영하지 않기 때문. XLA가 필요한 이유.

### (2) OLA 없이 SLA는 공수표

고객과 99.9% 약속해도 **내부 팀 간 협약이 없으면** 실제로는 구현 불가. 장애 발생 시 "우리 팀 일 아닌데요"가 나오면 끝.

### (3) UC가 SLA보다 약하면 위험

- 고객 SLA: 99.9%
- AWS UC: 99.5% (합의된 한계)
- → **구조적으로 SLA 달성 불가**. UC 강화 또는 SLA 하향 조정 필요.

### (4) XLA는 ITIL 5의 핵심 변화

ITIL 5는 **경험 관리(Experience Management)**를 공식 도입:
- 측정 대상: 시스템 → **사용자 여정**
- 지표 예: 체감 응답시간, 업무 완료율, NPS 변화, 불만 비율
- SLA를 대체가 아닌 **보완** (SLA + XLA 병행 운영)

### (5) 감사·계약 분쟁에서 증거력 차이

- SLA 위반 → 서비스 크레딧 지급 (계약 조항)
- OLA 위반 → 내부 징계·프로세스 개선
- UC 위반 → 외주사 페널티 청구
- XLA 위반 → 경험 개선 계획 수립 (통상 금전 배상 조항 없음)

## 5. 한눈 정리

```
대외 (계약)                           대내 (협약)
──────────                           ──────────

SLA                                  OLA
(Service Level Agreement)            (Operational Level Agreement)
제공자 ↔ 고객                        팀 ↔ 팀
시스템 가동·응답                      에스컬레이션, 내부 RTO
────────────                        ────────

XLA                                  UC
(Experience Level Agreement)         (Underpinning Contract)
제공자 ↔ 사용자                      제공자 ↔ 외주사
사용자 체감                          AWS/네트워크/라이선스
```

---

## 한 줄 요약

**SLA(대외)·OLA(내부 팀 간)·UC(외주사)·XLA(사용자 경험)** 4개 층위. SLA는 OLA+UC의 합성 결과이며, XLA는 SLA로 안 보이는 체감 품질을 포착한다.
