# RTO vs RPO vs MTD vs WRT

비즈니스 연속성(BCM)·재해복구(DR)의 **4대 시간 지표**. 계약·아키텍처·보험 가입 시 필수인데 실무에서는 RTO/RPO를 혼동하는 경우가 빈번하다.

## 1. 공식 정의

| 지표 | 풀네임 | 의미 | 질문 |
|------|--------|------|------|
| **RTO** | Recovery **Time** Objective | 재해 발생 후 **서비스 복구까지** 허용되는 시간 | "얼마나 빨리 다시 돌려야 하나?" |
| **RPO** | Recovery **Point** Objective | **허용 가능한 데이터 손실** 시간 | "얼마 전 데이터까지 유지해야 하나?" |
| **MTD** | **Maximum** Tolerable **Downtime** | 비즈니스가 **견딜 수 있는 최대** 중단시간 | "이보다 오래 멈추면 망함" |
| **WRT** | **Work** Recovery **Time** | 시스템 복구 후 **업무 재개까지** 걸리는 시간 | "인프라 살아나도 업무까지 돌리는 시간?" |

## 2. 시간축 위치

```
    정상 가동     재해 발생    시스템 복구    업무 정상화
       │            │            │            │
  ─────┼────────────┼────────────┼────────────┼───►
       │            │            │            │
       │◄──RPO─────►│            │            │
       │  (과거    )│            │            │
       │            │                         │
       │            │◄─── RTO ──►│            │
       │            │ (인프라)   │            │
       │            │            │◄── WRT ──►│
       │            │            │  (업무)   │
       │            │                         │
       │            │◄────────── MTD ────────►│
       │            │     (절대 한계)         │
```

### 관계식

```
MTD ≥ RTO + WRT
```

- MTD는 **비즈니스가 정한 절대 한계**
- RTO + WRT는 **실제 복구에 걸리는 시간**
- RTO + WRT > MTD → **비즈니스 파산 위험**

## 3. 실무 예시

### 예시 1: 이커머스 사이트

| 지표 | 값 | 의미 |
|------|-----|------|
| **RPO** | 5분 | 최대 5분치 주문 데이터 손실 허용 |
| **RTO** | 1시간 | 재해 후 1시간 내 시스템 복구 |
| **WRT** | 30분 | 복구 후 정산·재고 동기화 30분 |
| **MTD** | 4시간 | 4시간 초과 시 심각한 매출/평판 손실 |

→ 1시간 + 30분 = 1시간 30분 < MTD 4시간 ✓ (안전 마진 2.5시간)

### 예시 2: 병원 EMR 시스템

| 지표 | 값 |
|------|-----|
| RPO | 0분 (무손실, 동기 복제) |
| RTO | 15분 |
| WRT | 15분 |
| MTD | 1시간 (환자 안전 한계) |

→ 의료는 데이터 손실 불가, 복구 시간 극단적으로 짧음 → **고비용 투자 필요**.

### 예시 3: 월간 배치 시스템

| 지표 | 값 |
|------|-----|
| RPO | 24시간 |
| RTO | 48시간 |
| WRT | 12시간 |
| MTD | 7일 |

→ 여유 크므로 **백업 테이프 수준**도 충분.

### 예시 4: RPO/RTO 구현 수단

| 목표 | 기술 |
|------|------|
| RPO = 0 | 동기 복제 (Synchronous Replication) |
| RPO < 5분 | 비동기 복제 (Async), Log Shipping |
| RPO < 1시간 | 시간별 스냅샷 |
| RPO < 24시간 | 야간 백업 |
| RTO < 분 단위 | Active-Active 멀티리전 |
| RTO < 시간 단위 | Warm Standby, Pilot Light |
| RTO < 일 단위 | 백업에서 복원 |

## 4. 왜 이 구분이 중요한가

### (1) RTO ≠ RPO 혼동 방지

- RTO: **시간** (얼마나 빨리 살리나)
- RPO: **데이터** (얼마큼 손실 허용)
- "RPO 1시간, RTO 1시간"을 "둘 다 1시간"으로 섞어 이해 → 아키텍처 오설계

### (2) 비용과 직결

| 목표 | 비용 배수 (개략) |
|------|------------------|
| RPO 24h / RTO 24h | × 1 (테이프 백업) |
| RPO 1h / RTO 1h | × 3~5 (Warm Standby) |
| RPO 0 / RTO < 5분 | × 10~20 (Active-Active 다중 리전) |

→ 비즈니스가 **정말 RPO=0을 요구하는지** 반드시 검증. 과도 투자 방지.

### (3) MTD 초과 = 비즈니스 실패 시나리오

- MTD는 "이 이상 멈추면 회사 신뢰·매출이 회복 불능"의 한계
- RTO + WRT를 MTD **이내로** 맞춰야 함
- 맞출 수 없으면 **보험·리스크 수용** 결정 필요

### (4) WRT 누락의 실수

- 많은 조직이 "시스템 살아났으니 끝"으로 착각
- 실제로는 **데이터 정합성 확인·배치 재실행·사용자 재로그인** 등 업무 재개 시간(WRT) 필요
- MTD 계산 시 WRT를 포함하지 않으면 현실과 괴리

### (5) ITIL Service Continuity Management 프랙티스

- BIA (Business Impact Analysis) → MTD 도출
- DR Plan → RTO/RPO 정의
- DR Test → 실제 RTO/WRT 측정 검증
- 연 1회 이상 실제 테스트 필수 (서류상 목표 ≠ 실제 복구 시간)

### (6) ITIL 5 관점

- ITIL 5의 AI 거버넌스 6C 중 **Continuity**가 이 영역
- AI 시스템의 RTO/RPO는 **모델 상태·학습 데이터**까지 포함해야 함

## 5. 한눈 정리

```
      RPO ←── 재해 ──→ RTO ──→ WRT
    (과거)              │      │
  "얼마 잃어도 돼?"     │      │
    데이터             │      │
                    "시스템    "업무
                     복구"     재개"
                        
      MTD = 비즈니스가 견디는 절대 한계 (RTO + WRT ≤ MTD)
```

---

## 한 줄 요약

**RPO는 데이터 손실 허용, RTO는 시스템 복구 시간, WRT는 업무 재개 시간, MTD는 비즈니스 절대 한계.** 관계식: MTD ≥ RTO + WRT. 목표가 엄격할수록 비용이 10배씩 증가한다.
