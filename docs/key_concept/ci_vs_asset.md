# CI (Configuration Item) vs Asset

"서버 하나"를 CMDB의 CI로 보는가, ITAM의 Asset으로 보는가는 **관리 목적이 다르다**. 두 개념을 섞으면 ITSM과 ITAM이 중복되거나 누락된다.

## 1. 공식 정의

| 구분 | CI (Configuration Item) | Asset (자산) |
|------|------------------------|--------------|
| **정의** | 서비스 제공에 **기여하는 구성요소** | **재무적 가치를 갖는** IT 자원 |
| **관리 관점** | 서비스 관계, 의존성, 영향도 | 재무, 소유권, 계약, 라이프사이클 |
| **저장소** | **CMDB** (Configuration Management Database) | **Asset Register** (자산 원장) |
| **관심 질문** | "장애 시 무엇이 영향받나?" | "얼마짜리고 언제 갱신/폐기하나?" |
| **속성 예** | 의존 관계, 상태(Active/Down), 서비스 매핑 | 구매일, 비용, 감가상각, 계약서, 라이선스 |
| **담당 프랙티스** | Service Configuration Management | IT Asset Management (ITAM) |

## 2. 교집합과 차집합

```
      CI (CMDB)                    Asset (Register)
    ┌──────────┐                 ┌──────────┐
    │          │    교집합       │          │
    │   CI만   │  ┌──────┐       │ Asset만  │
    │          │  │ CI+A │       │          │
    │          │  └──────┘       │          │
    └──────────┘                 └──────────┘
```

### CI만 (재무 가치 없음, 서비스엔 기여)
- 서비스 정의 그 자체 ("메일 서비스")
- 논리적 구성요소 (VIP, DNS 레코드, API 엔드포인트)
- 프로세스 문서 (ITIL 문서는 CI, Asset 아님)

### Asset만 (재무 가치 있음, 서비스엔 미기여)
- 미사용 재고 노트북 (창고 보관)
- 계약서, 라이선스 증서 (미배포)
- 폐기 예정 자산 (아직 제거 전)

### 교집합 (대부분의 실물)
- 운영 중인 서버, 네트워크 장비
- 설치된 소프트웨어 라이선스
- 업무용 배포 노트북

## 3. 실무 예시

### 예시 1: 서버 한 대의 두 얼굴

| 관점 | 기록 내용 |
|------|----------|
| **CI (CMDB)** | host: prod-db-01 / 역할: MySQL Primary / 의존 서비스: CRM, 주문 / 상태: Active |
| **Asset (ITAM)** | 제조사: Dell / 모델: R750 / 구매가: 2,400만원 / 구매일: 2024-03-15 / 보증: 2027-03 / 담당자: 인프라팀 |

→ 같은 물리 서버지만 **CMDB는 "무엇에 쓰이나"**, **ITAM은 "얼마짜리고 언제 바꾸나"**에 관심.

### 예시 2: 소프트웨어 라이선스

| 관점 | 기록 |
|------|------|
| **CI** | MSSQL Server 2022 인스턴스 / 설치 호스트 / 버전 / 패치 레벨 |
| **Asset** | MSSQL 라이선스 계약 50 core, 연 1.2억원, 2027년 갱신 예정 |

장애 시엔 CMDB 조회 (어느 서비스가 죽나), 비용 감사 시엔 Asset 조회 (얼마 썼나, 준수 여부).

### 예시 3: 창고 노트북

| 관점 | 기록 |
|------|------|
| **CI** | 등록 안 됨 (아직 서비스 제공에 기여 안 함) |
| **Asset** | 재고 15대, 평균가 180만원, 2023년 구매분 |

→ 배포되는 순간 CI로도 등록됨.

## 4. 왜 이 구분이 중요한가

### (1) 데이터 소유권·업데이트 주기 차이

| 항목 | CMDB | Asset Register |
|------|------|---------------|
| 업데이트 주기 | **실시간** (Discovery 도구 자동) | 정기 (구매/폐기 시점) |
| 소유 | IT Ops / Service Management | 재무 / 구매 / ITAM팀 |
| 목적 | 장애 영향 분석, 변경 영향도 | 감가상각, 예산, 라이선스 준수 |

### (2) 잘못된 통합의 폐해

- CMDB에 재무 정보를 우겨넣으면: 재무팀은 안 보고, 운영팀은 더러움
- Asset Register로 장애 분석: 의존관계 없어 영향도 산출 불가
- **도구 통합 ≠ 개념 통합**. ServiceNow처럼 한 플랫폼에 담아도 **테이블·프로세스는 분리**

### (3) 클라우드 시대의 경계 이동

- 전통: 서버 1대 = CI 1개 = Asset 1개 (명확)
- 클라우드: Lambda 함수 = CI ○ / Asset ✗ (사용량 기반, 자산화 안 됨)
- SaaS: Salesforce 조직 = CI ○ / Asset ○ (월 구독도 Asset으로 관리)
- → **"물리적 실체"가 없어도 CI와 Asset은 각자 정의됨**

### (4) 감사 대응

- **라이선스 감사** (MS, Oracle 등): Asset Register 증거 필요
- **변경 영향 감사**: CMDB 증거 필요
- 둘을 합쳐놓으면 감사 범위도 모호해짐

### (5) ITIL vs ITAM 조직 역할 분리

| 역할 | 관심 |
|------|------|
| Configuration Manager | 서비스 지도, 의존성 그래프, CI 품질 |
| IT Asset Manager | TCO, 라이선스 준수, 폐기·갱신 계획 |

## 5. 한눈 정리

```
   CI                              Asset
(CMDB 기록)                   (Asset Register 기록)
──────────                    ──────────
"서비스에 뭘 기여하나"         "얼마짜리고 언제 바꾸나"
운영/변경/장애 관점            재무/계약/생애주기 관점
실시간 갱신                    정기 갱신
Configuration Management      IT Asset Management
```

---

## 한 줄 요약

**CI는 "서비스 관점"(CMDB), Asset은 "재무 관점"(Asset Register).** 같은 서버라도 목적이 다르며, 대부분 교집합이지만 순수 CI(논리 구성)와 순수 Asset(재고·계약)도 존재한다.
