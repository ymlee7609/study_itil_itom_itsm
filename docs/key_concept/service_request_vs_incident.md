# Service Request vs Incident

둘 다 "티켓"이라서 묶어 버리기 쉬운 용어. 그러나 **정상 업무의 요청**과 **비정상 중단의 대응**은 처리 경로·SLA·자동화 수준이 완전히 다르다.

## 1. 공식 정의

| 구분 | Service Request (서비스 요청) | Incident (인시던트) |
|------|------------------------------|---------------------|
| **정의** | 정상 서비스 제공의 **일부로서의 요청** (신청/조회/변경) | **계획되지 않은 중단** 또는 품질 저하 |
| **상태** | 서비스가 **정상 작동 중** | 서비스가 **비정상 또는 중단** |
| **담당 프랙티스** | Service Request Management | Incident Management |
| **목표** | 요청을 **효율적으로 이행** | 서비스를 **신속 복구** |
| **자동화 수준** | **매우 높음** (셀프서비스, 카탈로그) | 제한적 (긴급 대응 필요) |
| **SLA 유형** | 처리 소요시간 (Request Fulfillment Time) | 복구 시간 (MTTR) |
| **예** | "계정 생성", "VPN 접속 권한", "노트북 교체 신청" | "이메일 안 됨", "결제 실패", "사이트 다운" |

## 2. 판단 기준

```
          사용자 티켓 접수
                │
                ▼
    ┌──────────────────────────┐
    │ 서비스가 정상 작동 중인가? │
    └────┬─────────────────┬───┘
         │ 예              │ 아니오
         ▼                 ▼
  Service Request      Incident
  (신청/변경/조회)     (장애/오류)
         │                 │
         ▼                 ▼
   카탈로그에서         복구 절차에
   미리 정의된          따라 대응
   프로세스로 처리      (MTTR 목표)
```

**핵심 질문**: "**서비스가 동작하지 않아서** 사용자가 티켓을 냈나, **필요한 뭔가를 받고 싶어서** 냈나?"

## 3. 실무 예시

### 예시 1: 계정 관련

| 티켓 내용 | 구분 | 이유 |
|----------|------|------|
| "신규 입사자 계정 생성 요청" | **Service Request** | 정상 프로세스, 사전 정의된 이행 |
| "내 계정 로그인이 안 됩니다" | **Incident** | 기존 서비스의 비정상 상태 |
| "비밀번호 재설정 요청" | **Service Request** | 셀프서비스로 자동 처리 가능 |
| "비밀번호 재설정 후에도 로그인 불가" | **Incident** | 재설정 프로세스 장애 |

### 예시 2: 장비 관련

| 티켓 내용 | 구분 |
|----------|------|
| "추가 모니터 신청" | Service Request |
| "모니터 화면이 꺼졌다 켜졌다 함" | Incident |
| "노트북 신규 배포 신청" | Service Request |
| "노트북이 부팅 안 됨" | Incident |

### 예시 3: 소프트웨어

| 티켓 내용 | 구분 |
|----------|------|
| "Adobe 라이선스 추가 신청" | Service Request |
| "설치된 Adobe가 실행되다 튕김" | Incident |
| "신규 SaaS 도입 요청 (Notion 등)" | Service Request (또는 Change) |
| "기존 Slack에서 메시지 전송 실패" | Incident |

### 예시 4: 경계 사례

| 티켓 | 판단 |
|------|------|
| "VPN 접속 권한 없어서 접속 안 됨" | **Service Request** (권한 없음 = 정상, 권한 신청 프로세스 필요) |
| "VPN 권한은 있는데 접속 실패" | **Incident** (기능 장애) |

## 4. 왜 이 구분이 중요한가

### (1) 처리 경로가 완전히 다르다

| 항목 | Service Request | Incident |
|------|----------------|----------|
| 접수 | 서비스 카탈로그 (사전 정의) | 자유 서술 |
| 분류 | 카테고리 사전 선택 | 우선순위 판단 필요 |
| 처리 | Workflow 자동화 (승인 → 이행) | 조사 → 복구 → 검증 |
| 종결 | 이행 확인 | 서비스 정상 작동 확인 |

### (2) SLA 성격이 다르다

- Service Request SLA: "계정 생성 1영업일 이내" (**처리 시간**)
- Incident SLA: "Critical Incident 4시간 내 복구" (**복구 시간**)
- 한 티켓 테이블로 관리하면 SLA 리포팅이 왜곡

### (3) 자동화 ROI가 다르다

- Service Request: **자동화 최적 대상** (반복적, 정형화)
  - 예: 비밀번호 재설정 100% 자동 → 티켓 90% 감소
- Incident: 자동화 어려움 (비정형, 진단 필요)

### (4) KPI가 다르다

| 프로세스 | 핵심 KPI |
|---------|---------|
| Service Request | Fulfillment Time, 자동화율, 사용자 만족도, Self-Service 비율 |
| Incident | MTTR, 1차 해결율(FCR), SLA 준수율, 재발률 |

### (5) 조직 구조 영향

- Service Request 집중: L1 또는 셀프서비스가 대부분 처리 → L1 인력 절약
- Incident 집중: L2/L3 전문 인력 투입 → 기술 조직 확장
- **비율 측정**이 조직 구성 방향성 결정

### (6) ITIL v3 vs v4

- v3: Incident Management의 **하위**로 Service Request 취급
- v4: Service Request Management를 **독립 Practice**로 승격
- → v4 기준으로 명확히 분리 운영 권장

## 5. 한눈 정리

```
정상 서비스 ─── Service Request ─── 카탈로그 / 셀프서비스 / 자동화
(일상 운영)    "필요한 걸 주세요"    Fulfillment Time SLA

                    ─── 경계 ───

비정상 상태 ─── Incident         ─── 조사 / 복구 / 검증
(장애 발생)    "안 됩니다"        MTTR SLA
```

---

## 한 줄 요약

**Service Request는 정상 작동 중인 서비스에 대한 신청·변경·조회, Incident는 비정상 중단·저하에 대한 복구 요청.** 둘을 분리해야 자동화·SLA·조직 설계가 올바르게 작동한다.
