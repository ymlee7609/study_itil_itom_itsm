## 1. 인시던트 관리 (Incident Management)

**목적**: 서비스 중단이나 품질 저하를 가능한 빨리 정상 상태로 복구한다.

### 워크플로우

| 단계 | 활동 | 담당 |
|------|------|------|
| 1. 식별 (Identification) | 사용자 보고, 모니터링 알림, 자동 감지 | 서비스 데스크 / 모니터링 도구 |
| 2. 기록 (Logging) | 티켓 생성, 상세 정보 기록 | 서비스 데스크 분석가 |
| 3. 분류 (Categorization) | 유형 분류 (네트워크, 애플리케이션, 하드웨어 등) | 서비스 데스크 분석가 |
| 4. 우선순위 지정 (Prioritization) | 영향도(Impact) x 긴급도(Urgency) = 우선순위 | 서비스 데스크 분석가 |
| 5. 진단 (Diagnosis) | 원인 파악, 지식베이스 검색 | L1/L2 지원팀 |
| 6. 해결 (Resolution) | 해결책 적용, 서비스 복구 | L2/L3 지원팀 |
| 7. 종료 (Closure) | 사용자 확인, 티켓 종료, 만족도 조사 | 서비스 데스크 분석가 |

### 우선순위 매트릭스

| 영향도 \ 긴급도 | 높음 (High) | 중간 (Medium) | 낮음 (Low) |
|-----------------|-------------|---------------|------------|
| **높음 (High)** | P1 (Critical) | P2 (High) | P3 (Medium) |
| **중간 (Medium)** | P2 (High) | P3 (Medium) | P4 (Low) |
| **낮음 (Low)** | P3 (Medium) | P4 (Low) | P5 (Planning) |

### 에스컬레이션 체계

| 유형 | 설명 | 트리거 |
|------|------|--------|
| 기능적 에스컬레이션 (Functional) | L1 → L2 → L3 기술 수준 상향 | 해결 역량 부족 시 |
| 계층적 에스컬레이션 (Hierarchical) | 관리자/경영진에게 보고 | SLA 위반 임박, 비즈니스 영향 큼 |

## 2. 문제 관리 (Problem Management)

**목적**: 인시던트의 **근본 원인(Root Cause)**을 파악하고, 재발을 방지한다.

### 워크플로우

| 단계 | 활동 | 산출물 |
|------|------|--------|
| 1. 문제 식별 (Identification) | 반복 인시던트 분석, 트렌드 파악 | 문제 레코드(Problem Record) |
| 2. 근본 원인 분석 (Root Cause Analysis) | 5 Why, Ishikawa, Fault Tree 등 기법 활용 | 근본 원인 보고서 |
| 3. 알려진 오류 등록 (Known Error) | 원인은 파악했으나 해결 전 상태 기록 | Known Error Database (KEDB) |
| 4. 해결 (Resolution) | 영구적 해결책 구현 또는 변경 요청 생성 | 변경 요청(RFC) |
| 5. 종료 (Closure) | 해결 검증, 문서화, 지식베이스 업데이트 | 업데이트된 Knowledge Base |

### 인시던트 관리 vs 문제 관리

| 구분 | 인시던트 관리 | 문제 관리 |
|------|-------------|-----------|
| 목적 | 빠른 서비스 복구 | 근본 원인 제거, 재발 방지 |
| 시간 관점 | 단기 (Reactive) | 중장기 (Proactive) |
| 해결 방식 | 우회 방법(Workaround) 포함 | 영구적 해결(Permanent Fix) |
| 트리거 | 사용자 보고, 알림 | 트렌드 분석, 반복 인시던트 |
| KPI | MTTR, FCR | 반복 인시던트 감소율, KEDB 건수 |

## 3. 변경 활성화 (Change Enablement)

**목적**: IT 서비스 변경의 **리스크를 최소화**하면서 효율적으로 변경을 수행한다.

### 워크플로우

| 단계 | 활동 | 담당 |
|------|------|------|
| 1. 변경 요청 (RFC) | 변경 사유, 범위, 일정 기술 | 변경 요청자 |
| 2. 리스크 평가 (Risk Assessment) | 영향 분석, 롤백 계획 수립 | 변경 관리자 |
| 3. 승인 (Authorization) | CAB 검토 또는 사전 승인 확인 | CAB / 변경 권한자 |
| 4. 구현 (Implementation) | 변경 계획에 따른 실행 | 기술팀 |
| 5. 검토 (Review) | 성공 여부 확인, 교훈 기록 | 변경 관리자 |

### 변경 유형

| 유형 | 설명 | 승인 방식 | 예시 |
|------|------|-----------|------|
| 표준 변경 (Standard Change) | 사전 승인된 낮은 리스크 변경 | 사전 승인 (Pre-authorized) | 패치 적용, 사용자 계정 생성 |
| 일반 변경 (Normal Change) | CAB 검토가 필요한 변경 | CAB 승인 | 서버 마이그레이션, 신규 앱 배포 |
| 긴급 변경 (Emergency Change) | 즉시 처리해야 하는 변경 | ECAB 긴급 승인 | P1 인시던트 대응, 보안 패치 |

## 4. 서비스 요청 관리 (Service Request Management)

**목적**: 사용자의 **사전 정의된 서비스 요청**을 효율적으로 처리한다.

### 워크플로우

| 단계 | 활동 | 담당 |
|------|------|------|
| 1. 요청 접수 (Request Submission) | 셀프서비스 포털, 이메일, 전화 | 사용자 |
| 2. 승인 (Approval) | 관리자 승인 (필요 시) | 승인권자 |
| 3. 이행 (Fulfillment) | 요청 사항 처리 및 실행 | 이행팀 |
| 4. 완료 (Completion) | 사용자 확인, 만족도 조사 | 서비스 데스크 |

### 일반적인 서비스 요청 유형

| 카테고리 | 요청 예시 | 일반적 SLA |
|----------|-----------|-----------|
| 계정/접근 | 계정 생성, 비밀번호 초기화, 권한 변경 | 4시간 ~ 1영업일 |
| 하드웨어 | 노트북 요청, 모니터 추가, 주변기기 | 3~5영업일 |
| 소프트웨어 | SW 설치, 라이선스 요청, 업그레이드 | 1~3영업일 |
| 정보 요청 | 절차 문의, 상태 확인, 보고서 요청 | 1영업일 이내 |

## 5. 서비스 수준 관리 (Service Level Management)

**목적**: **SLA(Service Level Agreement)**를 정의, 모니터링, 보고하여 합의된 서비스 수준을 보장한다.

### 워크플로우

| 단계 | 활동 | 산출물 |
|------|------|--------|
| 1. SLA 정의 (Define) | 서비스 목표, 측정 지표, 범위 합의 | SLA 문서 |
| 2. 모니터링 (Monitor) | 실시간 SLA 준수율 추적 | 대시보드 |
| 3. 보고 (Report) | 정기 보고서 생성 및 공유 | SLA 보고서 |
| 4. 검토 (Review) | 정기 서비스 검토 회의, SLA 조정 | 개선 계획 |

### SLA 계층 구조

| 계층 | 영문 | 설명 | 관계 |
|------|------|------|------|
| SLA | Service Level Agreement | 서비스 제공자 ↔ 고객 간 합의 | 외부/내부 고객 대상 |
| OLA | Operational Level Agreement | 내부 지원 그룹 간 합의 | SLA 이행을 위한 내부 약속 |
| UC | Underpinning Contract | 외부 공급업체와의 계약 | SLA 이행을 위한 외부 계약 |

## 6. 지식 관리 (Knowledge Management)

**목적**: 조직의 지식을 체계적으로 **생성, 공유, 활용, 유지**하여 서비스 품질을 향상시킨다.

### 워크플로우 (KCS - Knowledge-Centered Service)

| 단계 | 활동 | 담당 |
|------|------|------|
| 1. 생성 (Capture) | 인시던트 해결 과정에서 지식 문서 작성 | 지원 분석가 |
| 2. 구조화 (Structure) | 템플릿에 맞춰 체계적으로 정리 | 지식 관리자 |
| 3. 재사용 (Reuse) | 유사 인시던트 발생 시 지식 검색/활용 | 모든 지원 인력 |
| 4. 개선 (Improve) | 피드백 반영, 최신 정보로 업데이트 | 지식 관리자 |

### 지식 문서 유형

| 유형 | 목적 | 대상 |
|------|------|------|
| FAQ (자주 묻는 질문) | 빈번한 문의 사전 답변 | 최종 사용자 |
| How-to 가이드 | 절차적 안내 | 사용자/지원 인력 |
| Known Error 문서 | 알려진 오류와 해결책 기록 | 지원 인력 |
| 트러블슈팅 가이드 | 진단 절차 상세 기술 | L2/L3 지원팀 |

## 7. IT 자산 관리 (IT Asset Management)

**목적**: IT 자산의 **전체 수명주기**를 관리하여 비용을 최적화하고 규정을 준수한다.

### 자산 수명주기

| 단계 | 활동 | 주요 고려사항 |
|------|------|---------------|
| 1. 계획 (Plan) | 수요 예측, 예산 수립 | TCO(Total Cost of Ownership) 분석 |
| 2. 조달 (Procure) | 구매/리스, 계약 관리 | 벤더 비교, 볼륨 라이선스 |
| 3. 배포 (Deploy) | 설치, 구성, 사용자 할당 | 자산 태깅, CMDB 등록 |
| 4. 운영 (Operate) | 유지보수, 모니터링, 업그레이드 | 성능 최적화, 보안 패치 |
| 5. 폐기 (Retire) | 데이터 삭제, 환경 안전 폐기 | 데이터 보안, 환경 규정 |

### 라이선스 관리

| 라이선스 유형 | 설명 | 관리 포인트 |
|---------------|------|-------------|
| 영구 라이선스 (Perpetual) | 일회성 구매, 영구 사용 | 감가상각 관리 |
| 구독 라이선스 (Subscription) | 월/연 단위 사용료 | 갱신 관리, 사용량 최적화 |
| 사용자 기반 (Per User) | 사용자 수 기반 과금 | 퇴직자/미사용 라이선스 회수 |
| 디바이스 기반 (Per Device) | 디바이스 수 기반 과금 | 하드웨어 폐기 시 연동 |

## 8. 구성 관리 (Configuration Management)

**목적**: **CMDB(Configuration Management Database)**를 통해 IT 인프라 구성요소와 관계를 관리한다.

### CMDB 핵심 개념

| 용어 | 영문 | 설명 |
|------|------|------|
| 구성 항목 | Configuration Item (CI) | 관리 대상이 되는 IT 구성요소 |
| CI 속성 | CI Attributes | CI의 특성 정보 (이름, 버전, 위치 등) |
| 관계 | Relationships | CI 간의 의존성, 연결 관계 |
| 기준선 | Baseline | 특정 시점의 승인된 CI 구성 상태 |

### CI 유형

| CI 유형 | 예시 | 주요 관리 속성 |
|---------|------|----------------|
| 하드웨어 (Hardware) | 서버, 네트워크 장비, 스토리지 | 위치, 사양, 보증 기간 |
| 소프트웨어 (Software) | OS, 미들웨어, 애플리케이션 | 버전, 라이선스, 패치 수준 |
| 네트워크 (Network) | 라우터, 스위치, 방화벽 | IP, VLAN, 설정 파일 |
| 문서 (Document) | SLA, 운영 절차서, 구성도 | 버전, 승인 상태, 담당자 |
| 서비스 (Service) | 이메일 서비스, ERP | 의존 CI, SLA, 사용자 수 |

### 관계 매핑

| 관계 유형 | 설명 | 예시 |
|-----------|------|------|
| 의존 (Depends on) | A가 B에 의존 | 애플리케이션 → 데이터베이스 |
| 설치 (Installed on) | A가 B에 설치됨 | 미들웨어 → 서버 |
| 연결 (Connected to) | A와 B가 네트워크 연결 | 서버 ↔ 스위치 |
| 사용 (Used by) | A가 B에 의해 사용됨 | 서비스 → 사용자 그룹 |

## 9. 서비스 데스크 (Service Desk)

**목적**: IT 서비스의 **단일 접점(SPOC: Single Point of Contact)**으로서 사용자와 IT 간의 소통을 관리한다.

### 서비스 데스크 유형

| 유형 | 설명 | 적합 조직 |
|------|------|-----------|
| 로컬 (Local) | 물리적으로 사용자 근처에 위치 | 단일 사업장, 소규모 조직 |
| 중앙 집중 (Centralized) | 하나의 장소에서 모든 요청 처리 | 비용 효율 중시 조직 |
| 가상 (Virtual) | 분산된 인력이 중앙 시스템으로 통합 | 다국적/원격 근무 조직 |
| Follow-the-Sun | 시간대별 교대 운영 (24/7) | 글로벌 기업 |

### 채널별 특성

| 채널 | 장점 | 단점 | 적합한 상황 |
|------|------|------|-------------|
| 전화 (Phone) | 즉각 소통, 복잡한 문제 | 비용 높음, 대기 시간 | 긴급 인시던트, 고령 사용자 |
| 이메일 (Email) | 기록 보존, 비동기 처리 | 응답 지연 가능 | 비긴급 요청, 상세 설명 필요 |
| 셀프서비스 포털 (Portal) | 24/7 이용, 비용 효율 | 복잡한 문제 부적합 | 표준 요청, 정보 조회 |
| 챗봇 (Chatbot) | 자동화, 즉각 응답, 확장성 | 복잡한 문의 한계 | FAQ, 비밀번호 초기화 |
| 워크인 (Walk-in) | 직접 대면, 하드웨어 문제 | 인력/공간 필요 | 하드웨어 교체, VIP 지원 |
