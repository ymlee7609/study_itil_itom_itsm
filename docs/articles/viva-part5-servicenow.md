ITSM/ITOM/ITAM 시리즈 Part 5 (마지막) — ServiceNow: 이 모든 걸 하나로 묶는 플랫폼

Part 1에서 ITSM, ITOM, ITAM의 큰 그림을 레스토랑 비유로 설명했고, Part 2~4에서 각 영역을 하나씩 깊이 들여다봤습니다. 이번 Part 5에서는 이 세 영역을 **하나의 플랫폼 위에서 연결하는 방법**과, 그 대표적인 플랫폼인 ServiceNow를 중심으로 정리합니다.


**도구가 쪼개져 있으면 벌어지는 일**

ITSM은 A 시스템, 모니터링은 B 시스템, 자산 관리는 엑셀 — 이런 조직이 상당히 많습니다.

- **수동 연결** — 모니터링 알림을 보고 별도 시스템에 티켓을 직접 만들어야 합니다. 장애 하나에 화면 세 개를 왔다 갔다 하게 됩니다.
- **데이터 단절** — "이 서버가 죽으면 어떤 서비스가 영향받는지" 한눈에 파악이 안 됩니다. 관계 정보가 시스템마다 다르게 관리되거든요.
- **사후 분석 어려움** — 로그는 여기, 티켓은 저기, 자산 이력은 또 저기 — 데이터를 끌어모으는 것 자체가 일이 됩니다.
- **사람이 접착제** — 자동화할 수 있는 연결을 사람이 수동으로 하고 있습니다. 야간이나 주말에 특히 문제가 되죠.

이 문제를 해결하는 접근이 **단일 플랫폼 전략**입니다. ITSM, ITOM, ITAM을 같은 데이터 모델 위에서 돌리는 겁니다.


**ServiceNow 플랫폼 구조**

ServiceNow는 ITSM, ITOM, ITAM을 하나의 플랫폼에서 제공합니다. 그 중심에 **CMDB**가 있습니다.

플랫폼 구조를 설명하면, ServiceNow라는 하나의 플랫폼 위에 ITSM, ITOM, ITAM 세 개 모듈이 올라가 있고, 이 모듈들이 모두 중앙의 CMDB(통합 DB)를 공유하는 형태입니다.

- **ITSM 모듈** — 인시던트 관리, 문제 관리, 변경 관리, 서비스 요청을 처리합니다.
- **ITOM 모듈** — Discovery, Event Management, Service Map, AIOps를 제공합니다.
- **ITAM 모듈** — 하드웨어 자산, 소프트웨어 자산, 라이선스 관리, 생애주기 추적을 담당합니다.
- **CMDB** — 세 모듈이 공통으로 참조하는 통합 데이터베이스입니다. 모든 CI(구성 항목) 데이터가 여기에 있습니다.


**ITOM 모듈 상세**

ServiceNow ITOM은 세 가지 핵심 영역으로 구성됩니다.

- **ITOM Visibility** — Discovery로 네트워크를 스캔해서 서버, 스위치, 클라우드 리소스를 자동으로 찾아내고 CMDB에 등록합니다. Service Mapping은 서비스-인프라 관계를 자동으로 그려줍니다. "우리 환경에 뭐가 있고, 뭐가 뭐에 연결되어 있는지" 파악하는 기반이 됩니다.
- **ITOM Health** — Event Management가 각종 모니터링 도구의 알림을 수집, 중복 제거, 상관분석합니다. AIOps가 노이즈를 줄이고 근본 원인을 추론합니다. 알림 폭풍 속에서 진짜 문제를 찾아내고, 자동으로 인시던트를 만들어줍니다.
- **ITOM Optimization** — 클라우드 비용 관리, 자동 프로비저닝, 거버넌스를 제공합니다. 멀티클라우드 환경에서 비용과 정책을 통제하는 역할입니다.


**모듈 간 연결 흐름**

단일 플랫폼의 핵심 가치는 **모듈 간 자동 연결**입니다.

- **ITOM에서 ITSM으로** — 이상 징후 감지 시 인시던트가 자동 생성되고, CMDB 기반 영향도가 자동 첨부됩니다. 도구가 쪼개져 있었다면 모니터링 화면 보고 수동으로 티켓을 생성하고 영향 범위를 수동으로 파악해야 했을 겁니다.
- **ITSM에서 ITAM으로** — 티켓 내에서 해당 자산의 보증, 라이선스, 교체 주기를 바로 조회할 수 있습니다. 쪼개져 있었다면 별도 시스템이나 엑셀에서 자산 정보를 검색해야 했겠죠.
- **ITOM에서 ITAM으로** — Discovery가 발견한 자산을 CMDB에 자동 등록하고, ITAM이 생애주기를 추적합니다. 쪼개져 있었다면 자산 대장을 수동으로 업데이트해야 합니다.
- **CMDB(허브)** — 모든 모듈이 같은 CI(구성 항목) 데이터를 참조합니다. 쪼개져 있었다면 시스템마다 자산 정보가 다르게 관리되는 상황이 발생합니다.


**Now Assist (생성형 AI)**

ServiceNow는 2024년부터 Now Assist라는 생성형 AI 기능을 플랫폼 전반에 통합하고 있습니다.

- **인시던트 요약** — 장애 컨텍스트를 자동으로 요약해서 담당자에게 전달합니다.
- **자연어 검색** — "지난 1시간 동안 지연이 높은 서비스는?" 같은 질문에 답변합니다.
- **런북 생성** — 장애 대응 절차를 자동으로 작성합니다.
- **가상 에이전트** — 직원의 IT 요청을 AI가 1차 대응합니다.


**주요 플랫폼 비교**

ServiceNow만 이런 통합 플랫폼을 제공하는 건 아닙니다. 주요 대안과 비교하면 이렇습니다.

- **ServiceNow**
  - 대상 규모: 대기업 (5,000명 이상)
  - 핵심 강점: 단일 플랫폼, ITSM+ITOM+ITAM 통합, 강력한 자동화
  - ITIL 정렬: 19개 프랙티스 인증 (업계 최고)
  - ITOM 통합: 네이티브 (Discovery, Event Mgmt, AIOps)
  - ITAM 통합: 네이티브 (같은 플랫폼)
  - 배포 방식: SaaS 전용 (클라우드)
  - AI 기능: Now Assist (생성형 AI)
  - 구현 기간: 6~18개월
  - 비용 수준: 높음
  - 약점: 높은 TCO, 긴 구현, 전문 인력 필요

- **Jira Service Management**
  - 대상 규모: 중견기업 / DevOps 조직
  - 핵심 강점: Atlassian 생태계 통합, DevOps 친화적, 합리적 가격
  - ITIL 정렬: 주요 프랙티스 지원, DevOps/Agile 통합에 강점
  - ITOM 통합: Opsgenie 통합 수준 (별도 제품)
  - ITAM 통합: 기본 자산 관리 (고급 기능 제한)
  - 배포 방식: 클라우드 + Data Center
  - AI 기능: 기본 자동화
  - 구현 기간: 2~8주
  - 비용 수준: 합리적
  - 약점: 대규모 ITIL 프로세스 약함, ITOM 제한적

- **BMC Helix**
  - 대상 규모: 대기업
  - 핵심 강점: ITIL 프로세스 깊이, 온프레미스 배포 가능
  - ITIL 정렬: 20년 이상 ITIL 경험, 깊은 프로세스 지원
  - ITOM 통합: BMC Helix Operations Management
  - ITAM 통합: 별도 모듈 연동
  - 배포 방식: SaaS + 온프레미스 + 하이브리드
  - AI 기능: Helix AI
  - 구현 기간: 6~12개월
  - 비용 수준: 높음
  - 약점: 레거시 UI, 학습 곡선


**어떤 조직에 어떤 플랫폼이 맞을까**

- 직원 5,000명 이상이고 복잡한 인프라를 운영한다면 **ServiceNow**가 맞습니다. 확장성과 ITSM+ITOM+ITAM 네이티브 통합이 강점이거든요.
- DevOps 문화가 강하고 Atlassian 도구를 이미 사용 중이라면 **Jira Service Management**가 맞습니다. 개발-운영 연계가 좋고 비용도 합리적입니다.
- 온프레미스 배포가 필수인 환경(규제, 보안)이라면 **BMC Helix**가 맞습니다. 하이브리드/온프레미스 배포를 지원하거든요.
- 중소기업이고 빠른 도입이 필요하다면 **Freshservice**도 고려할 만합니다. 2~4주 도입이 가능하고 직관적인 UX와 AI 기능을 제공합니다.


**도입할 때 알아야 할 것들**

ServiceNow의 장점은 분명하지만, 솔직하게 짚어야 할 부분도 있습니다.

**현실적 고려사항**

- **비용** — 라이선스 + 구현 컨설팅 + 커스터마이징 + 교육까지 포함하면 TCO가 상당합니다. 3년 기준으로 상당한 규모의 투자가 필요합니다.
- **구현 기간** — 단순 도입은 몇 달이지만, 프로세스 정리 + 데이터 마이그레이션 + 조직 적응까지 포함하면 6개월~1년 이상 걸리는 경우가 흔합니다.
- **전문 인력** — ServiceNow 관리자/개발자 확보가 도입 이후에도 지속적 과제입니다. 시장에서 인력 수요가 높아 확보 경쟁이 치열합니다.
- **프로세스 먼저** — 프로세스가 정리되지 않은 상태에서 도구만 바꾸면, 비싼 도구 위에서 비효율적인 프로세스를 돌리는 꼴입니다. ITIL 같은 프레임워크로 체계를 먼저 잡는 게 순서입니다.
- **변화 관리** — 새 시스템 도입은 기술 프로젝트가 아니라 조직 변화 프로젝트입니다. 현업 참여와 교육이 성공의 핵심 요소입니다.

**도입 평가 시 체크리스트**

- **기능 적합성** — 현재와 향후 ITSM/ITOM/ITAM 프로세스를 지원하는가
- **통합 역량** — 기존 모니터링, CMDB, AD 등과 연동 가능한가
- **확장성** — 조직 성장에 따라 확장할 수 있는가
- **TCO** — 라이선스 + 구현 + 운영 + 교육 비용의 총합은 얼마인가
- **벤더 지원** — 한국어 지원과 현지 파트너가 있는가
- **보안/컴플라이언스** — SOC 2, ISO 27001, 데이터 주권 요구를 충족하는가


**시리즈 핵심 정리**

Part 1부터 5까지의 내용을 네 가지로 압축하면 이렇습니다.

- **연결이 핵심이다** — ITSM, ITOM, ITAM은 따로 돌면 의미가 반감됩니다. 감지에서 처리, 자산 판단까지의 흐름이 끊기지 않아야 합니다.
- **CMDB가 허브다** — 자산의 기술적 관계와 비즈니스 영향을 파악하는 단일 진실의 원천입니다. CMDB가 부실하면 플랫폼을 도입해도 효과가 반감됩니다.
- **도구보다 프로세스가 먼저다** — 어떤 플랫폼을 쓰든, "인시던트를 어떻게 처리하고, 변경을 어떻게 관리하고, 자산을 어떻게 추적할 것인지" 체계가 먼저입니다.
- **완벽한 도구는 없다** — ServiceNow든 Jira SM이든 BMC든, 조직의 규모와 상황에 맞는 선택이 최선입니다.


**약어 정리**

- **CMDB** — Configuration Management Database: IT 구성요소와 관계를 관리하는 통합 DB
- **CI** — Configuration Item: CMDB에서 관리하는 개별 구성 항목
- **TCO** — Total Cost of Ownership: 총소유비용
- **AIOps** — AI for IT Operations: AI/ML 기반 IT 운영 자동화
- **SaaS** — Software as a Service: 클라우드 기반 소프트웨어 서비스
- **NMS** — Network Management System: 네트워크 관리 시스템
- **AD** — Active Directory: 사용자/장치 인증 및 디렉터리 서비스
