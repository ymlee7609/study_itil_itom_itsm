Part 1에서 ITSM을 "레스토랑의 홀 서비스"에 비유했습니다. 이번 글에서는 그 홀 서비스가 실제로 어떤 일을 하는지 좀 더 깊이 들여다봅니다.

- 서비스 데스크의 역할

**서비스 데스크 — 모든 것의 시작점**

ITSM에서 가장 먼저 만나는 것이 **서비스 데스크(Service Desk)**입니다.

레스토랑으로 치면 "안내 데스크"에 해당합니다. 손님이 뭔가 필요하면 — 예약이든, 불만이든, 메뉴 문의든 — 일단 여기서 접수합니다.

IT에서도 같습니다. 노트북이 고장 났든, VPN이 안 되든, 새 소프트웨어가 필요하든, 모든 건 서비스 데스크를 통해 들어옵니다. 이걸 **단일 접점(SPOC, Single Point of Contact)**이라고 부릅니다.

왜 중요하냐면, 창구가 하나로 통일되어야 "어디에 연락해야 하지?"라는 고민이 사라지거든요. 동시에 IT 쪽에서도 모든 요청이 한곳에 모이니까 우선순위를 정하고, 빠뜨리는 건 없는지 확인할 수 있습니다.

**서비스 데스크 유형**

- **로컬(Local)** — 사용자 근처에 물리적으로 위치합니다. 단일 사업장에 적합합니다.
- **중앙 집중(Centralized)** — 하나의 장소에서 모든 요청을 처리합니다. 비용 효율을 중시하는 조직에 적합합니다.
- **가상(Virtual)** — 분산된 인력이 하나의 시스템으로 통합됩니다. 원격 근무나 다국적 조직에 적합합니다.
- **Follow-the-Sun** — 시간대별 교대 운영(24/7)입니다. 글로벌 기업이나 통신사에 적합합니다.

**접수 채널**

- **셀프서비스 포털** — 24/7 이용 가능하고 비용 효율적입니다. 비밀번호 초기화, 표준 요청에 적합합니다.
- **전화** — 즉각 소통이 가능합니다. 긴급 인시던트에 적합합니다.
- **이메일** — 기록이 보존됩니다. 비긴급이면서 상세 설명이 필요한 요청에 적합합니다.
- **챗봇** — 자동화되고 확장성이 좋습니다. FAQ, 단순 문의에 적합합니다.


**핵심 프로세스 1: 인시던트 관리 (Incident Management)**

**인시던트**는 쉽게 말해 "지금 뭔가 안 되는 상황"입니다.

레스토랑에서 손님이 "국에 머리카락이 있어요"라고 하면, 원인 분석할 시간이 없습니다. 일단 새 국을 가져다드리는 게 먼저죠. 인시던트 관리도 마찬가지입니다. **원인 파악보다 빠른 복구가 목표**입니다.

**워크플로우**

- 1단계 **접수** — 사용자 신고 또는 모니터링 알림으로 시작됩니다. 서비스 데스크가 담당합니다.
- 2단계 **분류** — 네트워크/서버/애플리케이션 등 유형을 파악합니다. 서비스 데스크가 담당합니다.
- 3단계 **우선순위 결정** — 영향도 x 긴급도 = 우선순위로 결정합니다. 서비스 데스크가 담당합니다.
- 4단계 **진단 및 해결** — L1에서 해결하거나 L2/L3로 에스컬레이션합니다. L1/L2/L3 지원팀이 담당합니다.
- 5단계 **종료** — 사용자 확인 후 티켓을 닫습니다. 서비스 데스크가 담당합니다.

**우선순위 매트릭스**

영향도와 긴급도의 조합으로 우선순위가 결정됩니다.

- 영향도 높음 + 긴급도 높음 = **P1 (Critical)**
- 영향도 높음 + 긴급도 중간 = **P2 (High)**
- 영향도 높음 + 긴급도 낮음 = **P3 (Medium)**
- 영향도 중간 + 긴급도 높음 = **P2 (High)**
- 영향도 중간 + 긴급도 중간 = **P3 (Medium)**
- 영향도 중간 + 긴급도 낮음 = **P4 (Low)**
- 영향도 낮음 + 긴급도 높음 = **P3 (Medium)**
- 영향도 낮음 + 긴급도 중간 = **P4 (Low)**
- 영향도 낮음 + 긴급도 낮음 = **P5 (Planning)**

**통신사 시나리오**

월요일 아침, 고객센터 상담 시스템이 갑자기 응답이 느려집니다. 상담사 200명이 동시에 영향을 받으니 "영향도 높음 + 긴급도 높음 = P1(Critical)"로 분류됩니다. P1이면 15분 이내 응답, 4시간 이내 해결이 목표입니다.


**핵심 프로세스 2: 문제 관리 (Problem Management)**

인시던트 관리가 "불 끄기"라면, **문제 관리**는 "왜 불이 났는지 찾아서 다시는 안 나게 하는 것"입니다.

같은 장애가 반복되면, 매번 복구만 하는 게 아니라 **근본 원인(Root Cause)**을 파헤쳐서 영구적으로 해결합니다.

**인시던트 관리 vs 문제 관리**

- **목적** — 인시던트 관리는 빠른 서비스 복구, 문제 관리는 근본 원인 제거와 재발 방지입니다.
- **시간 관점** — 인시던트 관리는 단기(지금 당장), 문제 관리는 중장기(근본적으로)입니다.
- **해결 방식** — 인시던트 관리는 우회 방법 포함, 문제 관리는 영구적 해결을 추구합니다.
- **비유** — 인시던트 관리는 소방관, 문제 관리는 화재 조사관입니다.

**통신사 시나리오**

매주 월요일 아침마다 빌링 시스템이 느려지는 인시던트가 반복됩니다. 인시던트 관리에서는 매번 서버를 재시작해서 복구하지만, 문제 관리에서 추적해 보니 주말 배치 작업이 정상 종료되지 않아 월요일 아침에 리소스를 잡아먹고 있었습니다. 배치 작업의 종료 로직을 수정해서 영구적으로 해결합니다.

**워크플로우**

- 1단계 **식별** — 반복 인시던트를 분석하고 트렌드를 파악합니다. 산출물은 문제 레코드입니다.
- 2단계 **근본 원인 분석** — 5 Why, Ishikawa 등 기법을 활용합니다. 산출물은 근본 원인 보고서입니다.
- 3단계 **알려진 오류 등록** — 원인 파악이 완료된, 해결 전 상태를 기록합니다. 산출물은 KEDB 등록입니다.
- 4단계 **해결** — 영구적 해결책을 구현하거나 변경 요청을 생성합니다. 산출물은 RFC입니다.
- 5단계 **종료** — 해결을 검증하고 지식베이스를 업데이트합니다. 산출물은 Knowledge Base 업데이트입니다.


**핵심 프로세스 3: 변경 관리 (Change Enablement)**

어떤 IT 환경이든 변경은 피할 수 없습니다. 서버 패치, 신규 시스템 배포, 네트워크 구성 변경 등이 매일 일어나죠.

문제는 **변경이 새로운 장애의 원인이 되는 경우가 꽤 많다**는 겁니다. 변경 관리는 이 리스크를 최소화하는 프로세스입니다.

**변경 유형**

- **표준 변경** — 사전 승인된 낮은 리스크 변경입니다. 자동 승인됩니다. 예: 패치 적용, 계정 생성
- **일반 변경** — CAB 검토가 필요한 변경입니다. CAB 승인을 받습니다. 예: 서버 마이그레이션, 신규 앱 배포
- **긴급 변경** — 즉시 처리해야 하는 변경입니다. ECAB 긴급 승인을 받습니다. 예: P1 인시던트 대응, 보안 패치

**워크플로우**

- 1단계 **변경 요청(RFC)** — 변경 사유, 범위, 일정을 기술합니다. 변경 요청자가 담당합니다.
- 2단계 **리스크 평가** — 영향 분석, 롤백 계획을 수립합니다. 변경 관리자가 담당합니다.
- 3단계 **승인** — CAB 검토 또는 사전 승인을 확인합니다. CAB 또는 변경 권한자가 담당합니다.
- 4단계 **구현** — 변경 계획에 따라 실행합니다. 기술팀이 담당합니다.
- 5단계 **검토** — 성공 여부를 확인하고, 교훈을 기록합니다. 변경 관리자가 담당합니다.

**통신사 시나리오**

과금 시스템 업그레이드를 진행합니다. "그냥 주말에 올리면 되지"가 아니라 — 어떤 서비스가 영향받는지, 실패 시 롤백은 어떻게 하는지, 고객 안내는 필요한지를 전부 검토하고 CAB 승인을 받은 뒤에 진행합니다.


**핵심 프로세스 4: 서비스 요청 관리 (Service Request Management)**

인시던트가 "뭔가 고장 난 것"이라면, **서비스 요청**은 "새로 뭔가 필요한 것"입니다.

**대표적인 서비스 요청 유형**

- **계정/접근** — 계정 생성, 비밀번호 초기화, 권한 변경. 일반적인 SLA는 4시간에서 1영업일입니다.
- **하드웨어** — 노트북 요청, 모니터 추가. 일반적인 SLA는 3에서 5영업일입니다.
- **소프트웨어** — SW 설치, 라이선스 요청. 일반적인 SLA는 1에서 3영업일입니다.
- **정보 요청** — 절차 문의, 상태 확인. 일반적인 SLA는 1영업일 이내입니다.

서비스 요청은 미리 카탈로그로 정의해 놓으면 처리가 훨씬 빨라집니다. 셀프서비스 포털에서 "비밀번호 초기화" 같은 건 사용자가 직접 처리할 수도 있고요.


**SLA — 약속을 숫자로 만드는 것**

지금까지 나온 프로세스들이 "잘 돌아가고 있는지" 어떻게 알 수 있을까요? 여기서 등장하는 게 **SLA(Service Level Agreement)**입니다.

SLA는 IT 부서와 사용자(또는 비즈니스 부서) 사이의 **서비스 수준 약속**입니다.

**레스토랑 비유**

- "주문 후 15분 이내에 음식이 나갑니다" — 이게 SLA입니다.
- 실제로 평균 12분에 나가고 있다면 SLA 준수입니다.
- 20분이 걸렸다면 SLA 위반입니다.

**IT에서의 SLA 예시**

- **P1 (Critical)** — 응답 15분 이내, 해결 4시간 이내, 30분마다 업데이트
- **P2 (High)** — 응답 30분 이내, 해결 8시간 이내, 2시간마다 업데이트
- **P3 (Medium)** — 응답 2시간 이내, 해결 24시간 이내, 1일 1회 업데이트
- **P4 (Low)** — 응답 4시간 이내, 해결 72시간 이내, 진행 시 업데이트

**SLA 계층 구조**

SLA를 지키기 위해서는 내부 약속과 외부 계약도 필요합니다.

- **SLA** (Service Level Agreement) — IT와 비즈니스 부서 간 약속입니다. "4시간 안에 해결하겠습니다."
- **OLA** (Operational Level Agreement) — IT 내부 부서 간 약속입니다. "네트워크팀은 2시간 안에 진단 완료."
- **UC** (Underpinning Contract) — 외부 벤더와의 계약입니다. "벤더는 1시간 내 원격 지원 제공."

비기술 직군에게 SLA가 중요한 이유는, **"내가 요청한 게 언제 처리되는지" 기대치를 알 수 있다**는 겁니다. "IT에 요청했는데 언제 해주는지 모르겠다"가 "요청하면 4시간 안에 처리됩니다"로 바뀝니다.


**ITSM 도입 전 vs 도입 후**

- **장애 신고** — 도입 전: 아는 사람한테 전화, 메신저 / 도입 후: 서비스 데스크 티켓으로 접수, 자동 배정
- **해결 이력** — 도입 전: "지난번에 어떻게 했더라?" 기억에 의존 / 도입 후: 지식베이스에서 이전 사례 검색
- **시스템 변경** — 도입 전: "그냥 올려, 뭐 문제 있겠어" / 도입 후: 영향 분석 + 승인 + 롤백 계획 후 진행
- **서비스 요청** — 도입 전: 어디에 요청하는지부터 물어야 함 / 도입 후: 셀프서비스 포털에서 직접 요청, 진행 상황 확인
- **성과 측정** — 도입 전: "느낌상 잘 하고 있는 것 같은데..." / 도입 후: SLA 준수율, MTTR, FCR 등 KPI 기반 정량 평가
- **담당자 부재** — 도입 전: 그 사람 올 때까지 대기 / 도입 후: 프로세스에 따라 다른 담당자가 이어서 처리

결국 **"사람에 의존하는 운영"에서 "프로세스에 기반한 운영"**으로 전환되는 겁니다.


**통신사 환경에서 ITSM이 특히 중요한 이유**

- **24/7 운영** — 교대 근무 간 정보가 끊기지 않도록 체계적 인시던트 관리와 인수인계 구조가 필수입니다.
- **변경의 영향 범위** — 과금, CRM 같은 핵심 시스템 변경은 수백만 고객에게 영향을 줍니다. 변경 관리 프로세스 없이는 리스크가 너무 큽니다.
- **규제 준수** — 장애 대응 이력, 변경 승인 기록이 감사 추적(Audit Trail)으로 남아야 하며, ITSM은 이런 기록을 자연스럽게 남기는 구조입니다.


**약어 정리**

- **SPOC** — Single Point of Contact: 단일 접점 (서비스 데스크의 핵심 원칙)
- **SLA** — Service Level Agreement: 서비스 수준 합의 (응답/해결 시간 약속)
- **OLA** — Operational Level Agreement: 내부 운영 수준 합의 (부서 간 약속)
- **UC** — Underpinning Contract: 외부 공급업체 계약 (SLA 이행 뒷받침)
- **CAB** — Change Advisory Board: 변경 자문 위원회
- **ECAB** — Emergency CAB: 긴급 변경 자문 위원회
- **RFC** — Request for Change: 변경 요청서
- **KEDB** — Known Error Database: 알려진 오류 데이터베이스
- **L1/L2/L3** — Level 1/2/3 Support: 1/2/3차 기술 지원 단계
- **FCR** — First Contact Resolution: 첫 번째 연락 해결율
