ITSM/ITOM/ITAM 시리즈 2편 — ITSM, 서비스 데스크부터 SLA까지

Part 1에서 ITSM을 "레스토랑의 홀 서비스"에 비유했습니다. 이번 글에서는 그 홀 서비스가 실제로 어떤 일을 하는지 — 핵심 프로세스 4가지와 SLA를 중심으로 정리합니다.

(상세 버전은 Confluence Part 2를 참고해 주세요.)


━━━━━━━━━━━━━━━━━━━━
📌 서비스 데스크 — 모든 것의 시작점
━━━━━━━━━━━━━━━━━━━━

ITSM에서 가장 먼저 만나는 것이 **서비스 데스크(Service Desk)**입니다.

레스토랑의 "안내 데스크"라고 보면 됩니다. 노트북이 고장 났든, VPN이 안 되든, 새 소프트웨어가 필요하든 — 모든 건 서비스 데스크를 통해 들어옵니다. 이걸 **단일 접점(SPOC)**이라 부릅니다.

창구가 하나로 통일되어야 "어디에 연락해야 하지?"가 사라지고, IT 쪽에서도 우선순위를 정하고 빠뜨리는 건 없는지 확인할 수 있습니다. 유형은 로컬·중앙집중·가상·Follow-the-Sun 네 가지가 있고, 포털·전화·이메일·챗봇 등 다양한 채널로 접수합니다.


━━━━━━━━━━━━━━━━━━━━
🔧 핵심 프로세스 1: 인시던트 관리
━━━━━━━━━━━━━━━━━━━━

**인시던트** = "지금 뭔가 안 되는 상황". **원인 파악보다 빠른 복구가 목표**입니다.

레스토랑에서 "국에 머리카락이 있어요"라고 하면 원인 분석할 시간이 없습니다. 일단 새 국을 가져다드리는 게 먼저죠.

**왜 '인시던트'라는 개념이 따로 필요한가?**

이 개념이 없으면 현장에서 공통 언어가 없어 회의에서 말이 엇갈리고, 장애와 문의가 뒤섞여 급한 건이 밀리고, "지난달 장애가 몇 번?"에 아무도 답하지 못합니다. 인시던트는 **"IT 운영을 측정 가능하게 만드는 최소 단위"**이고, 문제 관리·변경 관리·SLA 관리가 전부 이 토대 위에 서 있습니다.

"크리티컬 이벤트만 관리하면 된다"는 반론도 있지만, 작은 인시던트를 기록하지 않으면 조기 경보도, 재발 분석도, SLA 측정도 불가능합니다. **인시던트 관리는 토대, 크리티컬 이벤트 관리는 그 위의 상위 레이어** — 둘은 층을 이뤄야 합니다. (상세 비교는 Confluence 참고)

**워크플로우**: 접수 → 분류 → 우선순위(영향도 x 긴급도) → 해결(L1/L2/L3) → 종료

우선순위는 영향도와 긴급도의 조합입니다. 둘 다 높으면 P1(Critical), 둘 다 낮으면 P5(Planning).

**통신사 시나리오**: 월요일 아침 고객센터 상담 시스템이 느려짐. 상담사 200명 영향 → P1 분류 → 15분 이내 응답, 4시간 이내 해결이 목표.


━━━━━━━━━━━━━━━━━━━━
🔍 핵심 프로세스 2: 문제 관리
━━━━━━━━━━━━━━━━━━━━

인시던트 관리 = **소방관** (불 끄기), 문제 관리 = **화재 조사관** (왜 불이 났는지 찾기).

같은 장애가 반복되면 매번 복구만 하는 게 아니라 **근본 원인(Root Cause)**을 파헤쳐서 영구적으로 해결합니다.

**워크플로우**: 반복 인시던트 식별 → 근본 원인 분석(5 Why, Ishikawa 등) → 알려진 오류 등록(KEDB) → 영구 해결 또는 변경 요청(RFC) → 종료

**통신사 시나리오**: 매주 월요일 아침 빌링 시스템 느려짐이 반복됨. 추적해 보니 주말 배치 작업 미종료 → 월요일 리소스 잠식. 배치 종료 로직 수정으로 영구 해결.


━━━━━━━━━━━━━━━━━━━━
🔄 핵심 프로세스 3: 변경 관리
━━━━━━━━━━━━━━━━━━━━

**변경이 새로운 장애의 원인이 되는 경우가 꽤 많습니다.** 변경 관리는 이 리스크를 최소화하는 프로세스입니다.

변경 유형 3가지:
- **표준 변경** — 사전 승인된 낮은 리스크 (예: 패치, 계정 생성)
- **일반 변경** — CAB 검토·승인 필요 (예: 서버 마이그레이션)
- **긴급 변경** — ECAB 긴급 승인 (예: P1 대응, 보안 패치)

**워크플로우**: 변경 요청(RFC) → 리스크 평가·롤백 계획 → 승인 → 구현 → 사후 검토

**통신사 시나리오**: 과금 시스템 업그레이드. "그냥 주말에 올리면 되지"가 아니라 — 영향 서비스, 롤백 방법, 고객 안내 여부를 전부 검토하고 CAB 승인 후 진행합니다.


━━━━━━━━━━━━━━━━━━━━
📋 핵심 프로세스 4: 서비스 요청 관리
━━━━━━━━━━━━━━━━━━━━

인시던트 = "뭔가 고장 난 것", 서비스 요청 = **"새로 뭔가 필요한 것"**.

- 계정/접근 (비밀번호 초기화, 권한 변경) — SLA 4시간~1영업일
- 하드웨어 (노트북, 모니터) — SLA 3~5영업일
- 소프트웨어 (설치, 라이선스) — SLA 1~3영업일
- 정보 요청 (절차 문의) — SLA 1영업일 이내

미리 카탈로그로 정의해 두면 처리가 빨라지고, 셀프서비스 포털로 사용자가 직접 처리할 수도 있습니다.


━━━━━━━━━━━━━━━━━━━━
📊 SLA — 약속을 숫자로 만드는 것
━━━━━━━━━━━━━━━━━━━━

**SLA(Service Level Agreement)** = IT와 사용자 사이의 서비스 수준 약속.

레스토랑 비유:
- "주문 후 15분 이내에 음식이 나갑니다" → SLA
- 평균 12분 → SLA 준수 / 20분 → SLA 위반

IT에서의 SLA:
- **P1** — 응답 15분, 해결 4시간, 30분마다 업데이트
- **P2** — 응답 30분, 해결 8시간
- **P3** — 응답 2시간, 해결 24시간
- **P4** — 응답 4시간, 해결 72시간

SLA를 지키려면 **내부 약속(OLA)**과 **외부 계약(UC)**도 필요합니다.
- **SLA** — IT와 비즈니스 부서 간 약속. "4시간 안에 해결하겠습니다."
- **OLA** — IT 내부 부서 간 약속. "네트워크팀은 2시간 안에 진단 완료."
- **UC** — 외부 벤더 계약. "벤더는 1시간 내 원격 지원 제공."

비기술 직군에게 SLA가 중요한 이유는 **"내가 요청한 게 언제 처리되는지" 기대치를 알 수 있다**는 겁니다.


━━━━━━━━━━━━━━━━━━━━
⚡ ITSM이 있으면 뭐가 달라지나
━━━━━━━━━━━━━━━━━━━━

- **장애 신고** — "아는 사람한테 전화" → 서비스 데스크 티켓 접수, 자동 배정
- **시스템 변경** — "그냥 올려" → 영향 분석 + 승인 + 롤백 계획 후 진행
- **성과 측정** — "느낌상 잘 하는 것 같은데" → SLA 준수율·MTTR·FCR 등 정량 평가

한 마디로, **"사람에 의존하는 운영"에서 "프로세스에 기반한 운영"**으로 전환됩니다.


━━━━━━━━━━━━━━━━━━━━
📡 통신사 환경에서 ITSM이 특히 중요한 이유
━━━━━━━━━━━━━━━━━━━━

- **24/7 운영** — 교대 근무 간 정보 끊김 방지를 위해 체계적 인시던트 관리와 인수인계 구조가 필수입니다.
- **변경의 영향 범위** — 과금·CRM 같은 핵심 시스템 변경은 수백만 고객에게 영향. 변경 관리 없이는 리스크가 너무 큽니다.
- **규제 준수** — 장애 대응 이력, 변경 승인 기록이 감사 추적(Audit Trail)으로 남아야 합니다. ITSM은 이런 기록을 자연스럽게 남기는 구조입니다.


━━━━━━━━━━━━━━━━━━━━
📖 약어 정리
━━━━━━━━━━━━━━━━━━━━

- **SPOC** — Single Point of Contact: 단일 접점
- **SLA** — Service Level Agreement: 서비스 수준 합의
- **OLA** — Operational Level Agreement: 내부 운영 수준 합의
- **UC** — Underpinning Contract: 외부 공급업체 계약
- **CAB** — Change Advisory Board: 변경 자문 위원회
- **ECAB** — Emergency CAB: 긴급 변경 자문 위원회
- **RFC** — Request for Change: 변경 요청서
- **KEDB** — Known Error Database: 알려진 오류 데이터베이스
- **L1/L2/L3** — Level 1/2/3 Support: 1/2/3차 기술 지원
- **FCR** — First Contact Resolution: 첫 번째 연락 해결율
