ITSM/ITOM/ITAM 시리즈 1편 — 도대체 뭐가 다른 건가요?

**이 글을 쓴 이유**

요즘 우리 회사에서 ITSM, ITOM, ITIL 같은 용어가 정말 자주 들리지 않습니까?

일하면서 이 용어들을 매일 듣지만, 사람마다 이해하는 범위가 달라서 회의할 때 미묘하게 어긋나는 경험이 몇 번 있었는데요. 그래서 이번 기회에 한번 정리해 봤습니다.


**레스토랑으로 이해하는 IT 용어**

레스토랑이 잘 돌아가려면 홀 서비스, 주방 운영, 재고 관리의 세 가지가 필요합니다. IT 업무와 비교해 보겠습니다.

- **ITSM** — 홀 서비스 (주문, 서빙, 피드백): 사용자 요청을 접수하고 해결해주는 체계입니다. 핵심 질문은 "직원들이 불편 없이 일할 수 있나?"입니다.
- **ITOM** — 주방 운영 (설비 점검, 화력 관리): 서버/네트워크를 24시간 모니터링하고 장애를 예방합니다. 핵심 질문은 "시스템이 멈추지 않고 잘 돌아가나?"입니다.
- **ITAM** — 식재료/장비 재고 관리: IT 장비와 라이선스의 수량, 상태, 비용을 추적합니다. 핵심 질문은 "뭐가 몇 개 있고, 돈이 새고 있진 않나?"입니다.

이 세 가지가 결국 우리가 실제로 하는 "일(운영)"입니다.


**좀 더 자세히 볼까요**

- **ITSM (IT Service Management)** — IT 서비스를 관리하는 활동 전체를 말합니다. "노트북 고장 났어요", "새 소프트웨어 깔아주세요" 같은 요청을 접수하고 해결해주는 것이 ITSM의 대표적인 모습입니다.
- **ITOM (IT Operations Management)** — 서버, 네트워크, 클라우드 같은 IT 인프라가 안정적으로 돌아가는지 24시간 모니터링하고, 문제가 터지기 전에 미리 잡아내는 역할입니다. 우리 눈에는 안 보이지만, ITOM이 멈추면 모든 IT 서비스가 마비됩니다.
- **ITAM (IT Asset Management)** — 회사가 가진 IT 자산(노트북, 서버, 소프트웨어 라이선스 등)이 몇 개 있고, 상태는 어떻고, 비용은 적절한지 추적하는 활동입니다. "우리 회사 라이선스 100개 샀는데 실제로 쓰는 건 60개뿐이었네?" 같은 걸 잡아내는 거죠.


**이 세 가지가 어떻게 연결되나요?**

ITSM, ITOM, ITAM은 따로 노는 게 아니라 서로 데이터를 주고받습니다. 그 중심에 **CMDB**라는 통합 데이터베이스가 있습니다.

ITSM(서비스 요청, 인시던트, 변경 관리), ITOM(인프라 상태, 이벤트, 모니터링), ITAM(자산 정보, 라이선스, 재고/비용) 세 영역이 양방향으로 데이터를 교환하고, 이 데이터가 모두 CMDB(통합 DB)에 모입니다.

실제로 어떻게 연결되는지 예를 들어 보겠습니다.

- **ITSM 연결 ITAM** — 직원이 "새 노트북 주세요" 요청하면 ITAM에서 재고 확인하고 바로 배치합니다.
- **ITOM 연결 ITAM** — 서버 장애 감지 시 ITAM에서 "이 서버 보증 남았나? 교체 시기 됐나?" 확인합니다.
- **ITSM 연결 ITOM** — 직원이 "시스템 느려요" 신고하면 ITOM 모니터링 데이터로 원인을 분석합니다.
- **CMDB** — 이 세 영역의 데이터가 한곳에 모이는 허브입니다. 자산, 서비스, 인프라 정보의 '단일 진실의 원천'이거든요.


**ITAM과 CMDB는 뭐가 다른 건가요?**

둘 다 "IT 자산"을 다루는 것 같지만, 사실 바라보는 관점이 다릅니다.

레스토랑으로 돌아가면 이렇습니다.
- **ITAM** = 장비 **구매 장부** — 오븐을 언제 샀고, 얼마이고, 보증은 언제까지인지
- **CMDB** = 주방 **배치도** — 오븐이 어디에 있고, 가스관과 어떻게 연결되어 있는지

각각이 보는 것이 다릅니다.
- **ITAM이 보는 것** — 구매일, 가격, 보증기간, 라이선스 수
- **CMDB가 보는 것** — OS 버전, IP 주소, 연결된 서비스

주로 쓰는 사람도 다릅니다.
- **ITAM** — 재무팀, IT 관리자, 감사팀
- **CMDB** — IT 운영팀, 변경관리팀

필요한 상황도 다릅니다.
- **ITAM** — "이 장비 교체할까 수리할까?"
- **CMDB** — "이 서버 죽으면 뭐가 영향받지?"

같은 서버 1대를 두고도 ITAM은 "2022년 구매, 5,000만원, 보증 2026년까지, 감가상각 4년차"를 봅니다. CMDB는 "CentOS 8, IP 10.0.1.50, 빌링 서비스에 의존, 스위치 A에 연결"을 봅니다.

장애가 나면 CMDB로 **영향 범위**를 파악하고, ITAM으로 **교체 vs 수리**를 판단합니다. 둘이 합쳐져야 완전한 그림이 나옵니다.


**우리 회사 업무에 적용해 볼까요**

**상황 1: 고객센터 시스템이 느려졌다**

금요일 오후, 고객센터 상담사들이 "시스템이 버벅거려요"라고 합니다.

- 1단계 **ITOM** — 모니터링이 "빌링 서버 CPU 95% 초과" 알림을 띄웁니다. 사람보다 시스템이 먼저 알아챈 겁니다.
- 2단계 **ITSM** — 알림을 기반으로 장애 티켓이 자동으로 만들어지고, 담당 엔지니어에게 배정됩니다. 상담사가 신고하면 "이미 처리 중입니다"라고 바로 안내할 수 있죠.
- 3단계 **ITAM** — 해당 서버 정보를 꺼내봅니다. "도입 5년 차, 보증 2개월 남음" — 임시로 때울지, 아예 교체할지 판단하는 근거가 됩니다.

**상황 2: 새벽에 기지국 장비가 이상하다**

새벽 2시, 특정 지역 기지국에서 패킷 로스(Packet Loss)가 급증합니다.

- 1단계 **ITOM** — NMS가 "A 지역 라우터 에러율 임계치 초과"를 감지합니다. 고객 불만이 들어오기 전에 경보가 울린 겁니다.
- 2단계 **ITSM** — 장애 티켓이 만들어지고 네트워크 운영팀에 배정됩니다. "약 3,000가구 영향" 정보가 티켓에 붙어서, VOC가 들어오면 "인지된 장애, 복구 중"이라고 바로 안내할 수 있습니다.
- 3단계 **ITAM** — 해당 라우터 정보: "도입 7년 차, EoS 1년 전 도래, 예비 부품 2개 남음" — 현장에서 바로 교체 가능한지 판단할 수 있습니다.

서버든 네트워크 장비든, 결국 같은 패턴입니다. **ITOM이 감지하고, ITSM이 처리하고, ITAM이 자산 판단을 돕습니다.**


**이걸 체계적으로 하려면? 가이드가 있습니다**

여기까지 읽으셨으면 ITSM, ITOM, ITAM이 뭘 하는 건지 감이 잡히셨을 겁니다.

그런데 "인시던트는 어떻게 분류하지?", "에스컬레이션 기준은?", "사후 분석은 어떤 절차로?" 같은 구체적인 방법이 필요하잖아요. 이때 참고하는 가이드가 있습니다.

- **ITIL 4** — 레시피북에 해당합니다. "인시던트는 이렇게, 변경은 이렇게 관리해라" — 가장 널리 쓰이는 모범 사례집입니다. IT 운영팀, 서비스 관리자가 주로 씁니다.
- **COBIT** — 위생 감사 체크리스트에 해당합니다. "IT가 비즈니스에 가치를 주고 있나? 리스크는 통제되나?" — 거버넌스 프레임워크입니다. 경영진, 감사팀, 이사회가 주로 씁니다.
- **ISO/IEC 20000** — 미쉐린 인증에 해당합니다. "우리 ITSM을 외부에 증명" — 제3자 감사를 통한 국제 표준 인증입니다. 서비스 제공업체, 규제 대응 조직이 주로 씁니다.

쉽게 말하면, **ITSM/ITOM/ITAM은 "우리가 하는 일"**이고, **ITIL/COBIT/ISO 20000은 "그 일을 잘 하기 위한 참고서"**입니다.

레스토랑에서 요리(ITSM)를 레시피북(ITIL) 없이도 할 수 있지만, 레시피를 따르면 더 맛있고 일관되게 할 수 있는 것처럼요.


**이걸 잘 하면 뭐가 달라지나요?**

체계적으로 도입한 기업들의 사례를 보면, 다음과 같은 개선이 보고되고 있습니다.

- 장애 평균 해결 시간(MTTR)이 **절반 수준**으로 단축
- 첫 번째 연락에서 바로 해결되는 비율(FCR)이 **대폭 향상**
- 변경 성공률 상승, 반복 인시던트 감소

IT 부서만의 이야기가 아닙니다. 시스템을 쓰는 모든 직원의 업무 경험이 달라진다는 뜻이거든요.

"IT에 요청했는데 어떻게 되고 있는지 모르겠다"가 "요청하자마자 진행 상황이 보인다"로 바뀌는 겁니다.


**약어 정리**

- **ITSM** — IT Service Management: IT 서비스 관리 (사용자 대면)
- **ITOM** — IT Operations Management: IT 인프라 운영 관리 (백엔드)
- **ITAM** — IT Asset Management: IT 자산 생애주기 관리
- **ITIL** — Information Technology Infrastructure Library: IT 서비스 관리 모범 사례집
- **COBIT** — Control Objectives for Information and Related Technologies: IT 거버넌스 프레임워크
- **ISO/IEC 20000** — ITSM 국제 표준 인증
- **MTTR** — Mean Time To Resolve: 평균 장애 해결 시간
- **FCR** — First Contact Resolution: 첫 번째 연락 해결율
- **CMDB** — Configuration Management Database: IT 구성요소와 관계를 관리하는 통합 DB
- **NMS** — Network Management System: 네트워크 관리 시스템
- **EoS** — End of Support: 제조사 지원 종료
- **VOC** — Voice of Customer: 고객의 소리
