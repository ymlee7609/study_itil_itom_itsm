## 이 글의 위치

Part 1~5에서 ITSM, ITOM, ITAM과 ServiceNow, 그리고 2026년 2월 출시된 ITIL 5를 다뤘습니다. 시리즈 이후 가장 많이 받은 질문이 "ITIL만으로 거버넌스 체계가 완성되는가?"였습니다. 결론부터 말하면 **ITIL만으로는 부족한 경우가 많습니다.** 이번 보너스 Part에서는 거버넌스 체계 구축에 꼭 등장하는 두 가지 프레임워크 — **COBIT**과 **ISO/IEC 20000** — 을 정리하고, 세 프레임워크를 층으로 쌓는 현실적 로드맵을 제시합니다.

---

## 1. ITIL 혼자서는 왜 부족한가

IT 운영과 거버넌스를 이야기할 때 자주 등장하는 세 프레임워크의 역할은 서로 다릅니다.

| 프레임워크 | 성격 | 핵심 질문 | 주요 대상 |
|-----------|------|---------|-----------|
| **ITIL 5** | 모범 사례집 (Best Practice) | 인시던트·변경·SLA를 어떻게 운영할 것인가? | IT 운영팀, 서비스 관리자 |
| **COBIT** | 거버넌스 프레임워크 (Governance Framework) | IT가 비즈니스 가치에 기여하고 리스크가 통제되고 있는가? | 경영진, 이사회, 감사인 |
| **ISO/IEC 20000** | 국제 표준 (International Standard, 인증 가능) | 우리 ITSM이 객관적으로 기준을 만족한다고 외부에 증명할 수 있는가? | 서비스 제공자, 규제 대응 조직 |

한 문장으로 정리하면 이렇습니다.

> **ITIL은 레시피북, COBIT은 위생 감사 체크리스트, ISO 20000은 미쉐린 인증입니다.**

세 프레임워크는 경쟁 관계가 아니라 층을 이루는 관계입니다. 운영 실무(ITIL) 위에 거버넌스(COBIT)가 얹히고, 그 전체를 외부 인증(ISO 20000)이 검증합니다.

## 2. COBIT — 경영진의 언어로 IT를 관리하기

### 2.1 COBIT 개요

**COBIT(Control Objectives for Information and Related Technologies)**는 ISACA(정보시스템 감사·통제 협회)가 1996년부터 발전시켜 온 IT 거버넌스 프레임워크입니다. 현재 실무에서 사용되는 버전은 **COBIT 2019**이며, 2023년 **Focus Areas 업데이트**로 정보 보안, DevOps, 중소기업(SMB) 영역이 보강됐습니다.

| 항목 | 내용 |
|------|------|
| **정식 명칭** | COBIT 2019 (Control Objectives for Information and Related Technologies) |
| **관리 기관** | ISACA (Information Systems Audit and Control Association) |
| **최초 출시** | 1996년 (COBIT 1) |
| **현재 버전** | COBIT 2019 (2018년 출시, 2023년 Focus Areas 업데이트) |
| **목표 수** | 거버넌스 5개 + 관리 35개 = 총 40개 |
| **주요 대상** | 경영진, 이사회, IT 감사인, 규제 대응 담당자 |

### 2.2 COBIT이 해결하는 질문

ITIL이 "인시던트를 어떻게 처리하나?"라는 실무 질문에 답한다면, COBIT은 경영진·이사회의 질문에 답합니다.

| 질문 | COBIT이 제공하는 답 |
|------|------------------|
| IT 투자가 비즈니스 목표에 기여하고 있는가? | Goals Cascade로 전략 정렬 구조 제공 |
| IT 리스크는 누가 어떻게 통제하는가? | EDM03(리스크 관리), APO12(리스크) 목표 |
| 규제 요구사항을 충족하고 있는가? | MEA03(외부 요구사항 준수 모니터링) 목표 |
| IT 의사결정의 책임과 권한은 명확한가? | 각 목표에 RACI 매트릭스 제공 |
| 이사회에 IT 상태를 어떻게 보고할 것인가? | EDM05(이해관계자 투명성 확보) 목표 |

### 2.3 COBIT 2019의 구조

COBIT은 **Governance(거버넌스)**와 **Management(관리)**를 구조적으로 분리합니다.

| 영역 | 약어 | 활동 유형 | 목표 수 | 주체 |
|------|------|---------|--------|------|
| **Evaluate, Direct, Monitor** | EDM | 거버넌스 (평가·지시·감시) | 5개 | 이사회, 경영진 |
| **Align, Plan, Organize** | APO | 관리 (정렬·계획·조직) | 14개 | CIO, IT 전략팀 |
| **Build, Acquire, Implement** | BAI | 관리 (구축·획득·구현) | 11개 | 프로젝트 매니저, 개발팀 |
| **Deliver, Service, Support** | DSS | 관리 (전달·서비스·지원) | 6개 | IT 운영팀, 서비스 데스크 |
| **Monitor, Evaluate, Assess** | MEA | 관리 (모니터링·평가·심사) | 4개 | 내부 감사, 성과 관리팀 |

각 목표에는 다음이 포함됩니다.

- 목표 설명과 비즈니스 가치
- 프로세스 실천 활동
- 입력·출력 산출물
- RACI 매트릭스 (책임 구조)
- 성숙도 레벨 (0~5단계)
- 관련 표준·프레임워크 매핑 (ITIL, ISO 27001 등)

### 2.4 Goals Cascade — 목표 연쇄 구조

COBIT의 핵심 개념 중 하나가 **Goals Cascade(목표 연쇄)**입니다. 조직의 최상위 목표가 IT의 실무 활동까지 연결되는 구조입니다.

| 계층 | 설명 | 예시 |
|------|------|------|
| **이해관계자 니즈** | 이사회·주주·고객·규제기관의 요구 | "규제 준수와 주주 가치 극대화" |
| **조직 목표** | 17개 표준 엔터프라이즈 목표 | "재무 투명성", "고객 지향 서비스 문화" |
| **정렬 목표** | 13개 IT 정렬 목표 | "비즈니스 프로세스 자동화", "IT 리스크 관리" |
| **거버넌스·관리 목표** | 40개 COBIT 목표 | EDM03, APO12, DSS02 등 |

이 구조 덕분에 "우리 팀이 이 일을 왜 하는지"를 이사회의 언어까지 거슬러 올라가 설명할 수 있습니다.

### 2.5 Design Factors — 조직별 맞춤 설계

COBIT 2019의 큰 변화 중 하나가 **Design Factors(설계 요인)**입니다. 11가지 요인을 입력하면 조직 상황에 맞게 우선순위가 조정된 거버넌스 시스템이 산출됩니다.

| # | Design Factor | 영향 |
|---|--------------|------|
| 1 | 엔터프라이즈 전략 | 어떤 거버넌스 목표가 핵심인가 |
| 2 | 엔터프라이즈 목표 | 어떤 비즈니스 가치를 추적할 것인가 |
| 3 | 리스크 프로파일 | 리스크 관련 목표의 가중치 |
| 4 | IT 관련 이슈 | 현재 해결해야 할 문제 영역 |
| 5 | 위협 환경 | 보안·가용성 목표의 중요도 |
| 6 | 규제 요구사항 | MEA03 등 컴플라이언스 목표의 가중치 |
| 7 | 소싱 모델 | 내부·외부·클라우드 혼합 정도 |
| 8 | IT 구현 방식 | 전통적·Agile·DevOps 혼합 |
| 9 | 기술 도입 전략 | 최신 기술 수용 수준 |
| 10 | 엔터프라이즈 규모 | 대기업·중소기업 구분 |
| 11 | 산업 | 금융·제조·공공 등 특성 |

"작은 조직에 40개 목표를 전부 들이대지 말라"는 현실적 조정 장치입니다.

### 2.6 COBIT이 필요한 조직

| 조직 특성 | COBIT 도입 이유 |
|-----------|---------------|
| 규제 산업 (금융, 의료, 공공, 통신) | 감독기관 감사 대응 |
| 이사회·경영진의 IT 검토 체계 필요 | EDM 목표로 정기 보고 구조 수립 |
| 전사 리스크 관리(ERM) 연계 | IT 리스크를 전사 리스크에 통합 |
| SOX, ISO 27001, GDPR 복합 규제 | 통합 컨트롤 프레임워크 제공 |

## 3. ISO/IEC 20000 — ITSM을 국제 표준으로 인증받기

### 3.1 ISO/IEC 20000 개요

**ISO/IEC 20000**은 ITSM에 대한 **유일한 국제 표준**입니다. 핵심은 가이드가 아니라 "인증 가능한 표준"이라는 점입니다.

| 항목 | 내용 |
|------|------|
| **정식 명칭** | ISO/IEC 20000 — Information Technology — Service Management |
| **관리 기관** | ISO (국제표준화기구) / IEC (국제전기기술위원회) |
| **최초 출시** | 2005년 (BS 15000을 기반으로) |
| **현재 유효 버전** | **ISO/IEC 20000-1:2018** (Part 1, 요구사항) |
| **인증 단위** | 조직 (제3자 감사) |
| **주요 대상** | 서비스 제공업체, 아웃소싱 업체, 규제 대응 조직 |

### 3.2 ISO/IEC 20000의 파트 구성

ISO/IEC 20000은 여러 파트로 구성됩니다. 인증 대상은 **Part 1**이며, 나머지는 구현·유지·평가를 돕는 부속 문서입니다.

| 파트 | 제목 | 역할 |
|------|------|------|
| **Part 1** | 서비스 관리 시스템 요구사항 | 인증 대상, "~해야 한다(shall)" 의무 사항 |
| **Part 2** | 서비스 관리 시스템 적용 지침 | Part 1 구현 방법 가이드 |
| **Part 3** | 범위 정의와 적용성 가이드 | 인증 범위 결정 지원 |
| **Part 4** | 프로세스 참조 모델 | 프로세스 성숙도 평가 지원 |
| **Part 5** | 구현 계획 가이드 | 도입 로드맵 |
| **Part 6** | 인증 기관 요구사항 | 심사원 자격 |
| **Part 7** | SMS에 대한 ISO 9001 적용 | 품질 경영 시스템 연계 |
| **Part 8** | 서비스 관리 원칙 | 개념·원칙 |
| **Part 9** | 클라우드 서비스 지침 | 클라우드 환경 적용 |
| **Part 10** | 개념과 용어 | 용어집 |
| **Part 11** | ITIL 관계 가이드 | ITIL과의 연계 설명 |

### 3.3 SMS (Service Management System) — 표준의 심장

ISO/IEC 20000의 핵심 개념은 **SMS(Service Management System)**입니다. ISO 9001의 QMS(품질경영시스템), ISO 27001의 ISMS(정보보안경영시스템)와 같은 "경영 시스템" 개념의 ITSM 버전입니다.

SMS는 **PDCA(Plan-Do-Check-Act) 사이클**로 운영됩니다.

| 단계 | 활동 | 주요 산출물 |
|------|------|-----------|
| **Plan** | 서비스 범위·정책·목표 수립, 리스크 평가 | 서비스 관리 계획, 정책 문서 |
| **Do** | 서비스 설계·전환·전달·해결·관계·개선 프로세스 실행 | 프로세스 실행 기록, 성과 지표 |
| **Check** | 내부 감사, 경영 검토, 성과 측정 | 감사 보고서, 경영 검토 의사록 |
| **Act** | 부적합 사항 개선, 시정·예방 조치 | CAPA(시정·예방 조치) 기록 |

### 3.4 ISO/IEC 20000-1:2018의 주요 요구사항 영역

| 영역 | 요구사항 |
|------|--------|
| **조직 컨텍스트** | 조직·이해관계자 이해, SMS 범위 결정 |
| **리더십** | 경영진 의지, 정책 수립, 역할·책임 |
| **계획 수립** | 리스크·기회 대응, 서비스 관리 목표 |
| **지원** | 자원, 역량, 인식, 의사소통, 문서화 |
| **운영** | 서비스 포트폴리오, 관계·합의, 공급자 관리, 서비스 설계·전환, 해결·이행, 서비스 보증 |
| **성과 평가** | 모니터링·측정, 내부 감사, 경영 검토 |
| **개선** | 부적합 사항, 지속적 개선 |

### 3.5 인증 프로세스

| 단계 | 활동 | 결과 |
|------|------|------|
| 1. 준비 | SMS 구축, 문서화, 내부 교육 | 실행 가능한 SMS |
| 2. 내부 감사 | 자체 감사로 부적합 사항 식별 | 내부 감사 보고서 |
| 3. 경영 검토 | 경영진이 SMS 성과 검토 | 경영 검토 의사록 |
| 4. 인증 기관 선정 | BSI, DNV, TUV, LRQA 등 | 인증 계약 |
| 5. 1단계 심사 | 문서·준비 상태 점검 | 1단계 심사 보고서 |
| 6. 2단계 심사 | 실제 운영 상태 심사 | 2단계 심사 보고서 |
| 7. 인증 발급 | 중대 부적합 사항이 없으면 발급 | ISO/IEC 20000-1 인증서 |
| 8. 사후 심사 | 매년 사후 심사, 3년마다 갱신 | 인증 유지 |

### 3.6 ISO/IEC 20000이 필요한 조직

| 조직 특성 | 인증 필요 이유 |
|-----------|-------------|
| **서비스 제공업체(MSP)** | 고객에게 ITSM 역량을 객관적으로 증명 |
| **아웃소싱 수주 기업** | RFP·제안서에 인증이 가점 또는 필수 |
| **공공·방산·금융 벤더** | 입찰 자격 요건으로 인증 요구 |
| **M&A·투자 실사 대응** | 외부 검증이 필요한 경우 |
| **글로벌 고객 대응** | 국제 표준 인증으로 신뢰 확보 |

## 4. 세 프레임워크 종합 비교

| 비교 항목 | ITIL 5 | COBIT 2019 | ISO/IEC 20000-1:2018 |
|----------|--------|------------|----------------------|
| **성격** | 모범 사례 | 거버넌스 프레임워크 | 국제 표준 (인증 가능) |
| **핵심 질문** | "어떻게 운영할 것인가" | "IT가 비즈니스에 가치를 주는가, 리스크는 통제되는가" | "객관적으로 기준을 만족하는가" |
| **주요 대상** | IT 운영팀 | 경영진·이사회·감사인 | 서비스 제공자, 규제 대응 조직 |
| **의무 수준** | 권고(should) | 권고(should) | 의무(shall) |
| **인증 단위** | 개인 | 개인 | **조직** |
| **관리 기관** | PeopleCert | ISACA | ISO/IEC |
| **비용 구조** | 교육·개인 인증 비용 | 교육·개인 인증 비용 | 인증 기관 감사 비용 (연간 유지) |
| **갱신 주기** | 비정기 (2019 → 2026, 7년) | 비정기 (2018 → 2023 업데이트) | 5~7년 (2018 현재) |
| **핵심 활용** | 내부 운영 개선 | 전사 거버넌스·감사 대응 | 외부 증명·계약 요건 |
| **분량** | 수천 페이지 | 수백 페이지 | 수십 페이지 (Part 1 기준) |

## 5. 거버넌스 체계 구축 로드맵

### 5.1 단계별 접근

세 프레임워크를 한 번에 적용하는 것은 거의 항상 실패합니다. 성숙도에 따라 층을 쌓아가는 방식이 정석입니다.

| 단계 | 목표 | 활용 프레임워크 | 핵심 산출물 |
|------|------|---------------|-----------|
| **1단계: 운영 기반** | 실무 프로세스 확립 | ITIL 5 (또는 ITIL 4) | 서비스 데스크, 인시던트·변경 프로세스, SLA, CMDB |
| **2단계: 거버넌스 레이어** | 경영진·리스크·감사 관점 도입 | COBIT 2019 (Design Factors 활용) | IT 리스크 등록부, 성과 지표, 경영진 보고 체계, RACI 매트릭스 |
| **3단계: 외부 인증** | 국제 표준 인증으로 신뢰 확보 | ISO/IEC 20000-1:2018 | SMS 문서, 내부 감사, 인증서 |
| **4단계: 통합 최적화** | 세 프레임워크의 중복 제거·통합 운영 | 세 프레임워크 연계 | 통합 컨트롤 매핑, 자동화된 증빙 수집 |

### 5.2 단계별 세부 활동

#### 1단계 — ITIL 기반 운영 확립

| 활동 | 구체 내용 |
|------|---------|
| 프로세스 정의 | 인시던트·요청·문제·변경 관리 워크플로우 문서화 |
| 도구 도입 | ServiceNow, Jira SM, BMC Helix 등 ITSM 플랫폼 |
| 서비스 카탈로그 | 셀프서비스 가능한 요청 유형 정의 |
| SLA 체결 | 비즈니스 부서와 응답·해결 시간 합의 |
| 교육 | 운영 인력 대상 ITIL 4/5 Foundation 교육 |

#### 2단계 — COBIT 거버넌스 도입

핵심은 **전체 40개 목표를 한 번에 적용하지 않는 것**입니다. Design Factors 분석으로 우선순위 10~15개 목표를 선별합니다.

| 우선순위 | 권장 목표 | 선정 이유 |
|---------|---------|---------|
| 높음 | EDM01~05 (거버넌스 5개) | 이사회 보고 체계 수립 |
| 높음 | APO12 (리스크 관리) | 전사 리스크 연계 |
| 높음 | APO13 (보안 관리) | ISO 27001과 연계 |
| 높음 | DSS02 (서비스 요청·인시던트) | ITIL과 직접 연계 |
| 높음 | DSS04 (연속성 관리) | BCP/DRP 연계 |
| 중간 | MEA01~04 (모니터링 4개) | 성과 측정과 규제 대응 |
| 중간 | BAI06 (변경 관리) | ITIL 변경 관리와 연계 |

산출물 예시:
- IT 리스크 등록부 (Risk Register)
- IT 성과 지표 대시보드 (KPI Dashboard)
- 경영진 정기 보고 템플릿
- 각 핵심 목표에 대한 RACI 매트릭스
- 컴플라이언스 매핑 (ISO 27001, SOX, GDPR 등)

#### 3단계 — ISO/IEC 20000 인증 추진

1~2단계가 제대로 돌아가고 있다면 인증 준비는 "문서화와 증빙 수집" 성격에 가깝습니다.

| 활동 | 구체 내용 |
|------|---------|
| SMS 범위 결정 | 어떤 서비스가 인증 대상인지 정의 |
| Gap 분석 | 현재 운영 vs ISO 20000-1:2018 요구사항 비교 |
| 문서화 | 서비스 관리 계획, 정책, 절차서, 기록 양식 |
| 내부 감사 | 자체 감사로 부적합 사항 식별 및 시정 |
| 경영 검토 | 경영진의 공식 검토 회의 |
| 인증 기관 선정 | BSI, DNV, TUV, LRQA 등 |
| 1·2단계 심사 | 공식 인증 심사 |

### 5.3 자주 빠지는 함정

| 함정 | 증상 | 예방책 |
|------|------|------|
| **도구부터 사기** | ServiceNow를 먼저 도입하고 프로세스는 나중에 정리 | 프로세스 정의 → 도구 선택 → 구현 순서 준수 |
| **문서만 만들기** | 인증 대응용 문서만 만들고 실제 업무와 괴리 | 문서는 실제 운영을 반영해야 함 |
| **전체를 한 번에 적용** | COBIT 40개 목표 전부 적용 시도 | Design Factors로 우선순위 10~15개 선별 |
| **거버넌스 위임** | IT 팀에만 거버넌스를 맡김 | 이사회·경영진의 공식 참여 필수 |
| **인증이 목적이 되는** | 인증서 취득이 목표가 되어 본질 상실 | 인증은 결과이지 목적이 아님 |
| **개선 활동 부재** | 인증 후 매년 같은 수준으로 유지만 | 성숙도 로드맵과 CAPA 활용 |

## 6. 통신사 환경에서의 적용 포인트

통신사는 세 프레임워크의 가치가 특히 크게 체감되는 환경입니다.

| 특성 | 관련 프레임워크 | 구체적 활용 |
|------|---------------|-----------|
| **24/7 운영 + 대규모 사용자** | ITIL 5 | 인시던트·변경 관리 체계화, SLA/XLA 측정 |
| **전기통신사업법·개인정보보호법** | COBIT MEA03 | 규제 준수 모니터링 체계 |
| **금융 규제(금융감독원 가이드라인)** | COBIT EDM03, APO12 | IT 리스크 관리와 내부 통제 |
| **공공·기업 고객 대상 서비스 계약** | ISO/IEC 20000 | 입찰 가점, 계약 요건 충족 |
| **국제 고객·해외 법인** | ISO/IEC 20000 | 국제 표준으로 신뢰 확보 |
| **생성형 AI 도입 (AICC, 챗봇)** | ITIL 5 AI Governance + COBIT | 6C 모델과 EDM02·APO11 연계 |

ITIL 5가 강조한 **임베디드 거버넌스**와 **AI 거버넌스(6C 모델)**는 사실상 COBIT의 거버넌스 관점을 ITSM 프레임워크 내부로 흡수한 것에 가깝습니다. 두 프레임워크가 점점 수렴하고 있다는 의미이며, 통신사처럼 규제·운영·혁신이 동시에 요구되는 환경에서는 이 수렴 방향성이 특히 유리합니다.

## 7. 시리즈 핵심 정리

Part 1부터 6까지의 핵심을 다섯 가지로 압축하면 이렇습니다.

| # | 핵심 메시지 |
|---|-----------|
| 1 | **ITSM·ITOM·ITAM**은 서로 다른 일이지만 연결되어야 의미가 있습니다. |
| 2 | **CMDB**가 연결의 허브이고, **ServiceNow** 같은 통합 플랫폼이 연결을 자동화합니다. |
| 3 | **ITIL 5**는 디지털 제품 관리, AI 거버넌스, 경험 관리로 범위를 넓혔습니다. |
| 4 | **COBIT**은 거버넌스, **ISO 20000**은 국제 표준 인증을 담당하며, ITIL과 층을 이뤄야 합니다. |
| 5 | 거버넌스 체계는 **작게 시작해서 성숙도를 높이는 것**이 정석입니다. 운영 → 거버넌스 → 인증 순서로. |

## 8. 약어 정리

| **ISMS** | Information Security Management System | ISO 27001의 정보보안경영시스템 |

| **BAI** | Build, Acquire, Implement | COBIT 구축 영역 (11개 목표) |
| **DSS** | Deliver, Service, Support | COBIT 전달·서비스·지원 영역 (6개 목표) |
| **MEA** | Monitor, Evaluate, Assess | COBIT 모니터링 영역 (4개 목표) |

| **PDCA** | Plan-Do-Check-Act | 지속적 개선 사이클 |
| **CAPA** | Corrective and Preventive Action | 시정·예방 조치 |
| **ERM** | Enterprise Risk Management | 전사 리스크 관리 |
| **MSP** | Managed Service Provider | 관리형 서비스 제공업체 |
| **CAB** | Change Advisory Board | 변경 자문 위원회 |
| **BCP/DRP** | Business Continuity Plan / Disaster Recovery Plan | 업무 연속성·재해 복구 계획 |
