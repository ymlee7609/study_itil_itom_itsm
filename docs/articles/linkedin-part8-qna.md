## Q-001. ITOM의 RCA와 ITSM의 RCA는 어떻게 다르고 어떻게 연결되는가?

> "Part 3에서 ITOM의 근본 원인 분석을 다루고, Part 2에서 문제 관리의 근본 원인 분석도 다뤘는데 — 이 둘이 같은 건가요, 다른 건가요? 그리고 만약 다르다면 어떻게 연결되나요?"

좋은 질문입니다. 이름이 같아서 헷갈리기 쉬운데, 실제로는 **완전히 다른 층위에서 이루어지는 보완적인 두 활동**입니다.

### 한 문장으로 정리하면

> **ITOM의 RCA는 "무엇이 고장났는지(What)"를 기계가 즉시 찾고, ITSM의 RCA는 "왜 그렇게 고장날 수밖에 없었는지(Why)"를 사람이 사후에 파헤칩니다.**

이 한 줄에 관계의 본질이 거의 다 들어 있습니다. 하나씩 풀어보겠습니다.

### 성격 비교

두 RCA는 수행 주체·시간 단위·대상 깊이가 전부 다릅니다.

**ITOM RCA**
- 수행 주체: **시스템**이 자동으로 (AIOps, Event Correlation Engine)
- 분석 대상: 인프라, 네트워크, 애플리케이션의 **기술적 원인 체인**
- 입력: 메트릭, 로그, 트레이스, 이벤트, CMDB 토폴로지
- 기법: 토폴로지 기반 추론, 시계열 상관 분석, ML 이상 탐지
- 시간: **실시간 ~ 분 단위**
- 산출물: 상관관계 분석, 용의 CI 리스트, 알림 압축, 자동 인시던트 티켓
- 목적: 지금 터진 장애의 **빠른 복구 지원**

**ITSM RCA (문제 관리 내부)**
- 수행 주체: **사람**이 (Problem Manager, SME, 엔지니어)
- 분석 대상: 기술 + **프로세스 + 사람 + 조직·의사결정**
- 입력: 인시던트 이력, ITOM RCA 결과, 변경 기록, 인터뷰
- 기법: 5 Why, Ishikawa(생선뼈), Kepner-Tregoe, FMEA 같은 정성적 기법
- 시간: **시간 ~ 일·주 단위**
- 산출물: 문제 레코드, Known Error(KEDB), RFC(변경 요청), 교훈 보고서
- 목적: 미래의 **재발 방지**

### 실제로 어떻게 이어지는지 — 통신사 시나리오

Part 2에서 다뤘던 "월요일 아침 빌링 시스템 느려짐" 장애가 두 RCA의 연계를 가장 잘 보여줍니다.

**09:12 — ITOM RCA의 시간**

분 단위로 경보가 튀어나옵니다. "billing-app-03 CPU 급증", "billing-db-01 응답 2.8초", "CRM 앱 5xx 에러 폭증". Event Correlation 엔진이 세 경보를 "billing 서비스 토폴로지"로 묶고, **용의 CI는 billing-db-01**이라고 지목합니다. 이걸 근거로 인시던트가 자동 생성되고 P1으로 분류됩니다.

ITOM이 한 일: **"어디가 문제냐"**에 대한 기술적 답을 분 단위로 제공.
ITOM이 못 한 일: **"왜 월요일마다 그러는가"**는 대답 못 함.

**10:30 — ITSM RCA의 시간**

DB 재시작으로 복구는 됐는데, 이게 3주 연속 월요일마다 반복되는 패턴이라는 게 밝혀집니다. 여기서 문제 관리가 발동하고, Problem Manager가 **5 Why**를 돌립니다.

- 왜 DB가 느려졌나? → 메모리와 커넥션 풀이 꽉 참
- 왜 꽉 찼나? → 주말 배치가 남긴 세션이 정리 안 됨
- 왜 정리가 안 됐나? → 배치 실패 시 finally에서 close() 누락
- 왜 코드에 누락이 있었나? → 리뷰에서 예외 경로 검토 빠짐
- 왜 리뷰 체크리스트에 예외 경로가 없었나? → **리뷰 가이드가 최신 아키텍처를 반영 못 함**

도달 지점은 단순한 코드 수정이 아니라 **"리뷰 가이드 개정"**이라는 조직·프로세스 개선입니다. ITOM이 던진 "billing-db-01" 단서가 ITSM RCA의 출발점이 된 거죠.

### 관계의 핵심 네 가지

1. **시간축이 다르다** — ITOM은 "지금", ITSM은 "이후". 같이 일어나지 않습니다.
2. **입력-출력 관계다** — ITOM이 남긴 상관관계·토폴로지·타임라인이 ITSM RCA의 가장 신뢰도 높은 입력이 됩니다. ITOM 없이 ITSM만 하면 "감으로 하는 추정"이 됩니다.
3. **깊이의 방향이 다르다** — ITOM은 **수직적으로** 깊이 탐색(서버 → DB → 네트워크), ITSM은 **수평적으로** 원인을 확장(코드 → 리뷰 → 가이드 → 교육 → 조직 문화).
4. **목적 지표가 다르다** — ITOM은 **MTTR 단축**, ITSM은 **재발률 감소**. 둘 다 필요합니다. MTTR만 쫓으면 같은 장애가 반복되고, 재발 방지만 쫓으면 현재 장애를 못 끕니다.

### 통합 플랫폼에서의 자동 연계

Part 5에서 다룬 ServiceNow 같은 단일 플랫폼에서는 이 관계가 자동화됩니다.

1. ITOM Event Management가 감지
2. AIOps가 상관 분석 → 용의 CI 지목
3. ITSM Incident Management에 티켓 자동 생성 (상관 데이터 첨부)
4. 담당자가 런북·KEDB로 복구
5. Problem Management가 반복 패턴 자동 탐지 → 문제 레코드 생성
6. 사람이 5 Why로 근본 원인 분석
7. Change Enablement로 RFC 생성·승인·구현
8. ITOM이 변경 후 메트릭을 다시 측정해서 **진짜 해결됐는지 검증**

특히 **8단계의 피드백 루프**가 중요합니다. ITSM RCA의 결론을 ITOM이 다시 측정해서 "진짜 해결됐는지" 확인해야 RCA 사이클이 완결됩니다. 이걸 안 하는 조직이 많은데, 하지 않으면 "보고서만 만들고 실제 개선은 없었던" 사태가 반복됩니다.

### 의료 비유로 이해하기

이해를 돕기 위한 비유입니다.

- **ITOM RCA** = CT, MRI, 혈액 검사. "어느 장기에 문제가 있나"를 응급실에서 수십 분 안에 찾는 일.
- **ITSM RCA** = 주치의 문진, 생활 습관 조사, 가족력 파악. "왜 이 환자가 이 병에 걸렸나"를 며칠·몇 주에 걸쳐 파악하는 일.

응급 환자가 오면 CT부터 찍지 문진부터 하지 않습니다. 반대로 퇴원만 시키고 생활 습관 개선을 안 하면 한 달 뒤에 또 실려 옵니다. 두 RCA가 공존해야 하는 이유입니다.

### 자주 보이는 안티패턴

현장에서 이 관계가 깨져 있는 조직들이 보이는 증상입니다.

- **ITOM RCA만 있는 조직**: 매번 빨리 복구는 되지만 같은 장애가 반복. 문제 관리 프로세스가 없거나 형식적임.
- **ITSM RCA만 있는 조직**: 사후 보고서는 많은데 장애 대응이 느림. 상관 분석·AIOps 부재.
- **둘이 단절된 조직**: ITOM의 상관 분석 결과가 Problem Record에 연결 안 됨. 같은 CMDB를 쓰지 않거나, 도구가 쪼개져 있거나(Part 5의 "도구 파편화").
- **RCA가 "누구 탓"으로 끝나는 조직**: 5 Why를 3번째에서 멈춤. "김대리가 실수했다"로 끝남. 프로세스·시스템 원인까지 도달해야 진짜 RCA입니다.

### 한 문장으로 마무리

두 RCA는 **하나의 장애 대응 생애주기에서 시간을 두고 이어지는 두 단계**입니다. 따로 존재하는 두 기능이 아니라, 이어져야 의미가 있는 쌍입니다. 통신사처럼 장애 빈도와 영향 범위가 큰 환경에서는 이 연계의 자동화 수준이 곧 운영 성숙도를 결정합니다.
