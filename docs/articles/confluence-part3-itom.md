## 이 글의 위치

Part 1에서 ITSM, ITOM, ITAM의 큰 그림을 레스토랑에 비유해서 정리했습니다. ITOM은 "주방 운영" — 손님 눈에는 안 보이지만, 주방이 멈추면 레스토랑 전체가 마비되는 역할이었죠.

이번 글에서는 그 주방 안을 좀 더 자세히 들여다봅니다. ITOM이 구체적으로 무엇을 하는지, AIOps는 왜 주목받는지, 그리고 ITSM과 어떻게 연결되는지를 다룹니다.

## ITOM의 세 가지 핵심 기능

ITOM의 핵심 기능을 세 가지로 나눠서 보겠습니다.

| 기능 | 레스토랑 비유 | IT에서는 | 핵심 질문 |
|------|-------------|----------|-----------|
| **모니터링 (Monitoring)** | 주방 온도계, 가스 압력 게이지 | 서버/네트워크/애플리케이션 상태를 24시간 실시간 측정 | "지금 정상적으로 돌아가고 있나?" |
| **이벤트 관리 (Event Management)** | 경험 많은 주방장의 판단력 | 수만 건의 알림에서 진짜 중요한 것만 걸러내기 | "이 알림이 진짜 문제인가, 노이즈인가?" |
| **디스커버리 (Discovery)** | 주방 장비 전수 조사 | 네트워크를 자동 스캔해서 장비/소프트웨어/연결 관계 파악 | "우리 환경에 뭐가 있고, 어떻게 연결되어 있나?" |

### 모니터링 상세

모니터링은 IT 인프라의 실시간 건강 상태를 측정하는 활동입니다. 측정 대상은 한 가지가 아닙니다.

| 모니터링 유형 | 대상 | 주요 측정 항목 |
|-------------|------|--------------|
| 인프라 모니터링 | 서버, VM, 컨테이너 | CPU, 메모리, 디스크, 네트워크 사용률 |
| 네트워크 모니터링 | 스위치, 라우터, 방화벽 | 대역폭, 지연시간, 패킷 손실률 |
| 애플리케이션 모니터링 (APM) | 비즈니스 애플리케이션 | 응답 시간, 처리량, 에러율 |
| 로그 모니터링 | 시스템/앱 로그 | 에러 패턴, 이상 징후 |

이 모니터링 데이터를 의미 있게 분석하려면 세 가지 요소가 필요합니다. 업계에서는 이를 **관측성(Observability)의 3대 요소**라고 부릅니다.

| 요소 | 설명 | 쉽게 말하면 |
|------|------|-----------|
| 메트릭 (Metrics) | 수치 데이터 (시계열) | "CPU가 몇 %인지" — 숫자 |
| 로그 (Logs) | 이벤트 기록 (텍스트) | "언제 무슨 일이 있었는지" — 일기장 |
| 트레이스 (Traces) | 요청 흐름 추적 | "사용자 요청이 어떤 경로로 처리됐는지" — 동선 추적 |

### 이벤트 관리 상세

하루에 수천, 수만 건의 이벤트가 발생합니다. 이걸 다 사람이 보면 정작 중요한 신호를 놓치게 됩니다. 이벤트 관리는 이 문제를 체계적으로 해결합니다.

| 처리 단계 | 하는 일 | 예시 |
|-----------|---------|------|
| 수집 | 다양한 소스에서 이벤트 데이터 수집 | 서버, 네트워크, 애플리케이션 알림 취합 |
| 필터링 | 중복, 불필요한 이벤트 제거 | 같은 서버에서 1분 간격으로 온 동일 알림 → 1건으로 통합 |
| 상관분석 | 관련 이벤트끼리 묶기 | 스위치 장애 + 연결된 서버 10대 알림 → 하나의 원인으로 연결 |
| 분류 | 심각도 판단 | 정보/경고/예외/위기로 분류 |
| 조치 | 대응 실행 | 자동 복구, 인시던트 생성, 담당자 알림 |

이벤트는 심각도에 따라 다르게 처리됩니다.

| 심각도 | 대응 | 예시 |
|--------|------|------|
| 정보 (Informational) | 로깅만 수행 | 백업 완료, 서비스 정상 시작 |
| 경고 (Warning) | 모니터링 강화 | 디스크 사용률 80% 도달 |
| 예외 (Exception) | 즉시 대응 필요 | 서비스 장애, 보안 이벤트 |
| 위기 (Critical) | 에스컬레이션 | 핵심 서비스 완전 중단 |

### 디스커버리 상세

디스커버리는 네트워크를 자동으로 스캔해서 "우리 환경에 뭐가 있는지"를 파악합니다. 수동으로 엑셀에 정리하던 방식과는 정확도와 속도 면에서 완전히 다릅니다.

| 수집 대상 | 수집하는 정보 | 활용 |
|-----------|-------------|------|
| 하드웨어 | 모델, 시리얼, CPU, 메모리 | 자산 관리, 용량 계획 |
| OS/플랫폼 | OS 버전, 패치 수준 | 패치 관리, 컴플라이언스 |
| 소프트웨어 | 설치된 SW, 버전 | 라이선스 관리, 취약점 스캔 |
| 네트워크 | IP, MAC 주소, 연결 관계 | 네트워크 토폴로지, 장애 분석 |
| 클라우드 | 인스턴스, 서비스, 태그 | 비용 관리, 거버넌스 |

수집된 정보는 CMDB(Configuration Management Database)에 자동으로 등록됩니다. Part 1에서 설명했던 "IT 구성요소와 관계를 관리하는 통합 DB"가 바로 이 데이터로 채워지는 겁니다.

## 서비스 매핑 — "이 장비가 죽으면 뭐가 영향받지?"

디스커버리가 "뭐가 있는지"를 알려준다면, **서비스 매핑(Service Mapping)**은 "그것들이 어떻게 연결되어 있는지"를 보여줍니다.

| 서비스 맵 구성 요소 | 설명 | 예시 |
|-------------------|------|------|
| 비즈니스 서비스 | 비즈니스 기능 단위 | 온라인 결제, 고객센터 상담 |
| 애플리케이션 서비스 | 앱 컴포넌트 | 웹서버, API 서버, 데이터베이스 |
| 인프라 구성요소 | 물리/가상 자원 | 서버, 네트워크 장비, 스토리지 |
| 의존관계 | 구성요소 간 연결 | 웹서버 -> API -> DB -> 스토리지 |

### 통신사에서 서비스 매핑이 중요한 이유

기지국 장비 하나에 문제가 생겼을 때, "이 장비 때문에 어느 지역, 몇 가구의 서비스가 영향받는지"를 즉시 파악해야 합니다.

| 상황 | 서비스 맵 없을 때 | 서비스 맵 있을 때 |
|------|----------------|-----------------|
| 장애 발생 | 담당자들이 각자 아는 범위에서 영향 추정 | 영향받는 서비스, 지역, 고객 수가 즉시 표시 |
| 변경 작업 | "이거 바꿔도 괜찮겠지?" 감으로 판단 | "이 변경이 어디까지 영향을 미치는지" 데이터로 확인 |
| 용량 계획 | 서비스별 리소스 현황 파악 어려움 | 서비스 단위로 인프라 사용 현황 조회 가능 |

## AIOps — 사후 대응에서 사전 예방으로

여기까지가 전통적인 ITOM의 영역이라면, 최근 가장 주목받는 흐름은 **AIOps(AI for IT Operations)**입니다.

### AIOps가 필요한 이유

IT 환경이 점점 복잡해지면서, 사람이 모든 이벤트를 직접 보고 판단하는 것이 물리적으로 불가능해졌습니다. AIOps는 AI와 머신러닝을 IT 운영에 적용해서 이 문제를 해결합니다.

| AIOps 핵심 기능 | 하는 일 | 효과 |
|---------------|---------|------|
| 노이즈 감소 | 중복 알림 묶기, 유지보수 중 알림 억제, 동적 임계값 적용 | 알림의 90% 이상 감소 사례 |
| 근본 원인 자동 분석 | 동시다발 이벤트에서 원인과 결과를 AI가 추론 | 장애 진단 시간 대폭 단축 |
| 예측 분석 | 과거 패턴 학습 기반 장애/용량 문제 사전 예측 | 장애 발생 전 사전 조치 가능 |
| 자동 인시던트 생성 | 분석 결과를 기반으로 ITSM 인시던트 자동 생성 | 대응 시간 단축 |
| 지능형 라우팅 | 인시던트를 최적의 담당자/팀에 자동 배정 | 해결 시간 단축 |

### 기존 방식 vs AIOps 방식

| 구분 | 기존 방식 (Reactive) | AIOps 방식 (Proactive) |
|------|-------------------|----------------------|
| 장애 감지 | 고정 임계값 초과 시 알림 | ML 기반 동적 임계값, 이상 패턴 자동 감지 |
| 원인 분석 | 엔지니어가 로그를 하나씩 확인 | AI가 관련 이벤트를 묶고 원인 자동 추론 |
| 알림 처리 | 모든 알림을 사람이 판단 | 노이즈를 자동 필터링, 핵심만 전달 |
| 용량 계획 | 경험 기반 추정 | 과거 패턴 학습 기반 예측 분석 |
| 장애 대응 | 장애 후 대응 (사후) | 장애 징후 감지 시 사전 조치 |

### 생성형 AI까지 결합되고 있습니다

2017년에 Gartner가 AIOps 개념을 처음 정의한 이후, 최근에는 생성형 AI까지 결합되면서 활용 범위가 넓어지고 있습니다.

| 활용 | 예시 |
|------|------|
| 자연어 쿼리 | "지난 1시간 동안 지연이 높았던 서비스는?" |
| 장애 요약 | 인시던트 컨텍스트를 자동으로 요약해서 전달 |
| 런북 생성 | 장애 대응 절차를 자동으로 작성 |
| 운영 보고서 | 일간/주간 운영 보고서 자동 생성 |

## 통신사 시나리오: 새벽 기지국 장애

구체적인 상황으로 보겠습니다. 새벽 3시, A 지역 기지국 라우터에서 패킷 로스가 급증합니다.

| 순서 | 누가 | 뭘 하나요 |
|------|------|-----------|
| 1 | **ITOM 모니터링** | 이상 징후를 먼저 감지합니다. 고객 불만이 접수되기 전에 경보가 울립니다. |
| 2 | **이벤트 관리** | 관련 알림들을 묶습니다. 라우터 에러율 초과 + 기지국 3곳 트래픽 저하 + 고객 단말 접속 실패 — 하나의 원인에서 비롯된 것이라고 상관분석합니다. |
| 3 | **서비스 매핑** | 영향 범위를 보여줍니다. "해당 라우터에 연결된 기지국 3곳, 영향 가구 약 5,000가구." |
| 4 | **ITSM 자동 연동** | 인시던트 티켓이 자동 생성되고, 네트워크 운영팀에 배정됩니다. "5,000가구 영향, 우선순위 높음" 정보가 티켓에 붙습니다. |
| 5 | **고객 대응** | VOC가 들어오면 상담사가 "인지된 장애이며 복구 중입니다"라고 바로 안내할 수 있습니다. |

이게 바로 **ITOM과 ITSM이 연결되는 지점**입니다. ITOM이 감지하고 분석한 결과가 ITSM의 인시던트로 자동 전환됩니다.

## ITOM과 ITSM의 연결

두 영역이 주고받는 데이터를 정리하면 이렇습니다.

| ITOM에서 ITSM으로 | 설명 |
|-------------------|------|
| 이벤트 -> 인시던트 | 모니터링 경보가 인시던트 티켓을 자동 생성 |
| 디스커버리 -> CMDB | 자동 스캔 결과가 CMDB를 실시간 갱신 |
| 서비스 매핑 -> 변경 관리 | 변경 요청 시 영향받는 서비스 범위를 자동 표시 |
| 성능 데이터 -> SLA 관리 | 모니터링 수치가 SLA 달성 여부 판단 근거 |

반대 방향도 있습니다.

| ITSM에서 ITOM으로 | 설명 |
|-------------------|------|
| 변경 요청 -> 인프라 변경 | 승인된 변경 사항을 ITOM 자동화 엔진이 실행 |
| 서비스 요청 -> 자동 프로비저닝 | 서비스 카탈로그에서 VM 요청 -> 클라우드에 자동 생성 |

Part 1에서 "ITOM이 감지하고, ITSM이 처리하고, ITAM이 자산 판단을 돕는다"고 했던 패턴이, 실제로는 이런 데이터 흐름으로 작동하는 겁니다.

## 정리

| 개념 | 한 마디로 | 레스토랑 비유 |
|------|-----------|-------------|
| 모니터링 | 24시간 실시간 상태 측정 | 온도계, 가스 게이지 |
| 이벤트 관리 | 노이즈 걸러내고 핵심만 전달 | 경험 많은 주방장의 판단 |
| 디스커버리 | 환경 자동 파악 및 CMDB 갱신 | 주방 장비 전수 조사 |
| 서비스 매핑 | 장비와 서비스의 연결 관계 시각화 | "이 화구 고장 나면 어떤 요리에 영향?" |
| AIOps | AI/ML 기반 예측, 자동 분석, 노이즈 감소 | AI가 주방 전체를 감독하고 문제를 예측하는 것 |

ITOM은 단순히 "서버 상태를 보는 것"이 아닙니다. 모니터링으로 상태를 파악하고, 이벤트 관리로 노이즈를 걸러내고, 서비스 매핑으로 영향 관계를 파악하고, AIOps로 예측과 자동화까지 하는 — IT 인프라 운영의 전체 흐름입니다.

## 약어 정리

글에서 나온 약어를 한 번에 정리했습니다.

| 약어 | 풀네임 | 한 마디로 |
|------|--------|-----------|
| **ITOM** | IT Operations Management | IT 인프라 운영 관리 |
| **ITSM** | IT Service Management | IT 서비스 관리 (사용자 대면) |
| **AIOps** | AI for IT Operations | AI/ML 기반 IT 운영 자동화 |
| **APM** | Application Performance Monitoring | 애플리케이션 성능 모니터링 |
| **CMDB** | Configuration Management Database | IT 구성요소와 관계를 관리하는 통합 DB |
| **SLA** | Service Level Agreement | 서비스 수준 합의 |
| **ML** | Machine Learning | 머신러닝 (기계학습) |
| **VOC** | Voice of Customer | 고객의 소리 |
| **NMS** | Network Management System | 네트워크 관리 시스템 |
