## 도구가 쪼개져 있으면 벌어지는 일

ITSM은 A 시스템, 모니터링은 B 시스템, 자산 관리는 엑셀 — 이런 조직이 상당히 많습니다.

| 문제 | 구체적 상황 |
|------|------------|
| **수동 연결** | 모니터링 알림을 보고 별도 시스템에 티켓을 직접 만들어야 합니다. 장애 하나에 화면 세 개를 왔다 갔다 합니다. |
| **데이터 단절** | "이 서버가 죽으면 어떤 서비스가 영향받는지" 한눈에 파악이 안 됩니다. 관계 정보가 시스템마다 다르게 관리되거든요. |
| **사후 분석 어려움** | 로그는 여기, 티켓은 저기, 자산 이력은 또 저기 — 데이터를 끌어모으는 것 자체가 일이 됩니다. |
| **사람이 접착제** | 자동화할 수 있는 연결을 사람이 수동으로 하고 있습니다. 야간이나 주말에 특히 문제가 됩니다. |

이 문제를 해결하는 접근이 **단일 플랫폼 전략**입니다. ITSM, ITOM, ITAM을 같은 데이터 모델 위에서 돌리는 겁니다.

## ServiceNow 전에 — ITIL 프레임워크와 ITIL 5 변화

ServiceNow의 구체적 구조를 보기 전에 반드시 짚고 가야 할 것이 **ITIL(IT Infrastructure Library)** 프레임워크입니다. ServiceNow는 독자적인 발명품이 아니라 ITIL이 정의한 프로세스를 플랫폼으로 구현한 것이기 때문입니다.

### ITIL과 ServiceNow의 관계

ITIL은 "IT를 어떻게 서비스처럼 운영할 것인가"에 대한 사실상의 국제 표준입니다. 영국 정부가 1980년대부터 축적해 온 IT 운영 모범 사례 모음이며, 현재는 PeopleCert가 관리합니다. Part 2~4에서 다룬 인시던트 관리, 문제 관리, 변경 관리, SLA, CMDB 같은 개념들은 모두 ITIL에서 체계화된 것입니다.

| 구분 | ITIL | ServiceNow |
|------|------|-----------|
| **성격** | 프레임워크 (무엇을 할 것인가) | 플랫폼 (어떻게 구현할 것인가) |
| **관리 주체** | PeopleCert | ServiceNow Inc. |
| **결과물** | 지침서, 인증, 모범 사례 | 소프트웨어, 모듈, API |
| **비유** | 건축 설계 원칙 | 실제 건축 자재 |

즉, **ITIL이 틀이고 ServiceNow는 그 틀을 구현한 도구**입니다. 플랫폼을 제대로 이해하려면 그 위에 올라탈 프레임워크가 어디로 가고 있는지부터 파악해야 합니다.

### ITIL 5 (2026년 2월 출시) — 7년 만의 대변화

ITIL은 2019년 **ITIL 4** 이후 약 7년 만에 **ITIL 5**가 **2026년 2월 12일** 정식 출시됐습니다. 단순한 용어 업데이트가 아니라 IT 운영의 관점 자체를 바꾸는 변화들이 포함되어 있어, ServiceNow를 포함한 모든 ITSM 플랫폼 전략과 직결됩니다.

#### 한눈에 보는 ITIL 4 vs ITIL 5

| 비교 항목 | ITIL 4 (2019) | ITIL 5 (2026) |
|----------|---------------|---------------|
| **핵심 초점** | IT 서비스 관리 (ITSM) | 디지털 제품 및 서비스 관리 (DPSM) |
| **라이프사이클** | 서비스 가치 사슬(SVC) 6개 활동 | 제품 및 서비스 라이프사이클(PSLM) 8개 활동 |
| **가치 시스템** | 서비스 가치 시스템(SVS) | ITIL 가치 시스템 |
| **AI 취급** | 자동화의 일부로 간접 언급 | AI 거버넌스 공식 프레임워크 + 6C 모델 |
| **거버넌스** | 작업 위에서 통제·승인 | 작업 곁에서 가이드 (임베디드 거버넌스) |
| **경험 관리** | 가치 공동 창출 개념 | XLA, 경험 중심 성과 평가 |
| **서비스 관계** | 기본 개념 | Basic / Cooperative / Collaborative 모델로 체계화 |
| **콘텐츠 구성** | - | 36% 신규 + 64% ITIL 4에서 발전 |
| **공식 도서** | 5권 | 6권 (AI Governance 도서 추가) |
| **인증 모듈** | 5개 핵심 | 9개 핵심 + 확장 모듈 |

#### ITIL 5 핵심 변화 6가지

##### 1. IT 서비스 관리 → 디지털 제품 및 서비스 관리 (DPSM)

ITIL 5는 범위를 "IT 서비스 관리"에서 **디지털 제품 및 서비스 관리(Digital Product & Service Management)**로 확장했습니다. IT 서비스를 "유지보수 대상"이 아니라 **"계속 진화하는 디지털 제품"**으로 다루자는 사고방식입니다. 사내 포털조차 앱 스토어의 앱처럼 버전, 로드맵, 사용자 피드백 루프를 가져야 한다는 관점입니다.

##### 2. 서비스 가치 사슬(SVC) → 제품 및 서비스 라이프사이클(PSLM)

ITIL 4의 6개 활동이 ITIL 5에서 8개 활동으로 확장됐습니다.

| ITIL 4 SVC (6단계) | ITIL 5 PSLM (8단계) | 변화의 의미 |
|--------------------|---------------------|-------------|
| Plan | **Discover** | 계획 이전에 니즈·기회 '발견'부터 시작 |
| Engage | *(서비스 관계 모델로 분리)* | 관계는 별도 모델로 체계화 |
| Design & Transition | **Design** + **Transition** | 설계와 전환을 분리해 각각 강화 |
| Obtain/Build | **Acquire** + **Build** | 획득과 구축을 독립 활동으로 |
| Deliver & Support | **Operate** + **Deliver** + **Support** | 운영·전달·지원을 세분화 |
| Improve | *(전체 활동에 내재화)* | 지속적 개선이 모든 단계의 일부 |

##### 3. AI 거버넌스 공식 도입과 6C 모델

ITIL 4에서 AI는 "자동화의 한 요소"로 간접 언급되는 수준이었습니다. ITIL 5는 **AI Governance를 독립된 공식 프레임워크로 편입**했으며, 별도의 공식 도서(ITIL AI Governance)까지 출간됐습니다. 핵심은 **6C 모델**로, AI 역량을 여섯 가지 축으로 분류하고 도입 시 리스크 평가·기회 식별·윤리 검토를 구조화된 방식으로 수행하도록 합니다.

| 6C 요소 | 초점 |
|---------|------|
| **Context** | AI가 적용되는 비즈니스·규제 맥락 |
| **Capability** | 조직의 AI 기술·데이터 역량 |
| **Consent** | 데이터 사용과 AI 결정에 대한 동의·투명성 |
| **Control** | AI 출력물에 대한 검증과 통제 |
| **Change** | AI 도입이 일으키는 조직적 변화 관리 |
| **Continuity** | AI 장애 시 서비스 연속성 보장 |

##### 4. 임베디드 거버넌스 (Embedded Governance)

ITIL 4의 거버넌스는 작업 위에서 검토·승인하는 구조였습니다. 실무에서는 CAB 회의 대기, 승인 지연 등 속도 저하의 원인이 되기도 했습니다. ITIL 5는 이것을 **임베디드 거버넌스**로 전환했습니다.

| 구분 | ITIL 4 | ITIL 5 |
|------|--------|--------|
| **거버넌스 위치** | 작업 위에서 검토·승인 | 작업 곁에서 가이드 |
| **속도** | 승인 대기로 지연 | 경계 안에서 자율 운영 |
| **도구 요구사항** | 별도 승인 워크플로우 | 워크플로우 자체에 승인·감사·정책 내재 |

##### 5. 경험 관리(Experience Management)의 격상과 XLA

Part 2에서 다룬 SLA가 기술적 지표 약속이었다면, ITIL 5는 여기서 한 단계 나아가 **XLA(Experience Level Agreement)**를 공식화했습니다. 가동률, MTTR 같은 기술 지표만으로는 "티켓은 4시간 안에 처리됐지만 사용자는 불만족" 같은 상황을 잡아낼 수 없기 때문입니다.

| 지표 유형 | 초점 | 예시 |
|-----------|------|------|
| **SLA** (Service Level Agreement) | 기술적 수준 | P1 인시던트 4시간 이내 해결 |
| **XLA** (Experience Level Agreement) | 사용자 경험 수준 | 사용자 만족도 85% 이상, 첫 접점 해결률 80% 이상 |
| **OLA** (Operational Level Agreement) | 내부 부서 간 약속 | 네트워크팀 2시간 이내 진단 |

##### 6. 서비스 관계 모델 (Service Relationship Model)

서비스 제공자와 소비자의 관계를 세 유형으로 분류해 관계 유형에 따라 SLA·거버넌스·도구 구조가 달라져야 함을 명시했습니다.

| 관계 유형 | 설명 | 예시 |
|-----------|------|------|
| **Basic** | 표준화된 요청-제공 관계 | 비밀번호 초기화, 장비 요청 |
| **Cooperative** | 양측이 성과를 공유하는 협력 관계 | 핵심 업무 시스템 운영 계약 |
| **Collaborative** | 가치 공동 창출 파트너 관계 | 디지털 전환 프로젝트 공동 추진 |

### ITIL 5가 ServiceNow 선택에 주는 시사점

ITIL 5의 방향성은 통합 플랫폼의 존재 이유와 정확히 맞물립니다. 쪼개진 도구로는 ITIL 5가 요구하는 구조를 구현하기가 사실상 불가능하기 때문입니다.

| ITIL 5 변화 | 플랫폼에 요구되는 기능 | ServiceNow의 대응 방향 |
|-------------|----------------------|----------------------|
| 디지털 제품 관점 | 제품 워크스페이스, 로드맵·피드백 루프 | Product Workspace, Product Success 모듈 |
| AI 거버넌스 (6C 모델) | AI 도입 리스크·승인·감사 워크플로우 | Now Assist + AI Governance 기능 |
| 임베디드 거버넌스 | 워크플로우 내부에 승인·감사·정책 내재 | Flow Designer, Policy Engine |
| 경험 관리 (XLA) | 사용자 경험 데이터 수집·분석 | Workplace Experience, Survey & Feedback |
| 서비스 관계 모델 | 관계 유형별 차별화된 SLA·계약 관리 | Service Catalog, Account Management |
| PSLM 8단계 라이프사이클 | 제품 기획~지원까지 단일 플랫폼 커버 | ITSM + SPM + APM 통합 |

정리하면, **ITIL 5를 향하는 조직일수록 통합 플랫폼의 가치가 커지고, 쪼개진 도구 전략은 ITIL 5를 따라가기가 점점 어려워진다**는 것이 이번 업데이트의 실무적 메시지입니다. 이 관점을 가지고 ServiceNow가 실제로 어떤 구조와 기능을 제공하는지 살펴보겠습니다.

│  │    ITSM      │  │    ITOM       │  │      ITAM         │    │
│  │              │  │               │  │                   │    │
│  │ 인시던트     │  │ Discovery     │  │ 하드웨어 자산     │    │

### ITOM 모듈 상세

ServiceNow ITOM은 세 가지 핵심 영역으로 구성됩니다.

| 모듈 | 하는 일 | 왜 필요한가 |
|------|---------|------------|
| **ITOM Visibility** | Discovery로 네트워크를 스캔해서 서버, 스위치, 클라우드 리소스를 자동으로 찾아내고 CMDB에 등록합니다. Service Mapping은 서비스-인프라 관계를 자동으로 그려줍니다. | "우리 환경에 뭐가 있고, 뭐가 뭐에 연결되어 있는지" 파악하는 기반입니다. |
| **ITOM Health** | Event Management가 각종 모니터링 도구의 알림을 수집, 중복 제거, 상관분석합니다. AIOps가 노이즈를 줄이고 근본 원인을 추론합니다. | 알림 폭풍 속에서 진짜 문제를 찾아내고, 자동으로 인시던트를 만들어줍니다. |
| **ITOM Optimization** | 클라우드 비용 관리, 자동 프로비저닝, 거버넌스를 제공합니다. | 멀티클라우드 환경에서 비용과 정책을 통제합니다. |

### 모듈 간 연결 흐름

단일 플랫폼의 핵심 가치는 **모듈 간 자동 연결**입니다.

| 연결 | 자동화되는 것 | 쪼개져 있었다면 |
|------|-------------|----------------|
| **ITOM → ITSM** | 이상 징후 감지 시 인시던트 자동 생성, CMDB 기반 영향도 자동 첨부 | 모니터링 화면 보고 → 수동 티켓 생성 → 영향 범위 수동 파악 |
| **ITSM → ITAM** | 티켓 내에서 해당 자산의 보증, 라이선스, 교체 주기 바로 조회 | 별도 시스템이나 엑셀에서 자산 정보 검색 |
| **ITOM → ITAM** | Discovery가 발견한 자산을 CMDB에 자동 등록, ITAM이 생애주기 추적 | 자산 대장 수동 업데이트 |
| **CMDB (허브)** | 모든 모듈이 같은 CI(구성 항목) 데이터를 참조 | 시스템마다 자산 정보가 다르게 관리됨 |

### Now Assist (생성형 AI)

ServiceNow는 2024년부터 Now Assist라는 생성형 AI 기능을 플랫폼 전반에 통합하고 있습니다.

| 적용 영역 | 하는 일 |
|-----------|---------|
| 인시던트 요약 | 장애 컨텍스트를 자동으로 요약해서 담당자에게 전달 |
| 자연어 검색 | "지난 1시간 동안 지연이 높은 서비스는?" 같은 질문에 답변 |
| 런북 생성 | 장애 대응 절차를 자동으로 작성 |
| 가상 에이전트 | 직원의 IT 요청을 AI가 1차 대응 |

## 주요 플랫폼 비교

ServiceNow만 이런 통합 플랫폼을 제공하는 건 아닙니다. 주요 대안과 비교하면 이렇습니다.

| 비교 항목 | ServiceNow | Jira Service Management | BMC Helix |
|-----------|-----------|------------------------|-----------|
| **대상 규모** | 대기업 (5,000명+) | 중견기업 / DevOps 조직 | 대기업 |
| **핵심 강점** | 단일 플랫폼, ITSM+ITOM+ITAM 통합, 강력한 자동화 | Atlassian 생태계 통합, DevOps 친화적, 합리적 가격 | ITIL 프로세스 깊이, 온프레미스 배포 가능 |
| **ITIL 정렬** | 19개 프랙티스 인증 (업계 최고) | 주요 프랙티스 지원, DevOps/Agile 통합에 강점 | 20년+ ITIL 경험, 깊은 프로세스 지원 |
| **ITOM 통합** | 네이티브 (Discovery, Event Mgmt, AIOps) | Opsgenie 통합 수준 (별도 제품) | BMC Helix Operations Management |
| **ITAM 통합** | 네이티브 (같은 플랫폼) | 기본 자산 관리 (고급 기능 제한) | 별도 모듈 연동 |
| **배포 방식** | SaaS 전용 (클라우드) | 클라우드 + Data Center | SaaS + 온프레미스 + 하이브리드 |
| **AI 기능** | Now Assist (생성형 AI) | 기본 자동화 | Helix AI |
| **구현 기간** | 6~18개월 | 2~8주 | 6~12개월 |
| **비용 수준** | 높음 | 합리적 | 높음 |
| **약점** | 높은 TCO, 긴 구현, 전문 인력 필요 | 대규모 ITIL 프로세스 약함, ITOM 제한적 | 레거시 UI, 학습 곡선 |

### 어떤 조직에 어떤 플랫폼이 맞을까

| 조직 특성 | 권장 플랫폼 | 이유 |
|-----------|-----------|------|
| 직원 5,000명 이상, 복잡한 인프라 | ServiceNow | 확장성, ITSM+ITOM+ITAM 네이티브 통합 |
| DevOps 문화, Atlassian 도구 사용 중 | Jira Service Management | 개발-운영 연계, 합리적 비용 |
| 온프레미스 필수 (규제, 보안) | BMC Helix | 하이브리드/온프레미스 배포 지원 |
| 중소기업, 빠른 도입 필요 | Freshservice | 2~4주 도입, 직관적 UX, AI 기능 |

## 도입할 때 알아야 할 것들

ServiceNow의 장점은 분명하지만, 솔직하게 짚어야 할 부분도 있습니다.

### 현실적 고려사항

| 항목 | 내용 |
|------|------|
| **비용** | 라이선스 + 구현 컨설팅 + 커스터마이징 + 교육까지 포함하면 TCO가 상당합니다. 3년 기준으로 상당한 규모의 투자가 필요합니다. |
| **구현 기간** | 단순 도입은 몇 달이지만, 프로세스 정리 + 데이터 마이그레이션 + 조직 적응까지 포함하면 6개월~1년 이상 걸리는 경우가 흔합니다. |
| **전문 인력** | ServiceNow 관리자/개발자 확보가 도입 이후에도 지속적 과제입니다. 시장에서 인력 수요가 높아 확보 경쟁이 치열합니다. |
| **프로세스 먼저** | 프로세스가 정리되지 않은 상태에서 도구만 바꾸면, 비싼 도구 위에서 비효율적인 프로세스를 돌리는 꼴입니다. ITIL 같은 프레임워크로 체계를 먼저 잡는 게 순서입니다. |
| **변화 관리** | 새 시스템 도입은 기술 프로젝트가 아니라 조직 변화 프로젝트입니다. 현업 참여와 교육이 성공의 핵심 요소입니다. |

### 도입 평가 시 체크리스트

| 평가 항목 | 확인 사항 |
|-----------|-----------|
| 기능 적합성 | 현재와 향후 ITSM/ITOM/ITAM 프로세스를 지원하는가 |
| 통합 역량 | 기존 모니터링, CMDB, AD 등과 연동 가능한가 |
| 확장성 | 조직 성장에 따라 확장할 수 있는가 |
| TCO | 라이선스 + 구현 + 운영 + 교육 비용의 총합은 얼마인가 |
| 벤더 지원 | 한국어 지원과 현지 파트너가 있는가 |
| 보안/컴플라이언스 | SOC 2, ISO 27001, 데이터 주권 요구를 충족하는가 |

## 시리즈 핵심 정리

Part 1부터 5까지의 내용을 네 가지로 압축하면 이렇습니다.

| 핵심 | 설명 |
|------|------|
| **연결이 핵심이다** | ITSM, ITOM, ITAM은 따로 돌면 의미가 반감됩니다. 감지 → 처리 → 자산 판단의 흐름이 끊기지 않아야 합니다. |
| **CMDB가 허브다** | 자산의 기술적 관계와 비즈니스 영향을 파악하는 단일 진실의 원천입니다. CMDB가 부실하면 플랫폼을 도입해도 효과가 반감됩니다. |
| **도구보다 프로세스가 먼저다** | 어떤 플랫폼을 쓰든, "인시던트를 어떻게 처리하고, 변경을 어떻게 관리하고, 자산을 어떻게 추적할 것인지" 체계가 먼저입니다. |
| **완벽한 도구는 없다** | ServiceNow든 Jira SM이든 BMC든, 조직의 규모와 상황에 맞는 선택이 최선입니다. |
