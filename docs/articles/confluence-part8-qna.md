## Q-001. ITOM의 RCA와 ITSM의 RCA는 어떻게 다르고 어떻게 연결되는가?

**관련 Part**: [Part 2 ITSM (문제 관리)](./confluence-part2-itsm.md), [Part 3 ITOM (AIOps)](./confluence-part3-itom.md), [Part 5 ServiceNow](./confluence-part5-servicenow.md)
**추가일**: 2026-04-09

### 한 줄 요약

두 RCA는 **같은 이름, 다른 층위**입니다. 경쟁 관계가 아니라 **시간·깊이·주체가 다른 보완 관계**이며, 통합 플랫폼에서는 자동으로 연결됩니다.

> **ITOM의 RCA는 "무엇이 고장났는지(What)"를 기계가 즉시 찾고, ITSM의 RCA는 "왜 그렇게 고장날 수밖에 없었는지(Why)"를 사람이 사후에 파헤칩니다.**

### 두 RCA의 성격 비교

| 축 | ITOM RCA | ITSM RCA (Problem Management) |
|---|---------|------------------------------|
| **질문** | "지금 어느 컴포넌트가 고장났는가?" | "왜 이 장애가 생겼고, 어떻게 재발을 막을 것인가?" |
| **수행 주체** | 시스템 (AIOps, Event Correlation Engine) | 사람 (Problem Manager, SME, 엔지니어) |
| **분석 대상** | 인프라·네트워크·애플리케이션의 기술적 원인 체인 | 기술 + 프로세스 + 사람 + 조직·의사결정 |
| **입력 데이터** | 메트릭, 로그, 트레이스, 이벤트, CMDB 토폴로지 | 인시던트 이력, ITOM RCA 결과, 변경 기록, 인터뷰 |
| **기법** | 토폴로지 기반 추론, 시계열 상관, ML, 이상 탐지 | 5 Why, Ishikawa(생선뼈), Kepner-Tregoe, FMEA |
| **시간 축** | 실시간 ~ 분 단위 | 시간 ~ 일·주 단위 |
| **산출물** | 상관관계 분석, 용의 CI 리스트, 알림 압축, 자동 티켓 첨부 | 문제 레코드, Known Error(KEDB), RFC(변경 요청), 교훈 보고서 |
| **프로세스 위치** | Event Management / Incident Management 내부 | Problem Management (별도 프로세스) |
| **반복 지향** | 현재 장애 해결 지원 | 미래 재발 방지 |

### 실제 시나리오 — 통신사 월요일 아침 빌링 장애

#### 09:12 — ITOM RCA의 시간

```
09:12:03  [ITOM] CPU 임계치 초과 경보 (billing-app-03)
09:12:05  [ITOM] DB 응답 시간 2.8초 초과 (billing-db-01)
09:12:07  [ITOM] 상담 앱 5xx 에러 급증 (crm-web-02)
09:12:08  [ITOM] Event Correlation 엔진:
          → 세 경보가 "billing 서비스 토폴로지"로 묶임
          → 용의 CI: billing-db-01 (fan-out 최상위)
          → 인시던트 자동 생성: INC-20260406-0142
          → 우선순위 자동 판정: P1 (영향도 200명)
```

ITOM이 한 일은 분 단위로 기술 원인 체인을 자동 추론해 "billing-db-01이 용의자"라고 지목한 것입니다. 한계는 "그래서 왜 월요일마다 그러는지"는 대답하지 못한다는 점입니다.

#### 10:30 — 복구 후 ITSM RCA의 시간

서비스는 DB 재시작으로 복구되었지만 동일 패턴이 3주 연속 발생. ITSM의 문제 관리가 발동합니다.

```
[ITSM Problem Record PRB-20260406-0019]
입력: INC-0142 외 동일 패턴 인시던트 3건
기법: 5 Why

Q1. 왜 DB가 월요일 아침에 느려지나?
A1. 메모리와 커넥션 풀이 꽉 차 있었음

Q2. 왜 메모리가 꽉 찼나?
A2. 주말 배치 작업이 남긴 세션이 정리되지 않았음

Q3. 왜 세션이 정리되지 않았나?
A3. 배치 실패 시 finally 블록에서 close()가 누락돼 있음

Q4. 왜 코드에 누락이 있었나?
A4. 코드 리뷰에서 예외 경로 검토가 빠졌음

Q5. 왜 리뷰 체크리스트에 예외 경로가 없었나?
A5. 팀 리뷰 가이드가 최신 아키텍처를 반영하지 못함

→ 근본 원인: 리뷰 가이드의 노후화 (프로세스 결함)
→ 해결책: 배치 코드 finally 수정(RFC) + 리뷰 체크리스트 개정
→ KEDB 등록 + 변경 관리로 연결
```

ITSM RCA가 한 일은 ITOM이 던진 "billing-db-01"이라는 단서를 받아서 **기술 → 프로세스 → 조직**으로 "왜"를 계속 파고든 것입니다. 도달한 지점은 코드 수정 하나가 아니라 **리뷰 가이드 개정**이라는 프로세스 개선까지입니다.

### 관계의 핵심 네 가지

| # | 관계 | 설명 |
|---|------|------|
| 1 | **시간축** | ITOM RCA는 "지금" 대응(분·시간 단위), ITSM RCA는 "이후" 분석(일·주 단위) |
| 2 | **입력-출력** | ITOM RCA가 남긴 상관관계·토폴로지·타임라인이 ITSM RCA의 가장 신뢰도 높은 입력 |
| 3 | **깊이 방향** | ITOM은 수직적으로 깊이 탐색(서버 → DB → 네트워크), ITSM은 수평적으로 원인 확장(코드 → 리뷰 → 가이드 → 교육) |
| 4 | **목적** | ITOM은 MTTR 단축, ITSM은 재발률 감소. 둘 다 필요 — 한쪽만으로는 운영이 성립 불가 |

### 통합 플랫폼(ServiceNow 기준)에서의 자동 연계

Part 5에서 다룬 단일 플랫폼 전략이 이 관계를 자동화합니다.

| 단계 | 담당 모듈 | 하는 일 |
|------|---------|--------|
| 1. 감지 | ITOM Event Management | 메트릭·로그·이벤트 수집, 이상 감지 |
| 2. 상관 분석 | ITOM Event Management + AIOps | 토폴로지 기반 상관 분석, 용의 CI 지목 |
| 3. 인시던트 생성 | ITSM Incident Management | 상관 결과가 자동 첨부된 티켓 생성 |
| 4. 복구 | ITSM Incident Management | 런북·KEDB·지식베이스 검색, 담당자 배정 |
| 5. 패턴 감지 | ITSM Problem Management | 반복 인시던트 자동 탐지 → 문제 레코드 생성 |
| 6. 근본 원인 분석 | ITSM Problem Management | 5 Why, Ishikawa 등 사람이 수행 |
| 7. 재발 방지 | ITSM Change Enablement | RFC 생성, CAB 승인, 변경 구현 |
| 8. 피드백 검증 | ITOM Event Management | 변경 후 메트릭 개선 여부 측정 |

**8단계의 피드백 루프**가 핵심입니다. ITSM RCA의 결론을 ITOM이 다시 측정해서 "진짜 해결됐는지" 확인해야 RCA 사이클이 완결됩니다.

### 의료 비유로 이해하기

| 의료 | ITOM RCA | ITSM RCA |
|------|---------|---------|
| **장비** | CT, MRI, 혈액 검사 | 문진, 생활 습관 조사, 가족력 |
| **질문** | "어느 장기에 문제가 있는가?" | "왜 이 환자가 이 병에 걸렸는가?" |
| **시간** | 응급실에서 수십 분 | 주치의 진료실에서 며칠·몇 주 |
| **결과** | 병변 위치 특정, 응급 처치 | 생활 습관 개선, 치료 계획 |

응급 환자가 오면 CT부터 찍지 문진부터 하지 않습니다. 반대로 퇴원만 시키고 생활 습관 개선을 안 하면 한 달 뒤에 또 실려 옵니다. 두 RCA가 공존해야 하는 이유입니다.

### 자주 나타나는 안티패턴

| 안티패턴 | 증상 | 해결 |
|---------|------|------|
| **ITOM RCA만 있는 조직** | 매번 빨리 복구는 되지만 같은 장애가 반복 | Problem Management 도입, ITOM 데이터를 입력으로 활용 |
| **ITSM RCA만 있는 조직** | 사후 보고서는 많은데 장애 대응이 느림 | Event Correlation, AIOps 도입 |
| **둘이 단절된 조직** | ITOM의 상관 분석 결과가 Problem Record에 연결 안 됨 | 단일 플랫폼 또는 API 연동, CMDB 공유 |
| **RCA가 사람 탓으로 끝남** | "누가 실수했나"로 귀결 | 5 Why를 끝까지 — 프로세스·시스템 원인까지 도달해야 함 |

## 약어 정리 (누적)

| 약어 | 풀네임 | 설명 |
|------|--------|------|
